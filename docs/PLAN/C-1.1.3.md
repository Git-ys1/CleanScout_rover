# C-1.1.3 计划：蓝牙协议重映射 + 风机控制权扩展 + IDE 编译兼容

> 本计划已被 `C-1.1.3B` 的协议修复轮补充冻结。当前现行协议请以 `docs/SOFTWARE/C-1.1.3B_protocol_freeze.md` 为准。

## Summary
- 本轮基线为当前 `feature/c-1.1.2b-fan-auto-control`；先把现有 `libraries/Tyler_1/examples/Tyler_1/Tyler_1.ino` 本地差异纳入本轮，而不是回退。
- 本轮同时完成两件事，并入同一版本 `C-1.1.3`：
  - 蓝牙协议改造：`T -> GEST`，`G -> FAN TOGGLE`
  - 编译入口兼容：保留仓库脚本为主标准，同时补齐 Arduino IDE 直接 Verify 的库发现方案

## Key Changes
- 蓝牙协议与事件模型：
  - 在权威入口 `sketches/c002_uno_baseline/c002_uno_baseline.ino` 中保留 `cmdChar` 作为持久模式/运动命令，新增单独的 `fanTogglePending` 布尔事件位承接一次性 `G`。
  - `loop()` 中收到 `G` 时只置位 `fanTogglePending`，不覆盖 `cmdChar`；收到其他字符时继续更新 `cmdChar`。
  - `runMode()` 中模式解析冻结为：`A -> AUTO`、`M -> MAN`、`T -> GEST`；删除旧手势入口路径。
  - `G` 事件只消费一次：`MAN/GEST` 下触发手动风机翻转，`AUTO` 下直接忽略并清空事件位。
  - `G` 不得把当前人工运动强制改成 `5`；用户已确认“翻转风机后保持当前运动”。

- 风机模块扩展为双控制源：
  - 在 `libraries/CleanScoutFan/` 内新增 `ControlSource`，固定为 `AUTO_SOURCE` 和 `MANUAL_SOURCE`。
  - 模块新增手动锁存 `manualLatchOn`，A2 输出始终由模块统一管理。
  - 公共接口扩展为至少包含：
    - `setControlSourceAuto(unsigned long nowMs)`
    - `setControlSourceManual(bool preserveManualLatch)`
    - `toggleManualLatch()`
    - `forceOff()`
    - `getControlSource() const`
    - `isManualLatchOn() const`
  - `AUTO_SOURCE` 下继续沿用 `C-1.1.2B` 自动状态机：3 秒冷却、6 秒运行、障碍暂停、转弯后重新冷却。
  - `MANUAL_SOURCE` 下不跑自动计时，只把 `manualLatchOn` 映射到 A2：`true -> HIGH`，`false -> LOW`。
  - `pauseByObstacle()` / `notifyTurnCompleted()` 仅在 `AUTO_SOURCE` 下有效；人工控制域调用时必须无副作用。

- 模式切换与控制权交接：
  - 在 sketch 中显式记录 `previousMode`，在模式解析后立即做风机控制权交接，再消费 `fanTogglePending`，最后调用 `fan.update(systemMode, millis())`。
  - `MAN/GEST -> AUTO`：调用 `setControlSourceAuto(millis())`，清空手动锁存，关闭输出，并从该时刻重新开始完整 3 秒冷却。
  - `AUTO -> MAN/GEST`：调用 `setControlSourceManual(false)`，退出自动控制，默认风机关，必须再次收到 `G` 才能开启。
  - `MAN <-> GEST`：调用 `setControlSourceManual(true)`，保留当前手动风机状态，不自动关机。

- 旧示例同步与仓库口径统一：
  - `libraries/Tyler_1/examples/Tyler_1/Tyler_1.ino` 同步到新协议和新风机模块行为，避免仓库内同时保留两套冲突协议。
  - 同步时保留当前本地已确认的方向修正 `Tyler_1(1, 0, 1, 1, ...)`。
  - 本轮不处理该旧示例中与 `C-1.1.1` 硬件冻结文档存在的 D9/D10 文字冲突；那是独立议题，不与本轮混合。

- IDE 直接编译兼容方案：
  - 新增 Windows 一次性安装脚本，例如 `tools/install_sketchbook_links.ps1`。
  - 该脚本默认操作 `%USERPROFILE%\Documents\Arduino\libraries`，创建目录联接（junction），把以下仓库库映射到 Arduino sketchbook：
    - `libraries/CleanScoutFan`
    - `libraries/Tyler_1`
    - `Adafruit-Motor-Shield-library-master`
  - 如果目标已存在且不是指向仓库的联接，脚本必须停止并报冲突，不自动覆盖用户现有库。
  - `README.md` 增加两套入口说明：
    - 官方主标准：仓库脚本 / VSCode task
    - 兼容入口：先运行联接脚本，再使用 Arduino IDE 直接 Verify
  - `README.md` 明确写出：`"Servo.h" 对应多个库` 只是提示，不是这次 `CleanScoutFan.h` 缺失的根因。

- 文档冻结：

## Public Interfaces
- 蓝牙协议冻结为：
  - `A -> AUTO`
  - `M -> MAN`
  - `T -> GEST`
  - `G -> FAN TOGGLE`
  - `1/2/3/4/5/6/7/8/9` 保持运动控制
- `CleanScoutFan` 对外接口新增控制源概念：
  - `AUTO_SOURCE`
  - `MANUAL_SOURCE`
  - 手动锁存与控制权切换接口
- 现有 `MAN / AUTO / GEST` 模式枚举和值不变；默认模式和既有模式框架不变。

## Test Plan
- 构建验证：
  - `.\tools\compile_uno_cli.ps1` 编译权威入口通过。
  - `.\tools\compile_uno.ps1` 回退编译权威入口通过。
  - 运行联接脚本后，用 Arduino IDE 直接 Verify `sketches/c002_uno_baseline` 通过。
  - 同步后的 `libraries/Tyler_1/examples/Tyler_1/Tyler_1.ino` 也能编译通过。
- 协议验证：
  - `M` 进入手动模式并停车。
  - `T` 进入体感模式，行为与旧体感入口路径等价。
  - `G` 在 `MAN` 下每发一次翻转一次风机，无重复抖动。
  - `G` 在 `GEST` 下同样有效。
  - `G` 在 `AUTO` 下被忽略，不打断自动风机状态机。
  - 一次 `G` 不会因 `cmdChar` 常驻而重复翻转。
  - 先发运动指令再发 `G`，风机翻转后当前运动保持不变。
- 控制权验证：
  - `MAN/GEST -> AUTO`：手动锁存被释放，自动风机从完整 3 秒冷却重新开始。
  - `AUTO -> MAN/GEST`：自动控制立即失效，人工域默认风机关，必须再次 `G` 才能开启。
  - `MAN <-> GEST`：手动风机状态保留。
- 回归验证：
  - 默认启动模式未改。
  - 自动避障流程未改。
  - 底盘动作语义未改。
  - 主程序和旧示例中都不存在散写 A2 业务控制。

## Assumptions
- 本轮版本边界已确定：把“IDE 直接编译兼容”并入 `C-1.1.3`，不再单独拆热修轮。
- 仓库脚本仍是主标准入口；Arduino IDE 直接 Verify 是受支持的兼容入口，但必须先运行一次 sketchbook 联接脚本。
- `G` 的事件消费采用“独立事件位 + 保留当前运动命令”方案，不采用“处理后写回 `5`”方案。
- 当前 `libraries/Tyler_1/examples/Tyler_1/Tyler_1.ino` 的本地方向修正视为有效基线，应在本轮一并纳管。
