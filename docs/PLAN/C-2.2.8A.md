# C-2.2.8A：UNO 侧 M3/M4 诊断与 W 协议观测增强

## Summary
- 本轮只处理本地端：`UNO` 代码、`UNO` 诊断文档、以及本地维护的共享串口契约文档。
- 不再为树莓派端编写 `Raspberrypi/` 目录下的发布文档；`C-2.2.8B` 不属于本地端实施范围。
- 当前起始事实直接冻结：
  - `UNO` 协议继续保持 `W,w1,w2,w3,w4`
  - `PID,...` 代码里已经存在
  - 你本地实测表明 `ENC_SIGN_W3 = -1` 能显著改善 `M3`
- 当前新增阶段性结果：
  - `W4` 根因已定位为电机驱动板接触不良
  - 接触问题排除后，四轮均已可转动
  - 当前联调参数推进为 `Kp=2.0`、`Ki=0.5`
  - 在 `W,500,500,500,500` 下四轮已实现齐转
- 本轮唯一目标是把 `M3` 和 `M4` 的问题拆开诊断，不回退协议、不碰树莓派桥、不推进 `cmd_vel -> W`。

## Key Changes
### 1. M3：冻结为方向符号修正，不采用 A/B 交换长期方案
- `wheel3` 物理映射继续保持：
  - `pinA = A4`
  - `pinB = A5`
- 不把“交换 A/B”写成正式修复方案。
- 正式冻结：
  - `ENC_SIGN_W3 = -1`
- 低速验证只用：
  - `W,80,80,80,80`
  - `W,120,120,120,120`
- 验证标准固定为：
  - `ENC` 里 `v3` 与 `v1/v2` 同号
  - `PID` 里 `meas3/out3` 不再呈现明显反向放大
  - 低速下 `M3` 不再明显暴冲

### 2. M4：先做 raw 探测，再决定查控制还是查硬件
- 新增编译期开关，例如：
  - `CSR_WHEEL4_RAW_PROBE`
- raw 模式下固定行为：
  - 不跑整车闭环
  - 不跑四轮 `PI`
  - 只初始化 `wheel4`
  - 串口持续打印：
    - `RAW4,ticks,currentA,currentB`
- `wheel4` 继续按当前物理真相：
  - `pinA = D9`
  - `pinB = D10`
- raw 诊断结论固定分两类：
  - `RAW4` 无 tick：
    - 结论写为“Port B 输入链未通过”
    - 下一步查 `D9/D10` 物理路径、servo 口导通、接头/焊点
  - `RAW4` 有 tick：
    - 说明 `Port B / PCINT0` 路径可用
    - 再回到闭环模式优先测试 `ENC_SIGN_W4`
- 本轮禁止跳过 raw 直接调 `Kp/Ki` 或几何参数。

### 3. 协议与观测保持不变，只补 raw 调试入口
- `UNO` 主协议继续冻结为：
  - 输入：`W,w1,w2,w3,w4`
  - 输出：
    - `CSR_UNO_READY`
    - `ACK:W`
    - `ERR:<reason>`
    - `ENC,t1,t2,t3,t4,v1,v2,v3,v4`
    - `PID,target1,target2,target3,target4,meas1,meas2,meas3,meas4,out1,out2,out3,out4`
- 本轮唯一新增调试输出：
  - `RAW4,ticks,currentA,currentB`
- 正式/联调 timeout 继续保留双口径：
  - 正式：`400ms`
  - 联调：当前继续用较长 timeout
- `WHEEL_KI` 继续固定为 `0.0`，本轮不加 `I`、不加 `D`。

### 4. 本地端文档范围
- 本地端本轮只交付：
  - `docs/SOFTWARE/C-2.2.8A_uno_m3_m4_diagnosis.md`
  - 必要时更新本地共享契约文档 `docs/SOFTWARE/C-2.2.7_serial_protocol_contract.md`
- 不新增、不更新任何 `Raspberrypi/` 目录下的 `C-2.2.8B` 文档。
- `C-2.2.8A_uno_m3_m4_diagnosis.md` 必须写清：
  - `M3` 最终是否通过 `ENC_SIGN_W3 = -1` 解决
  - `M4` raw 探测结果
  - `D9/D10` 是否有 tick
  - 是否仍需要继续查 `Port B` 硬件路径
  - 当前 `PID,...` 已经存在，不再把“补 PID”写成待办

## Public Interfaces
- 本轮不改现有联调协议：
  - 输入：`W,w1,w2,w3,w4`
  - 输出：`CSR_UNO_READY` / `ACK:W` / `ERR:<reason>` / `ENC,...` / `PID,...`
- 新增的仅是本地调试接口：
  - `RAW4,ticks,currentA,currentB`
- `CMD,vx,vy,wz`、`cmd_vel -> W`、odom 语义都不属于本轮实施范围。

## Test Plan
- **M3 低速验证**
  - 用 `W,80,80,80,80`、`W,120,120,120,120`
  - 看 `ENC/PID` 而不是只看轮子视觉现象
- **M4 raw 探测**
  - 抬车，正拨/反拨 `wheel4`
  - 观察 `RAW4` 的 tick 与 `currentA/currentB`
  - 先得出“Port B 通 / 不通”的结论，再决定后续动作
- **回归验证**
  - `c003` 闭环模式能编译
  - `c003` raw 模式能编译
  - `c002` 回退线不被破坏
- **通过标准**
  - `M3` 通过 `ENC_SIGN_W3 = -1` 稳定下来，且不改 A/B 物理真相
  - `M4` 至少拿到 raw 结论，不再停留在“猜 D9/D10 行不行”
  - `W` 协议、`ACK:W`、`ENC`、`PID` 保持不变

- `M4` 已定位为驱动板接触问题，不再停留在 `Port B` 能力猜测
- 当前 `PI` 参数实测推进到 `Kp=2.0`、`Ki=0.5`
- `W,500,500,500,500` 下四轮已齐转
- 当前可对项目负责人汇报为：小车已取得“轮级闭环第一版可运行实现”的阶段性成果

## Assumptions
- 当前本地工作树里 `ENC_SIGN_W3 = -1` 是用户已实测有效的临时改动；本轮实施默认将其正式化，而不是回滚后重试。
- `PID,...` 已在 `C-2.2.7A` 代码中实现，因此本轮只把它作为诊断工具使用。
- `wheel1~wheel4` 与物理轮位的最终对应关系本轮不重定义；诊断全部按当前逻辑编号进行。
- `c002_uno_baseline.ino` 的用户脏改动继续保持不碰。
