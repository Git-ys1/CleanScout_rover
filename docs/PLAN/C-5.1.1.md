# 任务书 C-5.1.1：D430/D435 深度抓取管线与 ROS 预留接口

## 一、任务背景

当前 CleanScout_rover 的 OrangePi/rk3588_ai 已经完成 YOLO11/RKNN/USB 摄像头实时识别和机械臂二维视觉追踪 dry-run。下一阶段进入“视觉识别追踪抓取”开发。开发期暂不接 ROS，便于频繁迭代；但本轮代码不能把后续 ROS 接口堵死，必须保留可接 ROS2/Noetic 的边界。

现有硬件状态：

- 今天可用 Intel D430：只有深度，没有 RGB；本轮只用于测试深度流、深度 ROI、中位数滤波、像素深度反投影、坐标变换前半段，不做 YOLO RGBD 抓取闭环。
- 明天到货 Intel D435：有 RGB + Depth；到货后执行 YOLO color bbox + aligned depth-to-color + 抓取闭环。
- 当前 RF1 是 STM32F1 下位机，适配总线舵机。RF4 学习包也是同厂家总线舵机执行层，可迁移优化 RF1 串口收帧、解析、ACK/ERR、watchdog、限位等执行层能力。

## 二、硬边界

1. IK、深度相机处理、手眼变换、抓取状态机全部运行在 OrangePi。
2. RF1 只做总线舵机执行层，不做 IK、不接 D430/D435、不处理 YOLO、不根据图像面积估距。
3. 当前阶段不跑 ROS 节点，但新增模块必须保留 ROS 兼容数据边界；后续接 ROS 时只允许增加 adapter，不重写 IK/geometry/state machine。
4. 不允许用 bbox 面积估距。距离只能来自 D430/D435 深度图 ROI 或真实标定数据。
5. 现有 `arm_tracking_demo/` 不删除，作为搜索/居中阶段继续保留；抓取管线新增在 `arm_grasp_pipeline/`。

## 四、OrangePi 开发任务

### T1：合入并跑无硬件 dry-run

命令：

```bash

python3 tools/ik_sweep_check.py
python3 tools/mock_grasp_cycle.py
```

验收：

- IK 能对 4 个测试点输出解。
- mock 抓取周期能打印完整 `{#000P...!#001P...!...}` 多舵机命令。
- 不连接 RF1 时不得打开真实串口。

### T2：今天用 D430 测深度链路

命令：

```bash
cd ~/rk3588_ai/arm_grasp_pipeline
python3 tools/d430_depth_smoke.py --frames 80
```

验收：

- 能打印 depth shape、intrinsics、中心 ROI 深度中位数。
- `point_cam_m` 随物体远近变化合理。
- 深度无效时返回 `None`，不进入抓取动作。

注意：D430 没 RGB，今天不做 YOLO + RGBD 对齐，不做真实抓取。今天只验证“深度图 -> ROI depth -> camera xyz”。

### T3：明天 D435 到货后接 RGBD

命令：

```bash
cd ~/rk3588_ai/arm_grasp_pipeline
python3 tools/d435_smoke.py
```

验收：

- color/depth 尺寸一致。
- depth 已 aligned 到 color。
- bbox 中心 ROI 能取稳定深度。

随后把现有 YOLO bbox 输出转换为：

```python

```

送入：

```python
gsm.update_detection(bbox)
gsm.try_lock_depth(frame.depth_m)
```

### T4：保留 ROS 后续入口

当前不启动 ROS，但要求所有状态机关键输出通过 `ros_compat.py` 的 `GraspEventMsg` 和 `ArmTargetMsg` 走一层接口。当前默认 `NullRosBridge`，调试可用 `PrintRosBridge`。后续接 ROS2/Noetic 时，只写一个新 adapter：

```text
GraspEventMsg -> ROS topic
ArmTargetMsg  -> ROS topic / action / service
```

不得把 ROS import 写进 IK、geometry、target_depth、serial_servo_adapter。

## 五、RF1 / STM32 开发任务

### F1：迁移 RF4 的总线舵机执行层能力

参考 RF4：

```text
CODE/src/arm_control.c
Core/Src/usart.c
Core/Src/freertos.c
Core/Src/y_servo.c
```

迁移目标不是照搬工程，而是把 RF1 的总线舵机执行层补强：

- 独立 OrangePi 控制串口 RX buffer。
- 支持 `#000P1500T1000!` 单舵机帧。
- 支持 `{#000P1500T1000!#001P1600T1000!...}` 多舵机原子帧。
- 每帧返回 ACK/ERR。
- 加 watchdog、限位、时间上限、速度斜坡。
- 明确 003 号舵机反向只在一端保留。

### F2：ST-Link 烧录验证

验收顺序：

1. 单独烧录 RF1 改动，先不接机械臂负载。
2. 串口发 `#000P1500T1000!`，收到 ACK。
3. 串口发 `{#000P1500T1000!#001P1500T1000!}`，收到 ACK 且两轴同步进入目标。
4. 发非法帧，收到 ERR，不动作。
5. 停止上位机心跳，watchdog 进入 hold/stop。

## 六、明确不做事项

本轮不做：

- ROS 节点正式接入。
- D430 真实抓取。
- bbox 面积估距。
- RF1 IK。
- 下位机视觉处理。
- 删除或重写现有 2D 追踪 demo。

## 七、交付物验收标准

1. 代码能在普通 Python 环境跑 dry-run。
2. D430 depth-only smoke 能跑到 intrinsics/depth/point_cam 输出。
3. D435 到货后只需替换 source，即可进入 aligned RGBD 测试。
4. RF1 任务有清晰迁移边界和串口验收路径。
5. ROS 后续接口已预留，非 ROS 快速迭代不受影响。
