# C-1.2.1：F411 平台层补齐与实板闭环打通

## Summary
- 基线从 `feature/c-1.2.0-cj-comm-poc` 出发，新建 `feature/c-1.2.1-f411-bridge-bringup`，先做 `checkpoint: before C-1.2.1`。
- 本轮只打通现有三段 scaffold，不扩协议、不切换端点、不重构 UNO/J 结构。
- `F411` 是本轮主施工对象：补齐 `USART2(PA2/PA3)`、`USART1(PA9/PA10)` 的真实平台层和 IRQ，把当前桥接状态机从“可编译”变成“可实板收发”。
- `UNO` 只做最小 bring-up 补丁：保留 boot=`MAN`，增加 USB 串口调试入口用于切到 `AUTO` 或人工持续动作态，并打印 `STOP_REQ/RESUME_REQ` 接收标记。
- `OpenMV` 继续作为 `J` 侧协议端点，但必须把 `WAIT_PICK_WINDOW` 做成可观测、可超时、可重发，避免现场看起来像“死掉”。

## Key Changes

### 1. F411：补齐真实平台层，不改桥接状态机
- 保留现有 [`cj_bridge.c`](/f:/Project/CleanScout_rover/firmware/cj_bridge_f411_cubeide/Core/Src/cj_bridge.c) 状态机和协议逻辑，只补 [`main.c`](/f:/Project/CleanScout_rover/firmware/cj_bridge_f411_cubeide/Core/Src/main.c) 的平台函数与 IRQ。
- 平台实现固定采用“直接寄存器 + HSI 16MHz”最小方案，不引入 HAL/Cube 自动生成：
  - `USART2`：`PA2=TX`、`PA3=RX`，`AF7`，`9600 8N1`
  - `USART1`：`PA9=TX`、`PA10=RX`，`AF7`，`9600 8N1`
  - `BRR` 固定按 `16MHz / 9600` 配置
  - 打开 `GPIOA`、`USART1`、`USART2` 时钟
  - RX 采用中断驱动，`USART1_IRQHandler/USART2_IRQHandler` 在 `RXNE` 时读取 DR 并调用 `usart1_rx_isr()/usart2_rx_isr()`
  - TX 先用阻塞发送实现 `platform_send_uno_line()` 和 `platform_send_j_bytes()`
- 启动文件补齐 `USART1/USART2` IRQ 向量，保留 `SysTick`。
- 在 `main.c` 增加最小 bring-up keepalive：
  - `IDLE` 状态下，若链路未确认，每 `1000 ms` 向 UNO 发一次 `PING\n`
  - 同时每 `1000 ms` 向 J 发一次 `PING` 帧
  - 用于 Gate1/Gate2/Gate3 单段 bring-up，不改变主协议集合
- F411 可观测性固定为“PB2 灯码 + 调试状态变量”，不占用第三路串口：
  - `IDLE`：1 短闪
  - `WAIT_UNO_STOP_ACK`：2 短闪
  - `WAIT_J_RESULT`：3 短闪
  - `WAIT_UNO_RESUME_ACK`：4 短闪
  - `FAULT`：快速连续闪
  - 另存 `volatile` 诊断变量：当前状态、最近 fault reason、USART1/2 RX 字节计数、USART1/2 错误计数、最近状态迁移时间
- `platform_log()` 不再是空桩，固定改成更新 `last_fault_reason` / `last_log_reason` 这类调试变量，方便 ST-LINK 观察。
- `tools/build_f411_bridge.ps1` 继续作为最小构建入口，更新产物和文档；不切换到完整 `.ioc` 工程。

### 2. UNO：只补 bring-up 调试入口，不改主流程语义
- 保留 [`c002_uno_baseline.ino`](/f:/Project/CleanScout_rover/sketches/c002_uno_baseline/c002_uno_baseline.ino) 现有 `bridgeSerial(A3/A4)`、`STOP_REQ/RESUME_REQ/PING` ASCII 协议和 `externalPauseActive` 语义。
- 固定新增 USB `Serial` 调试入口，波特率 `115200`：
  - 仅在 `CJ_COMM_POC_BUILD` 下启用
  - 接受单字符 `A/M/T/1..9/G`，复用现有 `classifyBluetoothCommand()` / `cmdChar` 逻辑
  - 用于 bring-up 时把车切到 `AUTO`，或切到 `MAN` 后给持续动作指令做“可被打断再恢复”的过渡验证
- 固定新增两条最小日志：
  - 收到 `STOP_REQ`：`CJ STOP_REQ`
  - 收到 `RESUME_REQ`：`CJ RESUME_REQ`
- 不改 boot=`MAN`，不恢复 HC-06，不碰风机、蓝牙协议或底盘行为语义。
- 文档里明确：Gate4 优先用 `AUTO` 做正式闭环；若自动流还没稳定，可先用 `MAN + 持续前进命令` 验证“持续动作态被截停并恢复”。

### 3. OpenMV：等待态可观测化 + 容错重发
- 保持 [`main.py`](/f:/Project/CleanScout_rover/jixiebi/experiments/C-1.2.0_color_grasp_uart_poc/main.py) 的主逻辑不变：`SCAN -> COLOR_FOUND -> WAIT_PICK_WINDOW -> PICKING`，不改回“看见就抓”。
- 固定增加顶部可调参数区，集中定义：
  - 红/黄/蓝阈值
  - `PIXELS_THRESHOLD`
  - `STABLE_FRAMES_REQUIRED`
  - `VERTICAL_BIAS`
  - `WAIT_PICK_WINDOW_TIMEOUT_MS = 2000`
  - `COLOR_FOUND_RETRY_INTERVAL_MS = 400`
  - `MAX_COLOR_FOUND_RETRIES = 4`
- 增加 `set_state()` / `state_label()` 辅助，把状态切换显式化：
  - `SCAN`
  - `COLOR_FOUND`
  - `WAIT_PICK_WINDOW`
  - `PICKING`
  - `DONE`
  - `TIMEOUT`
  - `FAIL`
- 可观测性固定为“两层反馈同时保留”：
  - 图像叠加状态文本
  - OpenMV IDE 终端在状态切换时打一行短日志
- `WAIT_PICK_WINDOW` 行为固定改为：
  - 首次进入时已发送一次 `COLOR_FOUND`
  - 等待期间每 `400 ms` 重发一次 `COLOR_FOUND`
  - 超过 `2000 ms` 或超过 `4` 次重发仍未收到 `PICK_WINDOW`，退回 `STATE_SCAN`
  - 收到 `ABORT` 立即退回 `STATE_SCAN`
- `PICK_WINDOW` 到达后立即切 `STATE_PICKING`，并发送 `ARM_BUSY`
- `DONE/TIMEOUT/FAIL` 增加短暂可视停留窗口（推荐 `1000 ms`）再回 `SCAN`，避免肉眼看不到终态。

## Public Interfaces
- **不变**：
  - `UNO <-> F411` ASCII 行协议：`PING/PONG/STOP_REQ/STOP_ACK/RESUME_REQ/RESUME_ACK`
  - `F411 <-> OpenMV` 帧协议：消息类型和帧格式不变
  - `J` 侧端点仍是 `OpenMV UART1(A9/A10)`，不切到 `J-STM32`
- **新增的 bring-up 接口**：
  - UNO USB `Serial@115200` 调试入口，接受现有单字符控制命令
  - F411 `PB2` 状态灯码作为 bring-up 可观测接口
- **行为变化**：
  - OpenMV `WAIT_PICK_WINDOW` 不再无限等待，而是“重发 + 超时回退”

## Test Plan
- **Gate1：F411 单板**
  - 烧录最新 `cj_bridge_f411.hex`
  - 观察 `PB2` 在 `IDLE` 下 1 短闪循环
  - 将 `PA2` 与 `PA3` 短接，确认 `PING/PONG` 计数或状态变量变化
  - 将 `PA9` 与 `PA10` 短接，确认 J 侧 `PING/ACK/HEARTBEAT` 路径计数变化
- **Gate2：UNO ↔ F411**
  - UNO 通过 USB 串口切到 `AUTO`
  - F411 周期 `PING`，UNO 回 `PONG`
  - 手工触发 F411 `STOP_REQ`，UNO 停车并打印 `CJ STOP_REQ`
  - 手工触发 F411 `RESUME_REQ`，UNO 恢复并打印 `CJ RESUME_REQ`
- **Gate3：OpenMV ↔ F411**
  - OpenMV 看到目标后画面显示 `COLOR_FOUND/WAIT_PICK_WINDOW`
  - F411 收到 `COLOR_FOUND` 后进入 `WAIT_UNO_STOP_ACK`
  - F411 送出 `PICK_WINDOW`
  - OpenMV 切到 `PICKING` 并回 `ARM_BUSY`
  - 终态回 `PICK_DONE / PICK_TIMEOUT / ARM_FAIL`
- **Gate4：全链路**
  - UNO 先进入 `AUTO` 并实际运动
  - OpenMV 发现颜色
  - UNO 被停下
  - OpenMV 抓取或超时
  - UNO 恢复
- **Gate5：重复性**
  - 至少 10 轮
  - 记录：`COLOR_FOUND` 触发次数、停车成功率、`PICK_WINDOW` 送达率、`PICK_DONE/TIMEOUT` 回传率、恢复成功率、是否卡在 `WAIT_PICK_WINDOW`、F411 fault 次数与原因

## Assumptions
- `C-1.2.1` 继续基于当前自写最小 F411 构建链路，不在本轮切回完整 CubeIDE 工程外壳。
- UNO bring-up 入口固定采用 USB `Serial`，这是为了不改 boot=`MAN`、不恢复 HC-06、同时仍能把车切进 `AUTO`。
- F411 调试优先采用 `PB2` 状态灯码 + 诊断变量，不新增第三路调试 UART。
- OpenMV 等待态参数默认冻结为：`2000 ms` 总等待、`400 ms` 重发间隔、`4` 次最大重发；如实板阈值需调，只允许在头部配置区改。
