# C-3.0.7：CN1/CN3 编码器链路专项排查计划

## Summary
- 静态检查结论：当前 Keil 工程只编译 `_work_3/User` 的 `main/csr_*` 自写层和 CMSIS/FWlib，没有把旧 `tracking/ps2/soft_i2c/y_encoder/y_motor/app_motor` 编进镜像；因此 PA0/PA1/PB3 被旧模块复用的概率低。
- 当前代码映射与 vendor 可见源码一致：`CN1=TIM5+PA0/PA1`，`CN3=TIM2 full remap+PA15/PB3`；协议、PI、方向表不是 CN1/CN3 无计数的首要怀疑点。
- C-3.0.7 只做 CN1/CN3 编码器链路诊断与恢复，不改 `W/M/E/STOP` 语义，不调 ROS/树莓派/运动学，不重测开环真值表。

## Key Changes
- 新建分支 `feature/c-3.0.7-openrf1-cn1-cn3-encoder-debug`，从当前 C-3.0.6 发布分支继续。
- 在 `_local/openrf1_keil_work_3/User` 增强诊断，不改闭环主流程：
  - 保留 `D,<ch>` 原始相位诊断。
  - 新增 `R,<ch>` 寄存器诊断命令，返回 `REG,<ch>,MAPR,SMCR,CCMR1,CCER,CNT`。
  - `R,0` 返回全局 remap 状态，重点输出 `AFIO->MAPR`，用于确认 `SWJ_CFG=010`、`TIM2_REMAP=full remap` 是否在实际镜像中成立。
- 在编码器初始化层增加诊断可配置项：
  - 将 `TIM_ICFilter=10` 改成每通道宏，例如 `CSR_ENCODER_FILTER_CN1/CN2/CN3/CN4`。
  - C-3.0.7 诊断默认：`CN1=0`、`CN3=0`、`CN2/CN4=10`，用于排除滤波过重导致手拨/低速霍尔波形不计数。
  - 若 CN1/CN3 恢复计数，再决定是否把滤波值调回 `10` 或取中间值。
- 新增文档 `docs/VERIFY/C-3.0.7_cn1_cn3_encoder_chain_debug.md`，记录：
  - `D,1/D,3` 相位是否变化。
  - `R,1/R,3/R,0` 寄存器状态。
  - `E,1/E,3` 是否恢复非零 delta。
  - 最终归因：硬件链路、TIM/remap 初始化、软件读取链，或多因素。

## Test Plan
- 编译与烧录：
  - Keil 编译 `_work_3` 当前工程，要求 `0 Error(s)`。
  - 通过 STLink 烧录，确认 `CSR_RF1_READY`。
- 全局 remap 验证：
  - 发送 `R,0`。
  - 验收：`SWJ_CFG=010`，`TIM2_REMAP=full remap`。
- CN1 诊断：
  - `STOP`
  - 手拨 CN1 对应轮，连续发送 `D,1` 和 `R,1`
  - 短时 `M,1,700`，再发送 `D,1 / R,1 / E,1`
  - 判定：相位不变=硬件链；相位变但 CNT 不变=TIM5/GPIO 初始化；CNT 变但 ENC 不变=读取链。
- CN3 诊断：
  - `STOP`
  - 手拨 CN3 对应轮，连续发送 `D,3` 和 `R,3`
  - 短时 `M,3,700`，再发送 `D,3 / R,3 / E,3`
  - 判定：相位不变=硬件链或 PA15/PB3 未释放；相位变但 CNT 不变=TIM2 full remap/滤波/输入模式；CNT 变但 ENC 不变=读取链。
- 回归边界：
  - `CN2/CN4` 不作为本轮调参目标，只做 `E,2/E,4` 确认仍有计数。
  - `W` 协议保持 C-3.0.6 行为，不新增上层语义。

## Assumptions
- 当前 C-3.0.6 已发布，`CN2/CN4` 两轮闭环阶段性成立，`CN1/CN3` 是局部编码器链路问题。
- 当前工程文件只编译 `_work_3/User` 自写层；旧 vendor 模块不会在镜像里覆盖 PA0/PA1/PA15/PB3。
- `TIM_ICFilter=10` 与 vendor 一致，但对诊断场景可能过重；C-3.0.7 允许把 CN1/CN3 滤波临时降到 `0` 来区分“波形进来了但被过滤”与“波形根本没进 MCU”。
- 本轮通过标准不要求四轮闭环完成；只要把 CN1/CN3 无反馈归因到明确层级，或恢复非零计数，即可通过。
