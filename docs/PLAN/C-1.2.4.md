# C-1.2.4：Vendor 直通夹爪链路证伪轮

## Summary
- 基线从 `hotfix/c-1.2.3-claw-primitive-recovery` 出发，新建 `hotfix/c-1.2.4-vendor-claw-bringup`，先做 `checkpoint: before C-1.2.4`。
- 本轮不再主攻 `main.py`、`WAIT_PICK_WINDOW`、`LOCAL_PICK_FALLBACK` 或并线逻辑，只做卖家原始 `Timer(2) + CH3 + B10` 夹爪链路的最小烟雾测试。
- 核心目标是把结论压成二选一：
  - A：卖家原始 PWM 路径能让夹爪动，说明当前抽象层/调用路径还要继续排查
  - B：卖家原始 PWM 路径也不能让夹爪动，说明问题升级为板级/固件/接线/供电问题
- `UNO` 不动，`F411` 不动，不改 `main.py` 现有抓取状态机，只新增两个最小 vendor 验证脚本和相应文档。

## Key Changes
- 在 `jixiebi/experiments/C-1.2.0_color_grasp_uart_poc/` 新增两个独立脚本，且它们都**不依赖** `claw_runtime.py`：
  - `vendor_claw_smoketest.py`
  - `vendor_pwm_sweep.py`
- `vendor_claw_smoketest.py` 固定按卖家原始链路最小复刻：
  - `from pyb import Pin, Timer`
  - `tim = Timer(2, freq=50)`
  - `claw = tim.channel(3, Timer.PWM, pin=Pin("B10"), pulse_width_percent=7.5)`
  - 原样保留 `claw_angle()` 和 `percent = (servo_angle + 45) / 18`
  - 只执行：`60 -> 等1s -> 150 -> 等1s -> 60 -> 循环`
  - 串口必须打印：`VENDOR_CLAW -> 60`、`VENDOR_CLAW -> 150`
  - 不导入 `cj_link.py`、`pid.py`、`claw_runtime.py`、`sensor` 识别逻辑或通信逻辑
- `vendor_pwm_sweep.py` 固定直接扫占空比，不再经过角度语义：
  - 使用与 vendor 相同的 `Timer(2) + CH3 + B10`
  - 依次输出 `2.5 / 5.0 / 7.5 / 10.0 / 12.5`
  - 每个点停留 `800~1000 ms`
  - 串口打印 `PWM_SWEEP -> 2.5` 等
  - 这一步专门用来排除“60/150 只是当前实物无效角度”的干扰
- `main.py` 本轮默认**不改逻辑**：
  - 不继续扩写 `WAIT_PICK_WINDOW`
  - 不继续扩写 `LOCAL_PICK_FALLBACK`
  - 不再优先改抓取状态机
  - 只有在需要补最小注释或入口说明时才允许触碰，且不改变运行行为
- `claw_runtime.py` 本轮默认**不改语义**：
  - 不在没有 vendor 直通结果前继续猜 `60/150`
  - 不改成 `pyb.Servo()` 方案
- 文档留痕必须新增/更新：
  - `docs/PLAN/C-1.2.4.md`
  - `docs/VERIFY/C-1.2.4_vendor_claw_bringup_log.md`
  - 更新实验目录 `test_record.md`
  - 更新实验目录 `deployment_checklist.md`
- 文档里要明确写死部署方式：
  - 跑 `vendor_claw_smoketest.py` 时，板端 `main.py` 必须来自这个脚本
  - 跑 `vendor_pwm_sweep.py` 时，板端 `main.py` 必须来自这个脚本
  - 这两个脚本是独立 smoke test，不与 `main.py` 共享入口
- 若硬件结论落到 B，文档必须同步补“板级核查清单”，至少固定记录：
  - 当前固件字符串/板型口径
  - 是否卖家定制固件
  - 夹爪信号线是否确实接到卖家定义的 `B10`
  - 夹爪 5V/GND 供电是否稳定
  - 是否加载了 LCD/WiFi/图传/串口附件
  - 是否怀疑平台 IO 复用冲突

## Public Interfaces
- `UNO` 公共接口不变：
  - `PING / PONG`
  - `STOP_REQ / STOP_ACK`
  - `RESUME_REQ / RESUME_ACK`
- `F411 <-> J` 协议不变：
  - 不改消息类型
  - 不改帧格式
  - 不继续扩状态机
- `OpenMV` 新增的只是两个独立 bring-up 入口：
  - `vendor_claw_smoketest.py`
  - `vendor_pwm_sweep.py`
- 本轮明确**不**把当前 `B10` 夹爪改成 `pyb.Servo()` 驱动，因为那已经不是同一条硬件链路验证。

## Test Plan
- **Gate1：Vendor smoke test**
  - 烧 `vendor_claw_smoketest.py`
  - 观察夹爪是否随 `60 -> 150 -> 60` 真正运动
  - 终端打印是否与动作一致
- **Gate2：Vendor PWM sweep**
  - 烧 `vendor_pwm_sweep.py`
  - 观察 `2.5 / 5.0 / 7.5 / 10.0 / 12.5` 是否任一点有物理动作
  - 记录哪些占空比有效、哪些无动作
- **Gate3：结论判定**
  - A：`vendor_claw_smoketest.py` 能动
    - 说明 vendor 直通链路活着
    - 下一轮回头排查 `claw_runtime.py` / `main.py` 调用层
  - B：`vendor_claw_smoketest.py` 和 `vendor_pwm_sweep.py` 都不动
    - 说明问题升级为板级/固件/接线/供电问题
    - 下一轮不再从 Python 抽象层入手，而进入 J 线板级 bring-up
- **记录要求**
  - 文档必须明确写：
    - `vendor_claw_smoketest.py` 是否动了
    - `vendor_pwm_sweep.py` 是否动了
    - 实测哪些占空比试过
    - 与当前 `claw_runtime.py` 行为是否一致
    - 最终结论是 A 还是 B

## Assumptions
- 基线默认采用当前 `C-1.2.3`，但本轮不会继续修改 `main.py` 的抓取流程，除非只是补说明文字；核心验证全部通过新独立脚本完成。
- `vendor_claw_smoketest.py` 和 `vendor_pwm_sweep.py` 都应尽量 verbatim 复刻卖家 `Timer(2) + CH3 + B10` PWM 路径，避免再经过 `ClawRig` 这层抽象。
- 本轮不会尝试 `pyb.Servo()` 替代当前 `B10` 夹爪，因为那会把“同一硬件链路是否存活”的问题混成另一轮改接线实验。
- 如果 Gate3 得出 B，本轮文档结论将明确把问题升级为板级/固件/接线/供电核查，而不是继续猜 Python 角度语义。
