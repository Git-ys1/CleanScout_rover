# J-0.3.0：黑色电容 / 金色五金磁吸接近基线轮

## Summary
- 先把当前 `feature/c-1.2.6-vendor-vision-grasp-baseline` 的脏树收敛为干净 checkpoint，再从它新开 `feature/j-0.3.0-black-gold-magnetic-pick`。当前必须先保留并纳入的本地差异是 `jixiebi/experiments/C-1.2.6_vendor_vision_grasp_baseline/main.py` 的本地改动。
- 本轮 scope 按你刚刚的明确指令，正式改成**双目标顺序轮**：先做黑色电容，再做金色螺帽；两者都在同一轮落地，但运行时始终一次只开一个 `TARGET_MODE`，不做混合多目标同时追踪。
- 新主线不再推进 `F411`、`WAIT_PICK_WINDOW` 或旧 `C-1.2.0_color_grasp_uart_poc`。`J-0.3.0` 只从 `C-1.2.6` 成功视觉基线 fork 出新的“小件目标识别 + 磁吸接近 + 抬起保持”基线。
- 目标状态机改为：`SCAN -> TRACK -> APPROACH_READY -> CONTACT -> LIFT -> HOLD -> RESET`。本轮不做自动释放，不做投放，不做闭爪分拣。
- 黑色电容和金色螺帽都必须先做手工磁铁试吸记录；软件链路可以先建好，但“磁吸成功”结论只以实物试吸结果为准。

## Key Changes
### 1. 新实验目录与基线留痕
- 新建 `jixiebi/experiments/J-0.3.0_magnetic_pick_baseline/`，固定包含：
  - `vendor_baseline_snapshot/`
  - `main.py`
  - `vision.py`
  - `actuator.py`
  - `pid.py`
  - `README.md`
  - `test_record.md`
  - `threshold_notes.md`
- `vendor_baseline_snapshot/` 直接复制当前 `C-1.2.6_vendor_vision_grasp_baseline/` 的运行时文件与说明，作为可回退基线证据；新目录根部是 `J-0.3.0` 的接管版。
- `README.md` 和 `test_record.md` 必须明确写出：本轮从 `C-1.2.6` fork，`F411` 联调暂停，太阳能板问题不纳入本轮。

### 2. 执行层：继承边界，改为磁吸动作语义
- 新 `actuator.py` 继续继承 `C-1.2.5` 的唯一真值：
  - `PAN_CENTER_DEG = 0`
  - `PAN_MIN_DEG = -90`
  - `PAN_MAX_DEG = 90`
  - `TILT_CENTER_DEG = 85`
  - `TILT_MIN_DEG = 0`
  - `TILT_MAX_DEG = 90`
  - `CLAW_OPEN_ANGLE = -60`
  - `CLAW_CLOSE_ANGLE = 40`
- 夹爪语义本轮改为“默认常闭”：
  - 上电和主流程默认执行 `claw_close()`
  - `claw_open()` 仍保留为调试/回退接口，但不进入主流程
- 执行器只允许通过这些接口驱动：
  - `pan_set(angle)`
  - `tilt_set(angle)`
  - `claw_close()`
  - `claw_open()`
  - `search_pose()`：`pan=0, tilt=85, claw=closed`
  - `contact_pose()`：接触位，初始冻结为 `tilt=90`
  - `lift_after_contact()`：接触后抬起位，初始冻结为 `tilt=70`
  - `hold_pose()`：保持位，初始冻结为 `tilt=70`
  - `center_pose()` 仅作调试接口，不再作为主流程默认位
  - `clamp_pan(angle)`
  - `clamp_tilt(angle)`
- 运行时默认动作常量先冻结为：
  - `CONTACT_TILT_DEG = 90`
  - `LIFT_HOLD_TILT_DEG = 70`
  - `CONTACT_HOLD_MS = 500`
  - `LIFT_SETTLE_MS = 800`
  - `HOLD_SETTLE_MS = 800`
- 所有视觉和状态机代码禁止再直接写 `Servo(3/4/1).angle(...)`。

### 3. 视觉层：从大色块抓取改成小件单目标检测
- 新 `vision.py` 继续保留 `find_target()` 框架，但新增目标模式：
  - `TARGET_MODE_BLACK_CAP`
  - `TARGET_MODE_GOLD_HARDWARE`
- 这轮将 `TARGET_MODE_GOLD_HARDWARE` 实际冻结为“金色螺帽”预设；金色螺丝不在本轮实现，只在文档里作为后续对象候选保留。
- 阈值不允许凭空猜写；实现上要提供明确的配置块，但文档和记录必须要求这些值来自 OpenMV IDE 的 `Histogram + Threshold Editor` 现场采样。
- `vision.py` 必须新增逐 blob 二级过滤，不再只看最大面积：
  - 黑色电容预设：
    - `min_pixels`
    - `aspect_ratio_min/max`
    - `density_min`
    - `solidity_min`
    - 必要时中心 ROI 限定
  - 金色螺帽预设：
    - `min_pixels`
    - `roundness_min`
    - `compactness_min`
    - `density_min`
    - 排除细长 blob
- 实现方式冻结为：
  - `find_target()` 先按颜色阈值取 blobs
  - 再走目标模式对应的 `blob_passes_filter(mode, blob)` 过滤
  - 最后从通过过滤的候选里取最可信一个
- `PIXELS_THRESHOLD` 不再沿用 `500` 全局固定值；改成按目标模式区分的配置项，并在 `threshold_notes.md` 中记录实际测得值。
- 距离代理仍沿用 `blob width` 公式框架，但不再沿用 `13500/(w*2), 90..110` 的旧大方块参数；改成每个目标模式各自一组：
  - `DISTANCE_NUMERATOR`
  - `APPROACH_DISTANCE_MIN`
  - `APPROACH_DISTANCE_MAX`
- 这些数值的填充顺序冻结为：先黑色电容，再金色螺帽；实测前代码里可以留模板常量和 `TODO` 标记，但报告里不得把模板值写成已验证结果。

### 4. 主状态机：从抓取投放改成接触抬起保持
- 新 `main.py` 继续保留 `SCAN / TRACK / APPROACH_READY`，但把后半段状态改成：
  - `CONTACT`
  - `LIFT`
  - `HOLD`
  - `RESET`
- 运行时默认参数：
  - `TARGET_MODE = TARGET_MODE_BLACK_CAP`
  - `ENABLE_SECOND_TARGET = False`
  - 黑色电容跑通后，再切 `TARGET_MODE = TARGET_MODE_GOLD_HARDWARE`
- 状态行为固定为：
  - `SCAN`
    - `search_pose()`
    - 无目标则维持闭爪搜索位
  - `TRACK`
    - 只做云台 PID 跟踪
    - PID 输出必须经过 `clamp_pan()` / `clamp_tilt()`
  - `APPROACH_READY`
    - 满足中心偏差与距离窗口后累计稳定帧
  - `CONTACT`
    - 执行 `contact_pose()`
    - 保持 `CONTACT_HOLD_MS`
  - `LIFT`
    - 执行 `lift_after_contact()`
  - `HOLD`
    - 执行 `hold_pose()`
    - 维持短暂观察窗口，用于判断目标是否随之抬起
  - `RESET`
    - 回到 `search_pose()`
- 本轮主链路不再调用：
  - `perform_grab_cycle()`
  - `drop_pose_for(...)`
  - `claw_open()` 主流程释放
- 夹爪动作语义只剩“保持闭合 + 磁吸接近”，闭爪不作为抓住动作，只作为默认姿态与防掉落姿态。

## Test Plan
- **Gate0：基线冻结**
  - 先把当前 `C-1.2.6` 脏树收敛成干净基线
  - `vendor_baseline_snapshot/` 可回退
- **Gate1：黑色电容磁性可行性**
  - 先做手工磁铁试吸
  - 记录“可吸 / 不可吸 / 只能接触不能吸起”
- **Gate2：黑色电容视觉识别**
  - 黑色电容能稳定出框
  - `SCAN -> TRACK` 稳定
  - 不被背景阴影大量误检
- **Gate3：黑色电容磁吸主链**
  - `TRACK -> APPROACH_READY -> CONTACT -> LIFT -> HOLD -> RESET`
  - 记录目标是否跟随抬起
- **Gate4：金色螺帽磁性可行性**
  - 手工磁铁试吸
  - 记录“可吸 / 不可吸 / 高光干扰严重程度”
- **Gate5：金色螺帽视觉识别**
  - 只开 `TARGET_MODE_GOLD_HARDWARE`
  - 形状过滤后能稳定框出螺帽，不被高光和细长干扰误导
- **Gate6：金色螺帽磁吸主链**
  - 同样验证 `CONTACT -> LIFT -> HOLD`
- **Gate7：稳定性**
  - 每个目标至少 10 轮
  - 记录成功数、失败数、失败原因、是否卡在某个状态、磁吸是否真实有效

## Assumptions
- 你已经正式把原来的“单目标轮”改成“双目标顺序轮”：黑色电容先做，金色螺帽后做；同一时刻仍只运行一个 `TARGET_MODE`，不做黑金混合追踪。
- 当前 `feature/c-1.2.6-vendor-vision-grasp-baseline` 的本地 `main.py` 改动需要先并入干净基线，再 fork `J-0.3.0`。
- 黑色目标本轮固定为“黑色电容”，不是黑色蜂鸣器或黑色 U 盘；若目标实物改型，视为下一次范围变更。
- 金色目标本轮固定为“金色螺帽”；金色螺丝不实现，只在文档中保留为下轮候选。
- 磁吸成功与否必须以实物试吸和 `HOLD` 观察结果为准；代码可以先准备好主链，但在实测前不能在文档里宣称“磁吸拾取成功”。
