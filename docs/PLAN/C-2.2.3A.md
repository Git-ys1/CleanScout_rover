# C-2.2.3A：Tyler 库“运动最小核”解耦改造

## Summary
- 从当前 `feature/c-2.2.2a-uno-usb-manual-trim` 基线继续推进，目标是把 `UNO` 侧正式升级为“运动最小核”版本：`Tyler_1` 不再对超声/舵机形成运动耦合，草图只保留树莓派 USB 手动控制。
- 实现上固定采用**方案 A**：保留现有 `Tyler_1(bool,bool,bool,bool,byte,int,int,int)` 构造函数，并加入禁用语义。`pin < 0` 或 `pin == 255` 均视为禁用；`c002_uno_baseline.ino` 统一使用 `255` 禁用态。
- 仓库里存在两套完全相同且都被跟踪的库副本：`libraries/Tyler_1` 与 `Tyler_1_Library`。编译实际使用 `libraries/Tyler_1`，但本轮必须**两套同步修改**，避免仓库内再次分叉。
- 需要新增一份说明文档 `docs/SOFTWARE/C-2.2.3A_motion_only_cutover.md`，把“历史综合库 -> 当前运动最小核”的切换口径写清。

## Key Changes
### 1. `Tyler_1` 库改成可无超声、可无舵机
- 在 `Tyler_1` 头文件与实现中新增并固定以下语义：
  - `trigPin` 或 `echoPin` 为 `-1/255` 时，超声禁用
  - `servoPin` 为 `-1/255` 时，舵机禁用
- 库内部增加最小状态成员，避免实现层再做隐式假设：
  - `ultrasonicEnabled`
  - `servoEnabled`
  - `headServoAttached`
  - `headPos`（禁用时保持哨兵值）
- 构造函数行为固定为：
  - 电机初始化永远执行
  - 只有启用超声时才对 `trig/echo` 做 `pinMode`
  - 禁用超声/舵机不得影响任何运动 API 的可调用性
- 传感器/舵机 API 语义固定为：
  - `headServoIni()`：舵机禁用时直接 no-op
  - `setHeadPos(int)`：舵机禁用时直接 no-op；启用时允许懒 attach，避免历史调用点漏掉 `headServoIni()` 时行为异常
  - `getHeadPos()`：舵机禁用时返回 `-1`
  - `getDistance()`：超声禁用时返回 `-1`
- 必须做一次静态审查并在文档中写明结论：
  - 当前库内**未发现** `forward()/turnL()` 等运动 API 直接依赖超声/舵机的阻断逻辑
  - 现有耦合点实质上是构造函数的 `pinMode` 副作用和传感器/舵机接口常驻于同一库
  - 本轮解耦的是这些副作用与接口耦合，不是“移除前进函数里的避障判断”（因为库里本来没有）

### 2. `UNO` 草图裁成 USB 手动上位控制最小版
- `sketches/c002_uno_baseline/c002_uno_baseline.ino` 固定只保留：
  - `Serial.begin(115200);`
  - 单字符手动控制
  - 启动停车
  - 超时停车
  - 串口可观测反馈
  - 板载 LED 调试
- 草图只支持以下字符：
  - `8` 前进
  - `2` 后退
  - `4` 左转
  - `6` 右转
  - `5` 停止
  - `7` 左前
  - `9` 右前
  - `1` 右后
  - `3` 左后
- 明确删除/停用全部历史主线：
  - `AUTO`
  - `GEST`
  - `M/A/T/G`
  - 蓝牙 `softSerial(9,2)`
  - `bridgeSerial(A3,A4)`
  - 风机
  - 超声自动避障
  - 舵机头部扫描
  - 一切非 USB Serial 控制入口
- 草图中的对象构造固定改为真禁用态：
  - `Tyler_1 tyler_1(1, 0, 1, 1, 200, 255, 255, 255);`
- 运行时反馈格式固定为：
  - 上电：`CSR_UNO_READY`
  - 有效命令：`ACK:<char>`
  - 非法命令：`ERR:<char>`
- 解析规则固定为：
  - 忽略 `\r` / `\n`
  - 只有 `1/2/3/4/5/6/7/8/9` 视为有效
  - 非法字符打印 `ERR:<char>`，且**不刷新**失联计时器
- 超时停车固定为：
  - `400ms`
  - 仅在最近一次有效运动命令后开始计时
  - 超时后立即 `stop()`
- 板载 LED 调试固定为：
  - 使用 `LED_BUILTIN`（UNO 即 `D13`）
  - 每次收到有效命令闪烁一次
  - 采用**非阻塞短脉冲**实现，避免 `delay()` 影响运动与串口收包

### 3. 文档与引脚口径
- 新增 `docs/SOFTWARE/C-2.2.3A_motion_only_cutover.md`，必须写清：
  - 当前串口波特率：`115200`
  - 当前支持命令与反馈格式：`CSR_UNO_READY` / `ACK:<char>` / `ERR:<char>`
  - 当前已裁掉的历史能力
  - `Tyler_1` 禁用语义如何实现
  - 当前引脚口径
- 当前引脚口径固定为：
  - 仍保留业务使用：`D0/D1`（USB Serial）、AFMotor 既有电机相关占用
  - 从主流程释放：`D2`、`D9`、`D10`、`A0`、`A1`、`A2`、`A3`、`A4`、`A5`
  - `D13` 不作业务扩展，只作本轮临时板载 LED 调试口
- 文档里必须显式写明：
  - 本轮完成后，`Tyler_1` 已达到当前 `C-2.x` 所需的“无超声、无舵机”运动最小核要求
  - 历史示例能力仍存在于库接口中，但当前实施线默认走 motion-only 口径
  - 若后续还要继续收紧库边界，可再立项把传感器/舵机 API 从库中完全拆出；但这不影响本轮通过

## Public Interfaces
- `Tyler_1` 构造函数接口名不变，但新增禁用语义：
  - `trigPin/echoPin/servoPin = -1 or 255` 表示对应能力禁用
- `Tyler_1` 相关返回值约定新增：
  - `getDistance() -> -1` 表示超声禁用
  - `getHeadPos() -> -1` 表示舵机禁用
- `UNO` USB 串口协议固定为：
  - 波特率 `115200`
  - 输入：`1/2/3/4/5/6/7/8/9`
  - 输出：`CSR_UNO_READY`、`ACK:<char>`、`ERR:<char>`

## Assumptions
- 仓库中 `libraries/Tyler_1` 是实际编译入口，`Tyler_1_Library` 是历史镜像副本；本轮两套都同步，避免后续再出现“改了文档副本没改编译副本”的分叉。
- 本轮不改 `AFMotor`、不加编码器、不接 ROS 节点、不动 `STM32/J` 线，不把范围扩成底盘重构。
- 反馈日志不增加复杂协议，只保留最小可观测字符串，便于树莓派侧串口 bring-up。
