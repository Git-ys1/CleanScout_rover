# C-3.0.5B：OpenRF1 四路自写驱动重写与真值表收口

## Summary
- 本轮不再试图用 `g_csr_motor_dir_sign[4]` 修正四路电机方向；这条路已经被当前实测否定。
- 目标只收敛两件事：  
  1. 让 `CN1~CN4` 都具备**明确的双向驱动语义**  
  2. 建立并冻结四路**驱动真值表**与**编码器真值表**
- 协议继续保持最小 bringup 口径，不扩范围：
  - `M,<ch>,<pwm>`
  - `E,<ch>`
  - `STOP`
- 本轮不做 PID、不做 `W,va,vb,vc,vd`、不做 m/s、不做树莓派/ROS、不做车体运动学。

## Key Changes
### 1. 驱动层重写方式
- 重写 `csr_motor_drv.c` 的核心控制模型：废弃“统一正负公式 + 单个 sign 表”的方案。
- 新驱动固定改为**每通道独立实现**：
  - `csr_motor_apply_cn1(int16_t pwm)`
  - `csr_motor_apply_cn2(int16_t pwm)`
  - `csr_motor_apply_cn3(int16_t pwm)`
  - `csr_motor_apply_cn4(int16_t pwm)`
- 每个函数内部单独定义该通道的：
  - `pwm > 0` 时 `IN1` 电平与 `TIM8 compare`
  - `pwm < 0` 时 `IN1` 电平与 `TIM8 compare`
  - `pwm == 0` 时停车动作
- `csr_motor_set(channel, signed_pwm)` 只负责：
  - 限幅
  - 死区提升
  - 分发到对应 `csr_motor_apply_cnx()`
- `g_csr_motor_dir_sign[4]` 本轮不再作为电机方向修正入口；最多保留为历史兼容字段，但不参与运行决策。

### 2. 真值建立顺序
- 先建立**驱动真值表**，再建立**编码器真值表**，最后再回填代码。
- 驱动真值测试固定按单通道双向执行：
  - `M,1,700` / `M,1,-700`
  - `M,2,700` / `M,2,-700`
  - `M,3,700` / `M,3,-700`
  - `M,4,700` / `M,4,-700`
  - 每步后必须 `STOP`
- 文档只记录物理现象，不写“前进/后退”高层语义。
- 当某一轮测试结果说明某通道：
  - `+700` 能转、`-700` 不转
  - 或 `-700` 能转、`+700` 不转  
  则不回填 `csr_board_map.c`，而是继续改该通道对应的 `csr_motor_apply_cnx()`，直到该通道实现双向可控。

### 3. 编码器收口方式
- `csr_encoder_drv.c` 继续只负责：
  - `count`
  - `delta`
- 本轮不把编码器接入驱动决策。
- 编码器方向修正仍只允许通过：
  - `g_csr_encoder_dir_sign[4]`
- 编码器测试固定在“已确认会转的方向”上执行：
  - `M,<ch>,±700`
  - `E,<ch>`
  - `STOP`
- 只有当该通道双向驱动都成立后，才允许根据正反 `delta` 符号回填 `g_csr_encoder_dir_sign[4]`。

### 4. 文档与代码回填纪律
- 唯一真值记录页继续使用：
  - `docs/VERIFY/C-3.0.5_openrf1_truth_table.md`
- 本轮把记录页改成两块明确表：
  - 驱动真值表：`+700 / -700 / 是否双向成立 / 当前驱动状态`
  - 编码器真值表：`count / delta / 正反符号 / 是否需反相`
- 代码回填位置收口为两处：
  - `csr_motor_drv.c`：四路独立驱动真值实现
  - `csr_board_map.c`：仅保留编码器方向表和轮位注释
- `main.c`、`csr_proto.c` 保持最小协议职责，不再承载任何电机方向语义。

## Test Plan
- **阶段 A：单通道驱动双向性**
  - 对 `CN1~CN4` 分别执行 `+700`、`-700`
  - 每步后 `STOP`
  - 验收标准：每个通道都必须出现“正值可转、负值可转”两个物理现象
- **阶段 B：单通道编码器方向**
  - 在每个通道的正向和反向都已可转之后，再执行 `E,<ch>`
  - 记录 `ENC,<ch>,<count>,<delta>`
  - 验收标准：`delta` 在正反向之间有可区分符号变化
- **阶段 C：统一语义检查**
  - 四个通道全部达到：
    - `M,<ch>,+700` = 该通道统一语义正向
    - `M,<ch>,-700` = 该通道统一语义反向
  - 这时才允许进入下一轮多通道同速与高层协议扩展
- **明确不做**
  - 不测四轮同向整车前进
  - 不测闭环
  - 不测树莓派桥接
  - 不测车体运动学组合

## Assumptions
- 当前 `C-3.0.5` 的最小协议已经打通，问题只在电机底层方向真值未建立。
- 当前这组首轮证据继续作为失败回溯点保留：
  - `CN1`: `+700=反转`, `-700=不转`
  - `CN2`: `+700=不转`, `-700=正转`
  - `CN3`: `+700=正转`, `-700=不转`
  - `CN4`: `+700=不转`, `-700=反转`
- 本轮默认继续使用 `_local/openrf1_keil_work_3/` 作为唯一自写运行层，不回退到 `_work_2` 运行语义。
- 本轮实施完成的最低标准不是“四轮同转”，而是“四个通道都具备独立、可重复、双向可控的底层真值”。
