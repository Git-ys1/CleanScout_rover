# C-2.2.1：树莓派 Noetic 接管归档与 C-2.x 边界切换任务书

## 1. 文档定位

- 版本号：`C-2.2.1`
- 本轮性质：接管、归档、边界切换入库轮
- 本轮目标：把当前树莓派 `Noetic` 镜像侧已经研究出的结论正式整理、归档、发布到云端，完成从 `C-1.x` 到 `C-2.x` 的认知切换

本轮不是功能开发任务。  
本轮不写新功能，不改底层控制代码，不先做导航、SLAM、EKF 实现。

## 2. 本轮任务目标

### 2.1 目标 A：项目负责人研究结果正式文档化

当前必须把 `C-2.x` 研究结论从聊天状态提升为仓库正式文档。

### 2.2 目标 B：树莓派环境审计与镜像接管材料统一发布

当前已形成的树莓派侧审计 / 接管材料，必须正式入库，不再散落在聊天记录或本地 staging 目录。

### 2.3 目标 C：历史冻结与当前实施边界显式分离

必须明确：

- `C-1.1.1` 是历史硬件冻结事实
- `C-2.x` 是新的实施阶段
- 不得再拿旧冻结口径直接替代当前新决策

## 3. 本轮必须完成的交付物

本轮至少交付以下 4 类内容：

- 树莓派原始审计日志正式归档
- 树莓派归档总览文档
- `C-2.x` 接管研究报告
- `C-2.x` 硬件边界更新说明
- `Noetic` 镜像接管入口审查文档

正式入口：

- `Raspberrypi/releases/C-2.2.1_noetic_handover/`
- `docs/SYSTEM/C-2.2.1_takeover_research_report.md`
- `docs/HARDWARE/C-2.2.1_hardware_boundary_update.md`
- `docs/SOFTWARE/C-2.2.1_noetic_entry_review.md`

## 4. 本轮明确不做

以下事项不纳入 `C-2.2.1`：

- 不切 `STM32`
- 不推翻当前 `Noetic` 主线
- 不直接改 `Tyler_1` 库 API
- 不擅自删 `clbrobot` 用户或大规模改系统结构
- 不先改导航、`SLAM`、`EKF` 代码
- 不先做新硬件接线拍脑袋施工
- 不跳过文档整理直接写新功能
- 不口头覆盖 `C-1.1.1` 历史文档

## 5. Git 纪律

本轮必须继续遵守：

- 新开独立分支，分支名包含 `C-2.2.1`
- 先 `checkpoint` 再施工
- 本轮优先提交文档与归档副本
- 不混入无关代码改动
- 不卷入 `J-jixiebi/`、镜像包、torrent、`package.json` 及未跟踪交接文档

建议提交节奏：

1. `checkpoint: before C-2.2.1`
2. `C-2.2.1: archive Raspberry Pi Noetic audit logs`
3. `C-2.2.1: add takeover, boundary update and Noetic review docs`
4. `C-2.2.1: add C-2.2.1 documentation entrypoints`

## 6. 结论

`C-2.2.1` 的作用不是“立刻把车改成新系统”，而是把当前树莓派 `Noetic` 接管研究结论正式收口，让后续 `C-2.x` 开发建立在可追踪、可审查、可回退的文档基线上。
