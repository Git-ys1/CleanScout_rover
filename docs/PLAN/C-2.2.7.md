# C-2.2.7：W 协议对齐与双端联调真相冻结

## Summary
- 当前首因已明确：`UNO` 端真实协议已经升级到 `W,w1,w2,w3,w4`，树莓派端仍停在 `CMD,vx,vy,wz`，两端错位的是**协议层级**，不是 `UNO` 是否“还没升级”。
- 本轮不让 `UNO` 回退，保持 `C-2.2.6A` 的 `W + ACK:W + ENC,...` 主线不变；树莓派端先降级成 **W 协议桥**，先说轮，再说车。
- 本轮同时冻结一条唯一联调真相线：
  - 集成分支：`feature/c-2.2.7-integration-pi-uno-w-bridge`
  - UNO 分支：`feature/c-2.2.7a-uno-w-protocol-align`
  - Pi 分支：`feature/c-2.2.7b-pi-w-bridge-align`
- 这轮只认 `Pi 发 W -> UNO 回 ACK:W -> ENC/PID 与动作一致`。`CMD,vx,vy,wz`、固定延时、ROS 导航链都不作为本轮通过标准。

## Key Changes
### 1. UNO 侧：冻结 W 协议，不回退到 CMD
- 以当前 `feature/c-2.2.6a-uno-encoder-pid-bridge` 为代码基线，新增 `C-2.2.7A` 文档和最小观测增强，不改协议前缀。
- 协议继续固定为：
  - 输入：`W,w1,w2,w3,w4`
  - 输出：`CSR_UNO_READY` / `ACK:W` / `ERR:<reason>` / `ENC,...`
- 必做改动只限于联调增强：
  - 增加 `PID,target1,target2,target3,target4,meas1,meas2,meas3,meas4,out1,out2,out3,out4`
  - 引入双 timeout 配置：
    - 正式模式：`COMMAND_TIMEOUT_MS = 400`
    - 联调模式：`COMMAND_TIMEOUT_DEBUG_MS = 1500` 或 `2000`
    - 用显式布尔开关选择当前编译值
  - 本轮继续固定 `WHEEL_KI = 0.0`
- 新文档：
  - `docs/SOFTWARE/C-2.2.7A_uno_w_protocol_freeze.md`
- 文档必须写清：
  - 当前协议不回退
  - 当前返回帧格式
  - 正式/联调 timeout 区分
  - 当前 `PI` 参数
  - 当前编码器映射
  - `D_MECH/D_EFF/WB/TW/K/CPR` 当前只算预配置，不是本轮联调失败判责依据

### 2. 树莓派侧：从 CMD 桥拆成 W 桥
- 树莓派端当前主问题不是发不出串口，而是发错了层级；本轮必须把主桥从 `CMD,vx,vy,wz` 改成 `W,w1,w2,w3,w4`。
- 树莓派侧拆成两层：
  - `wheel_bridge.py`
    - 当前主节点
    - 持久串口连接 `/dev/csr_uno`
    - 等待 `CSR_UNO_READY`
    - 以固定频率 `20Hz` 持续发送 `W,w1,w2,w3,w4`
    - 读取并记录 `ACK:W` / `ENC,...` / `PID,...` / `ERR:...`
  - `cmdvel_to_wheels.py`
    - 本轮只建骨架
    - 未来负责 `cmd_vel -> W`
    - 当前不默认启用，不作为本轮通过前提
- 第一阶段测试输入先人工固定：
  - 纯前进：`W,100,100,100,100`
  - 纯后退：`W,-100,-100,-100,-100`
  - 原地旋转：先试 `W,100,100,-100,-100`
  - 横移：先试 `W,100,-100,100,-100`
  - 符号以实测轮向校正，不在本轮文档里假装已经最终冻结
- 新树莓派发布文档：
  - `Raspberrypi/releases/C-2.2.7/C-2.2.7B_pi_w_bridge.md`
- 文档必须写清：
  - 当前不再发 `CMD,vx,vy,wz`
  - 当前主桥改发 `W,w1,w2,w3,w4`
  - `cmdvel_to_wheels.py` 只是第二阶段骨架
  - 当前验收依据是 `ACK:W + ENC + PID`

### 3. 双端共同真相：唯一契约文件与集成纪律
- 新增唯一协议契约：
  - `docs/SOFTWARE/C-2.2.7_serial_protocol_contract.md`
- 契约文件固定写死：
  - 上位机 -> UNO：`W,w1,w2,w3,w4`
  - UNO -> 上位机：
    - `CSR_UNO_READY`
    - `ACK:W`
    - `ERR:<reason>`
    - `ENC,t1,t2,t3,t4,v1,v2,v3,v4`
    - `PID,...`
  - `CMD,vx,vy,wz` 是旧树莓派桥协议，不是当前联调真相
  - `cmd_vel -> W` 是下一阶段
  - 当前几何参数只服务下一阶段，不作为本轮失败判责依据
- 新增联调前 checklist，建议直接放在同一契约文件末尾或单独附录：
  - UNO 端：分支名、commit、协议前缀、返回帧、timeout（正式/联调）、控制周期
  - Pi 端：分支名、commit、发送帧、发送频率、READY 规则、解析帧类型
- 集成纪律固定为：
  - 任何跨端联调都先合流到 `feature/c-2.2.7-integration-pi-uno-w-bridge`
  - 只有这个分支可以被称为“本轮联调真相”
  - UNO / Pi 子分支都向这个分支提 PR
  - PR 描述必须附：分支名、协议版本、checklist、日志样例
  - 联调通过后打 annotated tag：`c-2.2.7-w-bridge-pass`

## Public Interfaces
- 本轮正式冻结的唯一联调协议：
  - 输入：`W,w1,w2,w3,w4`
  - 输出：
    - `CSR_UNO_READY`
    - `ACK:W`
    - `ERR:<reason>`
    - `ENC,t1,t2,t3,t4,v1,v2,v3,v4`
    - `PID,target1,target2,target3,target4,meas1,meas2,meas3,meas4,out1,out2,out3,out4`
- `CMD,vx,vy,wz` 本轮明确降级为**树莓派旧协议**，不再作为当前联调接口。
- 长期接口语义继续承接 `C-2.2.4 / C-2.2.5A / C-2.2.6A`：
  - 最终仍是 `vx / vy / wz`
  - 但本轮只做 `W` 层打通，不提前冻结 `cmd_vel -> 轮速` 的最终换算实现

## Test Plan
- **UNO 侧通过标准**
  - 仍只接受 `W,w1,w2,w3,w4`
  - 稳定返回 `ACK:W`
  - 稳定输出 `ENC,...`
  - 新增 `PID,...`
  - 可区分正式 timeout 与联调 timeout
- **Pi 侧通过标准**
  - 能收到 `CSR_UNO_READY`
  - 能发送 `W,...`
  - 能收到 `ACK:W`
  - 能解析 `ENC,...`
  - 能解析 `PID,...`
  - 不再向 UNO 发 `CMD,...`
- **集成通过标准**
  - Pi 发 `W,...`
  - UNO 回 `ACK:W`
  - 底盘动作与 `ENC/PID` 同步变化
- **明确不算通过**
  - 只收到 `READY`
  - 只看到树莓派写串口成功
  - 只看到 UNO 本地打印
  - 只看到 `CMD,...` 自己闭环自洽

## Assumptions
- 当前仓库本地能确认的事实只有 UNO 侧：`SerialProtocol::parseWheelTargets()` 确实只认 `W`，`CsrBaseController` 当前只发 `ENC` 不发 `PID`，因此 `UNO` 侧 `C-2.2.7A` 需要补 `PID` 输出。
- 树莓派侧代码不在本地仓库，因此 `C-2.2.7B` 只能冻结为接口与发布要求，不在本地侧假装已实现。
- 当前物理参数 `D_MECH=80 / D_EFF=78 / WB=0.16535 / TW=0.17850 / K=0.171925 / CPR_X1_EST=260` 足够支撑下一阶段 `cmd_vel -> W`，但本轮不把它们当联调前置条件。
- 本地仍有 `c002_uno_baseline.ino` 的用户脏改动，本轮计划继续不碰它；`c002` 只作为动作码回退线存在。
