# C-1.2.0 计划：C↔J 全双工最小闭环验证轮

## 1. 版本定位

- 版本号：`C-1.2.0`
- 性质：`C` 主线牵引下的 `C↔J` 首次并线验证轮
- 目标：用 `F411` 优先搭出一个最小但真实的双向闭环：
  `UNO 默认流程运行 -> J 线发现颜色目标 -> F411 令 UNO 停车 -> J 抓取或 10 s 超时 -> F411 令 UNO 恢复`

## 2. 本轮冻结边界

- 只做软件侧最小闭环，不扩成完整系统整合
- `F411` 为主方案，`F103` 仅在 `F411 Gate0` 失败时作为回退
- `J` 侧本轮直接由 `OpenMV UART1(A9/A10)` 对接 `F411`
- `UNO` 保持当前 boot 默认 `MAN` 不变；闭环测试从“已切入默认运行流程”的状态开始
- 不接 `CAN / Wi-Fi / 上位机 / 卖家 APP / LCD / 图传`

## 3. 代码落点

- `sketches/c002_uno_baseline/`
  UNO 权威入口；本轮只在这里补外部暂停闸门
- `firmware/cj_bridge_f411_cubeide/`
  `F411` 桥接状态机与协议源码骨架
- `jixiebi/experiments/C-1.2.0_color_grasp_uart_poc/`
  `J` 线受控实验副本；不在 `J-jixiebi/` 卖家原目录上直接施工
- `docs/{SOFTWARE,HARDWARE,VERIFY}/`
  协议、接线、电平与 Gate 留痕
- `artifacts/C-1.2.0/{f411|f103}/`
  产物与阻塞说明目录

## 4. 本轮实现摘要

### 4.1 UNO 侧

- 新增 `bridgeSerial`：`A3=RX`、`A4=TX`、`9600 8N1`
- 新增 `externalPauseActive / externalPauseSourceJ`
- 新增 ASCII 行协议处理：`PING / STOP_REQ / RESUME_REQ`
- 收到 `STOP_REQ`：停车、拉起外部暂停闸门、回 `STOP_ACK`
- 收到 `RESUME_REQ`：释放外部暂停闸门、回 `RESUME_ACK`
- 通过 `CJ_COMM_POC_BUILD` 宏冻结本轮构建入口

### 4.2 F411 侧

- 桥接状态机固定为：
  `IDLE -> WAIT_UNO_STOP_ACK -> WAIT_J_RESULT -> WAIT_UNO_RESUME_ACK -> IDLE`
- `USART2(PA2/PA3)` 预留给 `UNO`
- `USART1(PA9/PA10)` 预留给 `J`
- 已落仓：环形缓冲区、帧协议、状态机、主循环集成桩
- 未完成：真实板级初始化、烧录与 `.elf/.hex` 产物

### 4.3 J 侧

- 基线例程：`COLOR_AUTOcatch_ov2640.py`
- 受控副本目录：`jixiebi/experiments/C-1.2.0_color_grasp_uart_poc/`
- 新增 `cj_link.py` 负责 UART 定界帧协议
- 颜色目标需连续 3 帧稳定成立后才上报 `COLOR_FOUND`
- 只有收到 `PICK_WINDOW` 才进入抓取执行段
- 终态固定回：`PICK_DONE / PICK_TIMEOUT / ARM_FAIL`

## 5. 验收标准

- `UNO` 只做补丁式改动，没有重构主流程
- `UNO` 能正确响应 `STOP_REQ / RESUME_REQ`
- `J` 侧已基于颜色抓取类例程建立受控实验副本
- `F411` 协议、状态机和接线口径已冻结
- 接线、电平、协议、测试 Gate 和已知阻塞均已留痕
- 本轮未混入 `Wi-Fi / CAN / 上位机 / 卖家 APP` 扩展开发

## 6. 当前已知阻塞

- 当前工作站未发现可用的 `STM32CubeIDE / arm-none-eabi-gcc` 工具链，因此 `F411 Gate0` 只能先落源码与验证清单，无法在本机产出 `.elf/.hex`
- `UNO` 与 `J(OpenMV)` 源码已落仓，但真实三板闭环仍需硬件上板和 10 轮重复测试
