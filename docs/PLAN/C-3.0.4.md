# C-3.0.4：OpenRF1 下位机迁移基线建立计划

## 2026-04-16 审批更新

当前基于 OpenRF1 的 `C-3.0.4` 施工，已补充出一组必须优先生效的实机真值：

- `A(CN1) = 左后`
- `B(CN2) = 左前`
- `C(CN3) = 右后`
- `D(CN4) = 右前`

本轮最新关键测试数据冻结为：

| 通道 | 轮位 | 接受正值 | 接受负值 |
| --- | --- | --- | --- |
| `CN1` | 左后 | 无响应 | 实际为负方向 |
| `CN2` | 左前 | 实际为正方向 | 无响应 |
| `CN3` | 右后 | 无响应 | 实际为正方向 |
| `CN4` | 右前 | 实际为负方向 | 无响应 |

基于这组实测真值，当前结论不是“协议已经打通”，而是：

- 四轮同向同转目标未达成
- 正负方向语义在四路上不一致
- 当前版本不具备可安全冻结为正式底盘基线的条件

因此本轮计划补充一条审批结论：

- 当前 `C-3.0.4` OpenRF1 协议化 bring-up 申请按“阶段性失败/建议弃项”提交项目负责人审批
- 现阶段保留文档和测试数据作为证据链，不再把当前实现包装为可继续扩写的稳定主线

## Summary
- 当前正式迁移判断已锁定：`OpenRF1 (STM32F103RCT6)` 取代 `UNO` 成为底盘下位机新主线；`UNO` 不再继续做主线闭环优化，只保留历史事实和回退证据。
- 本轮实施以 **Keil / vendor 例程 9** 作为首个 bring-up 权威入口，不先迁 `CubeIDE`，避免把“工程搬家”混入“底盘稳定闭环”。
- 仓库按 **公开安全** 策略入库：不把 vendor 原工程整包提交到公开仓库；公开仓库只跟踪自研应用层、接口契约、迁移说明和集成说明。vendor 源码继续留在 [`docs/STM32F103RCT6`](f:/Project/CleanScout_rover/docs/STM32F103RCT6) 作为本地参考依据。
- 当前唯一目标原本是：在电脑串口工具控制下，让 OpenRF1 板实现四轮“持续、平顺、可控、可回传”的最小闭环基线。
- 截至 2026-04-16，按上方真值表，该目标本轮未达成，当前文档转为保留迁移分析与失败证据。

## Key Changes
### 1. 仓库组织与迁移边界
- 新建 `docs/PLAN/C-3.0.4.md`，作为本轮正式任务书入库。
- 新建 `docs/SOFTWARE/C-3.0.4_openrf1_migration_analysis.md`，明确写清：
  - 为什么 `UNO` 到此止损
  - 为什么 `OpenRF1` 更适合当前底盘闭环
  - 为什么采用“底层复用 + 应用层重写”
  - 为什么本轮不接树莓派、不接 ROS、不接机械臂
- 新建 `docs/SOFTWARE/C-3.0.4_openrf1_runtime_baseline.md`，冻结当前运行时基线与失败边界。
- 当前审批输出补充到 `docs/VERIFY/C-3.0.4B_openrf1_bottom_driver_truth_freeze.md`。

### 2. 当前冻结的失败边界
- 当前仅冻结已验证的板口/轮位映射与正负响应真值。
- 不再把当前代码状态表述为“可安全扩写的协议版底盘基线”。
- 后续若重开 OpenRF1 线，必须从 `CN1~CN4` 单轮正负直驱真值重新起步，而不是从当前半完成协议版继续堆补丁。

## Public Interfaces
- 当前仅保留测试期接口事实，不再宣称其已通过：
  - 输入：`W,va,vb,vc,vd`
  - 预期输出：`CSR_RF1_READY / ACK:W / ERR / VEL / PWM`
- 但按本轮审批结论，这组接口尚未形成项目可冻结主线。

## Test Plan
- 保留 `CN1~CN4` 单轮正负测试真值作为唯一验收证据。
- 不再以“四轮一起转过一次”视为通过。
- 后续若重开，必须重新验收：
  - 四轮同向同转
  - 四轮正反双向可控
  - 协议收发稳定
  - 遥测与实际动作一致

## Assumptions
- 当前审批提交只覆盖本地 OpenRF1 bring-up 结果，不覆盖树莓派、ROS、机械臂或其他系统。
- 当前工作树里的 `_local/` 实验代码继续仅作本地证据，不作为本轮对外发布物。
