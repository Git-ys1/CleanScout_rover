# C-1.2.6：以本地 vendor `main.py` 接管 J 线视觉抓取基线

## Summary
- 先把当前本地脏树收敛成干净的 `C-1.2.5` 基线，再从 `feature/c-1.2.5-vendor-actuator-baseline` 新开 `feature/c-1.2.6-vendor-vision-grasp-baseline`，提交 `checkpoint: before C-1.2.6`。当前必须先保留并纳入的本地修正是：`TILT_CENTER_DEG = 85`、`TILT_MIN_DEG = 0`、`TILT_MAX_DEG = 90`。
- 新主线不再碰 `C-1.2.0_color_grasp_uart_poc` 和 `F411` 联调，直接在新目录 `jixiebi/experiments/C-1.2.6_vendor_vision_grasp_baseline/` 建立受控视觉抓取基线。
- 视觉抓取运行时以本地 `J-jixiebi/introduction and examples/main.py` 为主参考，`main.py~RFb594dd.TMP` 只作为黄色抓取备份参考，`main.py~RFb291ce.TMP` 只作为摄像头/LCD 自检备份。`README.txt` 和 `selftest.py` 当前为空文件，只做 snapshot 留痕，不参与运行。
- `pid.py` 统一使用仓内当前已实测良好的版本；经本地校验，它与 vendor `pid.py` 哈希一致，所以此项不会引入行为分叉。
- 本轮默认先做单目标黄色抓取，再在同一套状态机上打开三色分拣；`F411` 联调线在文档中明确暂停，不在本轮扩写。

  - `vendor_camera_lcd_selftest_snapshot.py`
  - `main.py`

  - `vision.py`

- 运行时代码不直接 import snapshot 文件，snapshot 只做证据链和差异对照。

### 2. 执行层完全接管到新 `actuator.py`
- 新 `actuator.py` 以 `C-1.2.5` 冻结边界为唯一真值：
  - `PAN_CENTER_DEG = 0`
  - `PAN_MIN_DEG = -90`
  - `PAN_MAX_DEG = 90`
  - `TILT_CENTER_DEG = 85`
  - `TILT_MIN_DEG = 0`
  - `TILT_MAX_DEG = 90`
  - `CLAW_OPEN_ANGLE = -60`
  - `CLAW_CLOSE_ANGLE = 40`
- 提供并强制全局只走这些接口：
  - `pan_set(angle)`
  - `tilt_set(angle)`
  - `claw_open()`
  - `claw_close()`
  - `center_pose()`
  - `lift_pose()`
  - `drop_pose_for(color_id)`
  - `clamp_pan(angle)`
  - `clamp_tilt(angle)`
- 所有视觉抓取逻辑禁止再直接写：
  - `Servo(3).angle(...)`
  - `Servo(4).angle(...)`
  - `Servo(1).angle(...)`
- 默认搜索姿态冻结为：
  - `pan = 0`
  - `tilt = 85`
  - `claw = open`
- 投放位默认值直接冻结为 vendor 语义映射后的初值，并全部经过边界钳制：
  - `RED_DROP_PAN = 45`
  - `YELLOW_DROP_PAN = 0`
  - `BLUE_DROP_PAN = -45`
  - `DROP_TILT_DEG = 60`
  - `LIFT_TILT_DEG = 55`
  - 若实机验证需调整，只允许改 `actuator.py` 常量并同步写入 `test_record.md`

### 3. 视觉抓取层拆到 `vision.py`，主入口只保留状态机
- 新 `vision.py` 负责：
  - 颜色阈值配置
  - `find_target(img, target_mode)`
  - `find_max_blob(blobs)`
  - `estimate_distance_proxy(blob_width)`
  - `target_center_error(blob, img_width, img_height, claw_offset)`
- 主运行结构固定在新 `main.py`，状态机冻结为：
  - `SCAN`
  - `TRACK`
  - `APPROACH_READY`
  - `GRAB`
  - `DROP`
  - `RESET`
- 默认参数冻结：
  - `TARGET_MODE = "yellow"` 作为 Gate4 单目标入口
  - `ENABLE_COLOR_SORTING = False`；Gate5 通过后再切 `True`
  - 黄色优先使用 vendor 主 `main.py` 阈值：`(43, 85, -31, 55, 27, 84)`
  - 红/蓝阈值也保留 vendor 主 `main.py` 值，供 Gate5 使用
  - `PIXELS_THRESHOLD = 500`
  - 距离代理使用主 `main.py` 公式：`distance_proxy = 13500 / (blob.w * 2)`
  - 抓取门槛默认：`90 <= distance_proxy <= 110`
  - 稳定进入抓取默认：连续 `5` 帧满足中心偏差与距离门槛
- `TRACK` 阶段的 PID 只服务云台：
  - `pan_output = pan_pid.get_pid(pan_error, 1) / 2`
  - `tilt_output = tilt_pid.get_pid(tilt_error, 1)`
  - 所有输出都必须经过 `clamp_pan()` / `clamp_tilt()` 后再落到舵机
- 单目标阶段只跑黄色链路：
  - `SCAN -> TRACK -> APPROACH_READY -> GRAB -> DROP -> RESET`
- 三色阶段在同一状态机上打开：
  - 目标色只影响 `drop_pose_for(color_id)`，不复制第二套抓取代码路径

### 4. 旧 vendor 延时写法先规范，再谈上板
- 本地 vendor `main.py` 里已确认存在 `time.sleep(2000)`、`time.sleep(1000)` 等明显把毫秒误写成秒的写法；这些必须在接管版中统一整改。
- 冻结规则：
  - 机械动作延时全部改为 `time.sleep_ms(...)`
  - 只有真正需要整秒停顿时，才允许 `time.sleep(1)` / `time.sleep(2)`
- 默认动作时序冻结：
  - `CLAW_SETTLE_MS = 500`
  - `LIFT_SETTLE_MS = 800`
  - `DROP_SETTLE_MS = 800`
  - `RESET_SETTLE_MS = 800`
- `C-1.2.6_motion_mapping_notes.md` 必须明确记录：
  - 哪些 vendor `sleep` 被判定为错误
  - 分别改成了什么 `sleep_ms(...)`
  - 这些修正对动作链的直接影响

## Public Interfaces
- 新视觉抓取基线只暴露两层接口：
  - `actuator.py`：执行器唯一入口
  - `main.py` 状态机：`SCAN / TRACK / APPROACH_READY / GRAB / DROP / RESET`
- `F411`/`UNO`/`C↔J` 通信接口本轮不变，但也不参与运行；文档中明确写为“暂停，不删除”。
- 视觉运行时对外冻结的行为接口：
  - 默认搜索位：`pan=0, tilt=85, claw=open`
  - 单目标模式：`TARGET_MODE = "yellow"`
  - 三色模式：`ENABLE_COLOR_SORTING = True`
  - 投放位统一经 `drop_pose_for(color_id)` 下发，不允许在视觉代码里直接写 pan/tilt 角度。
