任务：整理并开发 STM32 / OPENRF1 机械臂专用下位机固件，作为 Orange Pi YOLO 视觉追踪与总线舵机机械臂之间的安全网关。当前只处理机械臂，不接底盘，不接 ROS，不接清扫功能。

一、当前硬件与通路

当前硬件通路已经具备：

1. PC 通过 ST-Link 连接 STM32 / OPENRF1，可烧录与调试。
2. PC 通过 USB 调试串口连接下位机，可看日志和手动发命令。
3. Orange Pi 5 Max 通过 CH340 连接下位机，可作为视觉上位机发送机械臂控制命令。
4. Orange Pi 已完成 YOLO11 RKNN USB 摄像头实时识别，后续会把目标中心点转换为机械臂追踪命令。
5. 当前阶段先固定摄像头到机械臂上，完成机械臂视觉追踪，再考虑和小车底盘融合。

二、必须先学习的资料与目录

本地仓库：

```text
CleanScout_rover
```

重点看最新主线内容：

```text
docs/VERIFY/C-5.0.2_arm_bus_servo_protocol_note.md
firmware/mechanical_arm_official_baseline/
firmware/mechanical_arm_controller/
docs/001-总线舵机资料/
```

同时参考用户提供的：

```text
附件1《总线舵机指令表》.docx
```

重点结论：

1. 总线舵机使用 ASCII 文本协议。
2. 单舵机控制格式：

```text
#000P1500T1000!
```

3. 多舵机同步格式：

```text
{#000P1500T1000!#001P1500T1000!}
```

4. ID 范围：

```text
000-254
```

5. 广播 ID：

```text
255
```

6. P 字段范围：

```text
0500-2500
```

7. T 字段范围：

```text
0000-9999 ms
```

8. 默认波特率：

```text
115200
```

9. 危险命令包括但不限于：

```text
PID 改 ID
PBD 改波特率
PCLE 恢复出厂
PSMI / PSMX 设置最小最大值
PSCK 矫正中值
PCSD 设置启动位置
PCSM 设置开机模式
```

这些命令默认不允许 Orange Pi 直接透传，除非进入明确的维护模式。

三、总体开发目标

建立一版机械臂专用下位机固件：

```text
Orange Pi CH340 串口
↓
STM32 / OPENRF1 机械臂网关固件
↓
命令校验、限位、安全保护、日志
↓
总线舵机 UART
↓
机械臂总线舵机
```

当前只做机械臂，不保留底盘电机、编码器、清扫、风机、继电器、IMU、雷达等逻辑。

四、重要原则

1. 不在底盘正式固件上直接乱改。
2. 从机械臂官方基线或 `firmware/mechanical_arm_controller/` 新建独立工程。
3. 先建立可编译、可烧录、可回滚的最小工程。
4. 先做串口回环和舵机 ID 读取，再做运动。
5. 先单舵机，再多舵机。
6. 先手动命令，再 Orange Pi 自动控制。
7. 先限位和停止机制，再闭环追踪。
8. 所有危险命令默认禁用。
9. 不要把 ROS、底盘、导航相关代码带进机械臂下位机。

六、串口职责划分

必须先确认实际串口资源，然后在报告里写清楚。

建议逻辑命名：

```text
HOST_UART：连接 Orange Pi / CH340，上位机命令入口
DEBUG_UART：连接 PC 调试串口，可打印日志
SERVO_UART：连接总线舵机链路
```

如果硬件上 HOST_UART 和 DEBUG_UART 是同一个串口，则必须采用可区分日志和命令的格式，避免日志污染 Orange Pi 通信。

必须确认：

1. Orange Pi CH340 接到 STM32 哪个 USART。
2. PC USB 调试串口接到 STM32 哪个 USART。
3. 总线舵机接到 STM32 哪个 USART。
4. 每个 USART 的波特率。
5. 每个 USART 的引脚。
6. 是否存在半双工方向控制。
7. 是否需要转接板或电平转换。
8. 舵机供电和 STM32 / CH340 是否共地。

七、第一阶段：只读审计

先不要改代码，先输出：

```text
firmware/mechanical_arm_controller/docs/00_CODE_AUDIT.md
```

报告必须回答：

1. 当前机械臂官方基线入口文件是什么。
2. 当前是否已经有 `yh_pwm_text` 或类似文本命令封装。
3. 当前是否已经有 `parse_action()` 或类似命令解析函数。
4. 当前是否已经有 UART3 / 总线舵机串口初始化。
5. 当前是否把 `#000P1500T1000!` 转发给总线舵机。
6. 当前是否也会解析文本命令并控制本地 PWM。
7. 当前哪些模块是机械臂必须保留的。
8. 当前哪些模块属于旧实验、底盘或无关功能，可以先剔除。
9. 当前工程是否能直接编译。
10. 当前工程烧录后是否有启动日志。
11. 当前串口命令入口是哪一个。
12. 当前总线舵机输出口是哪一个。
13. 当前是否有急停、停止当前位置、释放扭力、恢复扭力等命令。
14. 当前是否有危险命令防护。

八、第二阶段：建立最小可运行固件

目标：下位机烧录后，只做三件事。

1. 启动日志：

```text
ARM_GATEWAY_BOOT
version=C-5.1.0-arm-gateway
host_uart=...
servo_uart=...
baud=115200
safe_mode=on
```

2. Host 串口命令回显：

```text
PING
```

返回：

```text
PONG
```

3. 总线舵机基础透传测试：

Host 输入：

```text
RAW #000PVER!
```

STM32 向总线舵机发送：

```text
#000PVER!
```

并把返回转发给 Host。

如果没有接舵机，也必须返回超时信息：

```text
ERR TIMEOUT
```

九、第三阶段：实现安全协议层

不要让 Orange Pi 直接无限制透传所有命令。

Host 到 STM32 的建议命令格式：

```text
PING
HELP
STATUS
STOP
TORQUE_OFF
TORQUE_ON
SERVO_READ_ID <id>
SERVO_READ_POS <id>
SERVO_READ_TEMP_VOLT <id>
SERVO_MOVE <id> <pwm> <time_ms>
SERVO_GROUP_MOVE <id:pwm,time> <id:pwm,time> ...
RAW_SAFE <raw_frame>
```

示例：

```text
SERVO_MOVE 0 1500 1000
SERVO_MOVE 3 1800 300
SERVO_GROUP_MOVE 0:1500,300 1:1600,300 2:1400,300
```

STM32 再转换成：

```text
#000P1500T1000!
{#000P1500T0300!#001P1600T0300!#002P1400T0300!}
```

返回：

```text
OK
ERR BAD_ID
ERR BAD_PWM
ERR BAD_TIME
ERR BUS_TIMEOUT
ERR UNSAFE_COMMAND
```

十、字段校验要求

必须实现：

```text
id: 0-254
broadcast_id: 255 仅允许白名单读命令或全扭力控制，不允许默认运动
pwm: 500-2500
time_ms: 0-9999
frame_len: 限制最大长度
group_count: 限制最大舵机数
control_rate: 限制上位机命令频率
```

危险命令默认禁止：

```text
PID
PBD
PCLE
PCLE0
PSMI
PSMX
PSCK
PCSD
PCSM
PMOD
```

维护模式才允许部分配置命令。维护模式必须有明确解锁指令，并且上电默认关闭。

十一、机械臂关节映射

建立文档：

```text
firmware/mechanical_arm_controller/docs/SERVO_ID_MAP.md
```

表格至少包括：

```text
关节名
舵机 ID
机械含义
中位 P 值
安全最小 P 值
安全最大 P 值
方向
备注
```

初始未知时，不要猜。先写：

```text
UNKNOWN
```

然后通过单舵机测试逐步确认。

十二、舵机 ID 与单舵机测试流程

严禁多个未确认 ID 的舵机同时执行改 ID 命令。

推荐测试顺序：

1. 只接一个舵机。
2. 读取版本：

```text
#255PVER!
```

3. 读取 ID：

```text
#255PID!
```

4. 读取位置：

```text
#255PRAD!
```

5. 如果需要改 ID，必须只接一个舵机。
6. 改 ID 后立即贴标签。
7. 每个舵机建立 ID 与机械关节对应表。
8. 所有 ID 确认后，再接入整条总线。

十三、急停与保护

必须实现并测试：

```text
STOP
```

对应下发：

```text
#255PDST!
```

或者对已知 ID 逐个下发：

```text
#000PDST!
#001PDST!
...
```

必须实现：

```text
TORQUE_OFF
TORQUE_ON
```

对应：

```text
PULK
PULR
```

注意：释放扭力后机械臂可能因重力下坠，必须在报告中说明风险。

十四、和 Orange Pi 的接口边界

Orange Pi 后续只需要发高层命令，例如：

```text
SERVO_GROUP_MOVE 0:1510,80 3:1490,80
```

或者：

```text
SERVO_MOVE 0 1510 80
```

不建议第一版让 Orange Pi 直接发：

```text
#000P1510T0080!
```

原因：STM32 必须承担限位、频率限制、危险命令过滤和急停保护。

十五、与视觉追踪的关系

当前 Orange Pi 侧 `yolo11_camera.py` 已经能输出目标框。

后续视觉追踪脚本应把目标中心误差转换为：

```text
yaw_pwm
pitch_pwm
time_ms
```

然后通过串口发给 STM32：

```text
SERVO_GROUP_MOVE <yaw_id>:<yaw_pwm>,80 <pitch_id>:<pitch_pwm>,80
```

STM32 再生成总线舵机文本帧。

当前阶段 STM32 不做视觉、不做 PID。PID 放在 Orange Pi 侧更方便调试。

十六、不要删除但可以隔离的代码

如果当前下位机代码很乱，不要直接物理删除大量文件。采用以下策略：

1. 新建机械臂专用工程或编译目标。
2. 不参与编译的旧文件移到 `legacy/` 或从 Keil 工程中移除。
3. 保留原始官方基线。
4. 每一步能编译后再继续。
5. 不要同时重构多个模块。

可以从编译目标剔除：

```text
底盘电机
编码器
PID 闭环底盘控制
IMU
雷达
清扫风机
继电器
ROS 相关
OpenMV 旧实验逻辑
无关传感器
```

必须保留或重写：

```text
系统时钟
delay / systick
USART
GPIO
总线舵机协议
安全限位
调试日志
启动入口
错误处理
```

十七、PC 侧调试工具

如果仓库已有 Windows PowerShell 串口工具，优先复用。

建议新增或整理：

```text
tools/arm_gateway_serial_probe.ps1
tools/arm_gateway_serial_probe.py
```

功能：

1. 自动列串口。
2. 打开指定 COM。
3. 发送 PING。
4. 发送 STATUS。
5. 发送 SERVO_READ_ID。
6. 发送 SERVO_MOVE 小幅测试。
7. 打印返回。
8. 支持日志保存。

十八、烧录与验证流程

每次修改后按这个顺序：

1. Keil 编译通过。
2. ST-Link 烧录通过。
3. PC 调试串口看到启动日志。
4. PING / PONG 通过。
5. STATUS 通过。
6. 不接舵机时 RAW 读命令能返回 timeout。
7. 接单舵机，读取版本通过。
8. 读取 ID 通过。
9. 小幅移动通过。
10. STOP 通过。
11. 多舵机同步移动通过。
12. Orange Pi 通过 CH340 发送命令通过。

十九、最终验收标准

必须完成：

1. 新机械臂专用固件可独立编译。
2. ST-Link 可烧录。
3. 启动日志清晰。
4. PC 调试串口可交互。
5. Orange Pi CH340 可发送命令。
6. 能读取舵机 ID / 位置 / 版本。
7. 能控制单舵机小幅运动。
8. 能控制多个舵机同步小幅运动。
9. STOP 有效。
10. 危险命令默认被拦截。
11. 舵机 ID 与关节映射文档建立。
12. 不影响底盘正式固件。
13. 不影响 Orange Pi YOLO 视觉脚本。
14. 不接 ROS。
