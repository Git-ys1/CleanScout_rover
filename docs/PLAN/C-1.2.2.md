# C-1.2.2：J 线抓取解锁轮

## 1. 本轮目标

本轮只解决一个死结：

- `OpenMV` 识别到颜色后不再永远卡在 `WAIT_PICK_WINDOW`
- 无论 `PICK_WINDOW` 是否及时下发，`J` 线都必须能进入一次真实抓取流程
- 抓取结果仍然必须通过当前 `C↔J` 链路回传给 `F411`

本轮不是废弃通信主线，而是引入“优先门控、超时回退”的 bring-up 策略。

## 2. 本轮边界

### 2.1 允许修改

- `firmware/cj_bridge_f411_cubeide/Core/Src/cj_bridge.c`
- `firmware/cj_bridge_f411_cubeide/Core/Inc/cj_bridge.h`
- `firmware/cj_bridge_f411_cubeide/Core/Src/main.c` 的最小诊断映射
- `jixiebi/experiments/C-1.2.0_color_grasp_uart_poc/main.py`
- 文档与测试记录

### 2.2 禁止修改

- 不改 `UNO` 主流程
- 不改 `STOP_REQ / RESUME_REQ` 接口名
- 不把协议端点从 `OpenMV` 切到 `J-STM32`
- 不引入 `CAN / Wi-Fi / 上位机主控`
- 不改卖家原始基线目录
- 不把 `WAIT_PICK_WINDOW` 整体删除
- 不改成“看见颜色永远立即抓”

## 3. 本轮冻结策略

### 3.1 J 线

- 仍先上报 `COLOR_FOUND`
- 仍优先等待 `PICK_WINDOW`
- 若在 `LOCAL_PICK_FALLBACK_DELAY_MS` 内未收到 `PICK_WINDOW`，且目标仍在视野中，则本地进入抓取
- 本地抓取后仍必须回发：
  - `PICK_DONE`
  - `PICK_TIMEOUT`
  - `ARM_FAIL`

### 3.2 F411

- 保留主优先路径：`COLOR_FOUND -> STOP_REQ -> STOP_ACK -> PICK_WINDOW`
- 新增过渡性解锁：
  - 在 `WAIT_UNO_STOP_ACK` 期间，也接收 `PICK_DONE / PICK_TIMEOUT / ARM_FAIL`
  - 不再把“早到结果”直接丢掉
  - 一旦接住终态结果，立即发 `RESUME_REQ` 给 `UNO`

## 4. 验收重点

1. `OpenMV` 识别后能从 `WAIT_PICK_WINDOW` 进入 `LOCAL_PICK_FALLBACK / PICKING`
2. 本地回退抓取后，结果帧仍然发回链路
3. `F411` 不再丢弃 `WAIT_UNO_STOP_ACK` 阶段的 J 线终态结果
4. 本轮不修改 `UNO`

## 5. 过渡性策略说明

本轮新增的“本地抓取回退”是 bring-up 过渡策略，不是最终系统形态。

最终目标仍保持不变：

`COLOR_FOUND -> STOP_REQ -> STOP_ACK -> PICK_WINDOW -> PICK_DONE -> RESUME_REQ`

本轮只是为了拆开当前协议联锁，让：

- 识别链路先继续往前走
- 抓取能力不再被状态门锁死
- 结果链路继续留在现有 `C↔J` 通讯体系内
