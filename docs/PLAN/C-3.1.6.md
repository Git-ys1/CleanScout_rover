# C-3.1.6：RF1 稳定基线补充回灌到 `main`

## Summary
- 当前可核到的最新 RF1 稳定分支就是 `feature/c-3.1.4a-openrf1-timer-good-encoder-retest`；本地和远端都没有 `C-3.1.5+` 的 RF1 分支，因此本轮以它作为唯一回灌源。
- `feature/c-3.1.3-openrf1-tracking-port-soft-encoder` 继续保留为问题定位/止损证据分支，不作为正式主线继续开发。
- 本轮**不要直接把 `feature/c-3.1.4a...` 整分支 merge 到 `main`**。原因是相对 `main`，该分支历史里还带着大量非 RF1 文档演进；直接 merge 会违反“只发布自己负责路径”的新纪律。
- 正式做法是：从 `main` 新开一个 **RF1 main-merge 集成分支**，只从 `feature/c-3.1.4a...` 拉取 RF1 白名单路径，然后再用 `--no-ff` 合并回 `main`。

## Key Changes
- 分支与工作区策略
  - 保留现有自测分支：`feature/c-3.1.4a-openrf1-timer-good-encoder-retest`
  - 新建集成分支：`feature/c-3.1.6-openrf1-main-merge`
  - 推荐在**单独 git worktree** 或干净工作副本里执行 `main` 合并流程，避免当前仓库根目录那些无关脏文件干扰 RF1 合并。
  - 先推送自测分支，再从 `main` 拉最新，再创建 `feature/c-3.1.6-openrf1-main-merge`。

- 只允许进入 `main` 的 RF1 路径
  - 代码：
    - `_local/openrf1_keil_work_3/`
    - `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/BH-STM32.uvprojx`
  - 文档：
    - `docs/PLAN/` 中当前已跟踪的 RF1 计划文档：`C-3.0.4`、`C-3.0.5`、`C-3.0.5B`、`C-3.0.5C`、`C-3.0.6`、`C-3.0.7`、`C-3.0.8`、`C-3.1.3`
    - `docs/SOFTWARE/` 中当前已跟踪的 RF1 文档：`C-3.0.4_openrf1_runtime_baseline.md`、`C-3.0.5_openrf1_min_protocol.md`、`C-3.0.6_pid_close_loop_and_protocol_update.md`
    - `docs/VERIFY/` 中当前已跟踪的 RF1 验证文档：`C-3.0.4B`、`C-3.0.5`、`C-3.0.6_*`、`C-3.0.7`、`C-3.0.8`、`C-3.1.0_*`、`C-3.1.3`、`C-3.1.4A`
  - 明确排除：
    - `Raspberrypi/`
    - `vue3/`
    - UNO 相关目录
    - 当前未跟踪草稿文档，例如 `docs/SOFTWARE/C-3.0.4_openrf1_migration_analysis.md`、`docs/PLAN/C-3.1.0.md`、`docs/PLAN/C-3.1.4A.md`；这些不属于“已冻结稳定基线”，本轮不带入 `main`

- 实施流程
  - 在 `feature/c-3.1.4a...` 上确认没有额外待补的 RF1 提交后，推送该分支。
  - 切到干净的 `main` 工作副本，拉最新 `main`。
  - 新建 `feature/c-3.1.6-openrf1-main-merge`。
  - 先打空提交：
    - `checkpoint: before C-3.1.6 main merge`
  - 仅从 `feature/c-3.1.4a...` 恢复白名单 RF1 路径到当前集成分支，不恢复其他历史文档或目录。
  - 提交内容变更：
    - `C-3.1.6: merge RF1 latest stable baseline into main`
  - 在 `main` 上执行：
    - `git merge --no-ff feature/c-3.1.6-openrf1-main-merge`
  - merge commit 也使用中文信息，明确它是“补充合并”，不是覆盖主线。

- 文档纪律
  - 本轮至少新增或更新一份 `docs/VERIFY/` 或 `docs/SOFTWARE/` 里的 `C-3.1.6` 归档文档，内容固定写清：
    - 本轮从哪个 RF1 分支回灌到 `main`
    - 回灌了哪些路径
    - 为什么没有直接 merge `feature/c-3.1.4a...`
    - 为什么 `feature/c-3.1.3...` 只保留为证据
    - `main` 现在形成的 RF1 统一基线是什么
  - 不允许只做 git 合并不留结论。

## Test Plan
- 合并前检查
  - `feature/c-3.1.4a-openrf1-timer-good-encoder-retest` 已推到远端。
  - `main` 工作副本干净，或使用单独 worktree。
  - `git diff main..feature/c-3.1.4a...` 的实际导入范围仅限 RF1 白名单路径。
- 集成分支验证
  - `git status` 只出现 RF1 白名单路径。
  - `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/BH-STM32.uvprojx` 仍指向 `_local/openrf1_keil_work_3` 的定时器主线版本，不再把 `csr_soft_encoder.c` 编进镜像。
  - Keil 在集成分支上编译通过。
- 主线验收
  - `main` 上通过 `--no-ff` 得到独立 merge 节点。
  - `main` 最终不包含 `Raspberrypi/`、`vue3/`、UNO 目录改动。
  - RF1 当前统一交接基线可追溯到：
    - `feature/c-3.1.4a-openrf1-timer-good-encoder-retest`
    - `feature/c-3.1.3-openrf1-tracking-port-soft-encoder` 证据链
  - 文档能说明“当前 RF1 主线是原生定时器基线，不是软编码器止损基线”。

## Assumptions
- 当前仓库里没有比 `feature/c-3.1.4a-openrf1-timer-good-encoder-retest` 更新且稳定的 RF1 分支；若执行前出现新的 `C-3.1.5+` 稳定 RF1 分支，则把回灌源替换为最新稳定分支，其余流程不变。
- 当前仓库根目录存在大量无关脏文件，因此默认采用**单独 worktree/干净副本**完成 `main` 回灌，避免污染现有开发树。
- 本轮的核心是“把 RF1 最新稳定基线干净补进 `main`”，不是继续推进 RF1 新功能；任何新增实现都不在 `C-3.1.6` 范围内。
