# C-1.1.6 计划：统一系统口径 + J 线资料评估轮

## Summary
- 本轮只做文档，不做代码、不导入 `J-jixiebi/` 原始资料包、不做 FSD 本地分析。
- 目标是把 J 线卖家资料从“能跑哪些例程”整理成“能否作为实验室清扫巡检系统操作子系统并入主线”的判断材料。
- 本轮固定结论口径：
  - `C` 是当前核心地面终端
  - `J` 是近场操作与局部处置候选子系统
  - `J` 当前更像“半开放二开平台，带明显卖家固件/APP 依赖”，不是纯黑盒，也不是已成熟可直接并线模块

## Key Changes
- 新建 `feature/c-1.1.6-j-line-assessment`，先提交 `checkpoint: before C-1.1.6`。
- 新增 4 份文档并更新 `README.md`：
  - `docs/PLAN/C-1.1.6.md`
  - `docs/SYSTEM/C-1.1.6_lab_system_positioning.md`
  - `docs/JLINE/C-1.1.6_J_line_material_inventory.md`
  - `docs/JLINE/C-1.1.6_J_line_system_assessment.md`
- `README.md` 改为系统级入口：
  - 顶部统一口径改为“实验室清扫巡检系统”
  - 保留 `CleanScout_rover` 作为现有仓名/工程代号
  - 增加系统定位文档和 J 线评估文档入口
  - 明确写出：`J-jixiebi/` 原始资料未入库，仅作为本轮本地分析来源

- `C-1.1.6_lab_system_positioning.md` 固化总口径：
  - 解释为什么当前项目不是“会吸尘的小车 / 独立机械臂 / 独立飞行器”，而是实验室运维系统
  - 明确三线角色和当前成熟度
  - 强调以后各线输出都必须回答：系统角色、解决的主线问题、与 C 线协同方式、当前成熟度、并线缺口

- `C-1.1.6_J_line_material_inventory.md` 只做高信号归纳，不抄文件树：
  - 资料类别固定分为：手册/调试、安装接线、协议通信、APP/UI、机械臂例程、视觉/AI 扩展、固件/板级文件
  - 机械臂例程类型固定分为：追踪抓取、颜色分拣、手势控制、串口控车协同
  - 单列“关键证据源”章节，只列本地来源路径和文件名，不做仓库链接
  - 单列“资料冲突/不稳定项”章节，至少记录：
    - 固件推荐 6.1.0，但部分 UI/案例要求 6.5.5
    - 卖家板级/引脚定义存在专用口径
    - 蓝牙/App 语义明显绑定卖家演示流程
    - 机械爪/LCD/WiFi/串口等存在引脚复用冲突风险

- `C-1.1.6_J_line_system_assessment.md` 必须按系统视角下结论：
  - 平台判断：半开放二开平台，接近可接入子系统的平台，但仍带卖家生态依赖
  - 当前已证明的能力：
    - 视觉追踪抓取
    - 红黄蓝目标分拣
    - 手势触发夹爪动作
    - OpenMV 视觉识别与舵机控制
    - UART 控外部小车的协同雏形
  - 对主线有价值的能力：
    - 小件拾取
    - 遗落物清理
    - 拨动/试探性局部操作
    - 视觉引导近场处置
  - 展示价值大于主线价值的内容：
    - 手势演示
    - 卖家 APP 模式切换流程
    - 泛化云台/YOLO 展示案例
  - 最值得保留的二开方向：
    - 视觉引导抓取链路
    - 颜色/目标分拣
    - 与 C 线的串口/任务协同接口
  - 最大风险固定写清：
    - 缺少 `C <-> J` 系统级任务/状态协议
    - 卖家固件版本与板级定义绑定
    - APP/蓝牙协议语义偏演示，不是主线任务接口
    - 通信、LCD、WiFi、机械爪等存在 IO 资源冲突
    - 视觉稳健性依赖阈值和场景调参
  - 给出并线前必补项，固定为：
    - 任务接口
    - 状态回传
    - 动作原语抽象
    - 最小硬件形态冻结
    - 稳定性验证

- 文档引用规则固定：
  - 由于 `J-jixiebi/` 不入库，文档中的证据使用纯文本路径，例如 `本地来源：J-jixiebi/.../claw_catch.py`
  - 不使用仓库内可点击链接指向这些未入库文件
  - 关键结论至少绑定一条本地来源
