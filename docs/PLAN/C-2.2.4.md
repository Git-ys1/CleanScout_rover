# C-2.2.4：上位-下位控制真相文档冻结轮

## Summary
- 本轮只做文档冻结，不改 `UNO`、`Tyler_1`、树莓派脚本或任何协议实现。
- 新分支建议固定为 `feature/c-2.2.4-upper-lower-control-truth`。从今往后分支名继续保留现有英文风格，但**提交消息改为中文**；`checkpoint:` 前缀保留，后半句改中文。
- 当前本地已发现一个未提交实测差异：`sketches/c002_uno_baseline/c002_uno_baseline.ino` 中 `COMMAND_TIMEOUT_MS` 已被改为 `4000UL`。`C-2.2.4` 只把它作为“联调实测证据”写入文档，**不把该代码差异纳入本轮提交**。
- 文档冻结的核心真相只有两层：
  - 联调临时态：允许单次命令保持较长时间，便于人工观察。
  - 正式控制态：必须是“树莓派持续发送速度命令 + UNO 维护最近一帧 + 短超时急停 + 下一轮接编码器闭环与 odom 回传”。

## Key Changes
### 1. 新增两份正式文档
- 新增 [docs/SOFTWARE/C-2.2.4_upper_lower_control_freeze.md](/abs/path/to/be/created)  
  内容固定按 4 段写：
  - **A. 当前已验证事实**
    - 树莓派侧可看到 `CSR_UNO_READY`
    - 发送 `5/8/X` 可收到 `ACK/ERR`
    - 当前链路已打通到“命令与回显”
    - `UNO` 串口打开即可能自动复位，`READY` 是可信握手信号
  - **B. 当前联调模式**
    - 当前脚本是“一次打开串口 -> 发一个字符 -> 结束”
    - 因此 `400ms` 只适合持续刷命令，不适合人工单发测试
    - 当前把命令保持拉长到 `2s~5s` 是联调友好策略，不是最终导航参数
    - 固定等待 `2 秒` 不是长期方案，后续改为等待 `CSR_UNO_READY`
  - **C. 正式控制模式冻结**
    - 上位机持续发送速度命令
    - 下位机始终执行最近一帧有效命令
    - 下位机短超时急停
    - 多来源控制未来统一进入 `Twist` 总线 / `twist_mux`
    - 长期正式接口语义是速度，不是 `1~9` 数字动作码
    - 对麦轮底盘，长期速度语义保留 `linear.x / linear.y / angular.z`
  - **D. 下一轮前置条件**
    - 若要做成熟 `PID / odom / 巡航 / SLAM`，必须接入霍尔编码电机
    - 当前无编码器阶段只能定义为开环/半开环联调，不是正式导航底盘
- 新增 [docs/SOFTWARE/C-2.2.4_serial_link_notes.md](/abs/path/to/be/created)  
  内容固定写：
  - `UNO` 打开串口会因 DTR 自动复位
  - `CSR_UNO_READY` 作为握手机制
  - 当前设备名 / 固定别名策略
  - 当前波特率 `115200`
  - 当前协议只是联调协议，不是长期正式底盘协议
  - 当前实测 `4000ms` 超时只说明“单发脚本需要更长观察窗”，不代表正式控制口径

### 3. 文档必须冻结的“长期真相”
- 当前 `UNO + AFMotor` 只承担底层执行，不是成熟底盘控制器。
- 长期正式架构固定为：
  - 树莓派持续发送速度设定值
  - `UNO` 保存最近一帧命令并执行
  - 短超时急停
  - 下一轮接霍尔编码电机后，`UNO` 再做轮速闭环与编码器/里程计回传
- 文档必须明确区分：
  - **联调临时态**：允许单发命令保持数秒，方便看现象
  - **正式控制态**：`10Hz~20Hz` 持续发送速度命令，`0.3s~0.5s` 超时停车
- 文档必须写明：
  - `1~9` 动作码只作为当前联调过渡协议
  - 最终接口不冻结为动作码，而冻结为速度语义帧
  - 下位机未来回传最少包含 `ACK / 最近命令 / 模式 / 错误码`，接编码器后再扩展编码计数、轮速、odom 基础量
