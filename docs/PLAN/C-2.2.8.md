# C-2.2.8：M3 / M4 编码器诊断与 Port B 原始探测轮

## Summary
- 当前主故障不再是协议错位，`UNO` 侧 `W,w1,w2,w3,w4`、`ACK:W`、`ENC,...`、`PID,...` 已经是当前真实基线；树莓派端本轮只做观测配合，不改协议。
- 仓库当前真实状态有两点必须先写死：
  - `PID,...` 实际代码里已经补上，项目负责人上一版“代码还没发 PID”已过时。
  - 你本地工作树里 `c003` 已手改 `ENC_SIGN_W3 = -1`，且你已实测 M3 明显变正常；这应作为 `C-2.2.8A` 的起始事实，而不是重新回到 `ENC_SIGN_W3 = 1` 再猜。
- 本轮唯一目标是把问题拆开定位：
  - M3：先按“方向符号问题”处理并冻结
  - M4：先做 `wheel4` 原始 tick 探测，确认是 `Port B / D9,D10` 输入链问题，还是方向/控制问题
- 本轮不改协议层级，不推进 `cmd_vel -> W`，不调几何参数，不推进 odom。

## Key Changes
### 1. UNO 侧：M3 冻结为方向符号修正，M4 进入原始探测
- 代码基线继续使用 `sketches/c003_uno_encoder_pid_bridge/c003_uno_encoder_pid_bridge.ino`，协议保持：
  - 输入：`W,w1,w2,w3,w4`
  - 输出：`CSR_UNO_READY` / `ACK:W` / `ERR:<reason>` / `ENC,...` / `PID,...`
- `M3` 的实施口径直接冻结为：
  - 保持物理真相映射不变：`wheel3 pinA=A4, pinB=A5`
  - 不采用长期 A/B 交换方案
  - 将 `ENC_SIGN_W3` 正式改为 `-1`
  - 低速联调只看 `ENC/PID` 的符号与幅值关系，不再用“肉眼看轮子快慢”做主判断
- `M4` 的实施口径固定为“先 raw，后闭环”：
  - 新增编译期开关，例如 `CSR_WHEEL4_RAW_PROBE`
  - raw 模式下：
    - 不做整车闭环
    - 不用 `PI`
    - 只初始化 `wheel4`
    - 持续打印 `RAW4,ticks,currentA,currentB`
  - raw 探测通过前，不允许先拍脑袋改 `Kp/Ki` 或几何参数
- `wheel4` raw 模式的实现要求写死：
  - 输入映射继续是 `pinA = D9`, `pinB = D10`
  - 使用现有 `PCINT0_vect` 路径
  - 只关注 `D9` 的 A 相上升沿是否能让 tick 改变
  - `currentA/currentB` 必须直接反映实时引脚状态，便于区分“没进中断”和“进了但方向逻辑有问题”
- `M4` 的诊断决策树固定为：
  - raw 无 tick：
    - 结论写成“Port B 输入链未通过”
    - 先查硬件路径，不再调 PI
  - raw 有 tick 但方向反：
    - 优先试 `ENC_SIGN_W4 = -1`
  - raw 有 tick 且方向正常，但闭环异常：
    - 再进入 `PI / PWM / 驱动方向` 诊断
- `UNO` 文档新增：
  - `docs/SOFTWARE/C-2.2.8A_uno_m3_m4_diagnosis.md`
- 该文档必须明确记录：
  - 当前真实代码状态：`PID,...` 已存在
  - `M3` 最终是否通过 `ENC_SIGN_W3 = -1` 解决
  - `M4` raw 探测结果
  - `D9/D10` 是否能产生 tick
  - 下一轮是否还需要继续查 `Port B` 物理路径

### 2. 树莓派侧：本轮只增强观测，不改协议
- 树莓派侧继续以 `W,...` 为唯一真相协议，不改发送频率，不恢复 `CMD,vx,vy,wz`。
- 本轮树莓派桥只做 3 件事：
  - 所有返回帧打时间戳并分类型显示：
    - `ACK:W`

    - `PID,...`

  - 预置低速测试模板：

    - `W,120,120,120,120`
    - 一组单轮/单侧差速模板，例如 `W,120,120,80,120`
  - README 明确写：本轮只看 `ACK + ENC + PID`，不看 odom，不看 `cmd_vel`
- 树莓派发布文档新增：
  - `Raspberrypi/releases/C-2.2.8/C-2.2.8B_pi_observation_notes.md`
- 该文档必须写清：
  - 当前发送的测试命令
  - 当前收到的返回帧
  - 如何从日志判断 M3/M4 是编码器链问题还是控制问题
  - 当前仍未启用 `cmdvel_to_wheels`

### 3. 当前真实代码/文档口径要先对齐
- `C-2.2.8` 实施前先更新认知，不要按旧判断开工：
  - `PID,...` 现在已经在 `SerialProtocol::sendPidTelemetry()` 和 `CsrBaseController::runControlLoop()` 中实现
  - 当前未提交工作树里 `ENC_SIGN_W3 = -1` 已存在，且用户实测有正向效果
- 因此本轮不再把“补 PID 输出”当任务，而是把它当**已完成基线**写进诊断文档。
- 文档中必须明确说明：
  - `D9/D10` 不是外部中断脚，但它们是 `Port B` 的 `PCINT` 输入
  - `A0~A5` 是 `Port C / PCINT1`
  - 当前 `wheel4` 问题尚不能下结论为“D9/D10 不能用”，必须以 raw tick 探测结果为准

## Public Interfaces
- 本轮不改协议契约，继续沿用：
  - 输入：`W,w1,w2,w3,w4`
  - 输出：
    - `CSR_UNO_READY`
    - `ACK:W`
    - `ERR:<reason>`
    - `ENC,t1,t2,t3,t4,v1,v2,v3,v4`
    - `PID,target1,target2,target3,target4,meas1,meas2,meas3,meas4,out1,out2,out3,out4`
- 本轮唯一新增的调试接口是 `UNO` 的 raw 探测输出：
  - `RAW4,ticks,currentA,currentB`
- `CMD,vx,vy,wz`、`cmd_vel -> W`、odom 语义都不进入本轮实施。

## Test Plan
- **M3 低速验证**
  - 使用 `W,80,80,80,80`、`W,120,120,120,120`
  - 验收标准：
    - `wheel3` 物理映射仍是 `A4/A5`
    - `ENC_SIGN_W3 = -1`
    - `ENC` 中 `v3` 与 `v1/v2` 同号
    - `PID` 中 `meas3/out3` 不再呈现明显“反向放大”现象
- **M4 raw 探测**
  - 抬车，手拨 `wheel4`
  - 正拨/反拨都测试
  - 看 `RAW4`：
    - tick 是否变化
    - `currentA/currentB` 是否随拨动切换
  - 分支结论：
    - raw 无 tick：先查 `D9/D10` 物理路径和 servo 口导通
    - raw 有 tick：回到闭环，优先验证 `ENC_SIGN_W4`
- **树莓派观测**
  - 20Hz 持续发低速 `W` 帧
  - 日志中必须同时出现：
    - `ACK:W`
    - `ENC,...`
    - `PID,...`
  - 不能只看动作现象，不看回传数据
- **回退保护**
  - `c002` 继续能单独编译和工作
  - `c003` raw 模式和闭环模式都能编译
  - 不引入协议回退、不引入 `cmd_vel`
