## 1. 本轮定位

`C-3.0.4` 是底盘下位机从 `UNO` 切换到 `OpenRF1` 的首轮运行时基线建立。

2026-04-16 当前已冻结的阶段性结论：

- 底层驱动最小测试入口已打通，不再是“板子完全不受控”的状态
- 当前最可信的本地工作副本是：
  - `_local/openrf1_keil_work_2/`
- 当前最可信的底层真值入口是：
  - `_local/openrf1_keil_work_2/User/main.c`

本轮只处理：

- `OpenRF1` 控制板
- 四个霍尔编码电机

## 2.1 底层通道映射真值

当前必须明确区分三层语义：

- `CN1/CN2/CN3/CN4`：板上四个 6Pin 电机接口
- `MOTOR_A/B/C/D`、`ENCODER_A/B/C/D`：vendor 代码里的底层通道名
- `L1/R1/L2/R2`：vendor 在 `app_motor.c` 注释里的逻辑轮位

当前经 `y_motor.c`、`y_encoder.c` 与网表核对后的底层真值为：

| vendor 底层通道 | 实际板上接口 | 关键底层资源 |
| --- | --- | --- |
| `MOTOR_A / ENCODER_A` | `CN2` | `TIM8_CH2 + PA11` / `TIM3` |
| `MOTOR_B / ENCODER_B` | `CN4` | `TIM8_CH4 + PC10` / `TIM4` |
| `MOTOR_C / ENCODER_C` | `CN1` | `TIM8_CH1 + PA8` / `TIM5` |
| `MOTOR_D / ENCODER_D` | `CN3` | `TIM8_CH3 + PA12` / `TIM2(remap)` |

因此：

- vendor 底层 `A/B/C/D` 不是板子 `CN1/CN2/CN3/CN4` 的顺序
- 单轮方向测试必须优先按 `CN1~CN4` 记录
- 不能再把 `MOTOR_A` 直接理解成“板上第 1 路”
- `app_motor.c` 中的 `L1/R1/L2/R2` 注释代表的是 vendor 上层逻辑轮位，不是板上接口序号

## 3.3 2026-04-16 实机方向记录

在将板上 `B/C` 反接恢复为原始接法后，用户对“单轮正向轮测”的肉眼结论为：

- `A+ = 反`
- `B+ = 反`
- `C+ = 正`
- `D+ = 正`

这条记录必须保留为当前 OpenRF1 bring-up 阶段的已知事实。

当前解释约束：

- 该结论来自 `vendor` 代码拷贝上的最小单轮测试，不来自旧 `_local/openrf1_keil_work` 的自定义桥接代码
- 该记录使用的是当时测试入口里的 `A/B/C/D` 代码通道名，不等于板子 `CN1/CN2/CN3/CN4` 顺序
- 当前仍不能据此直接冻结“哪两个物理轮反了”，因为底层通道顺序与板子接口顺序并不一致

## 3.4 2026-04-16 串口自测结果

- 已在 `_local/openrf1_keil_work_2/User/main.c` 加入 `USART2(COM14)` 最小调试输出。
- 当前调试输出格式：
  - `CSR_RF1_READY`
  - `PHASE,CNn,±,START`
  - `PHASE,CNn±,CNT,a,b,c,d`
- 当前串口自测结论：
  - `UART2/CH340` 已打通，可以稳定抓到 phase 与计数回传
  - 强响应主要出现在：
    - `CN2- -> ENCODER_A` 大计数
    - `CN4- -> ENCODER_B` 大计数
  - `CN1/CN3` 当前未出现与其预期编码器通道一致的大计数回传
- 这说明：
  - 当前问题已经不是“完全看不到底层现象”
  - 真正需要继续排查的是 `CN1~CN4` 的物理驱动/编码器链与 vendor 通道映射之间的关系

## 3.5 2026-04-16 底层真值测试通过结论

- 用户已确认：
  - 当前板上已经能够稳定看到“四轮轮流前后转测试”
- 这条结论的意义是：
  - OpenRF1 板级底层驱动资源并未失效
  - 先前的混乱主要来自 vendor 代码中的语义错位与测试入口不干净，而不再是“板子根本驱不动”
- 因此本轮正式冻结：
  - 底层方向排查以 `_local/openrf1_keil_work_2` 为唯一真值入口
  - vendor `app_motor_run()` 仅保留为历史闭环参考，不再作为底层正反方向判断依据
  - 后续若要做 `W,va,vb,vc,vd` 协议版，必须建立在这份最小底层测试入口之上

## 4. 串口协议冻结

### 4.1 上位机输入

当前最小控制协议固定为：

```text
W,va,vb,vc,vd
```

其中：

- `va~vd` 为四个轮子的目标速度
- 单位固定为 `m/s`

示例：

```text
W,0.10,0.10,0.10,0.10
```

### 4.2 板端输出

启动完成：

```text
CSR_RF1_READY
```

合法命令：

```text
ACK:W
```

非法命令：

```text
ERR:bad_prefix
ERR:arg_count
ERR:parse_float
```

周期速度遥测：

```text
VEL,rt1,rt2,rt3,rt4,tg1,tg2,tg3,tg4
```

周期 PWM 遥测：

```text
PWM,p1,p2,p3,p4
```

## 5. 控制节拍与超时

当前控制节拍冻结为：

- `20ms / 50Hz`

当前串口遥测默认节拍冻结为：

- `100ms / 10Hz`

当前命令超时冻结为：

- `400ms`

超时动作固定为：

- 四轮目标速度清零
- 调用 `motor_speed_set(0,0,0,0)`
- PWM 输出回零

## 6. 应用层职责

### 6.1 app_csr_bridge

负责：

- 串口接收行缓冲
- 解析 `W,va,vb,vc,vd`
- 下发 `ACK:W`
- 下发 `ERR:<reason>`
- 维护命令超时计时

### 6.2 app_chassis

负责：

- 维护四轮目标速度
- 对接 `motor_speed_set(float,float,float,float)`
- 提供停止与超时停车

### 6.3 app_telemetry

负责：

- 周期输出 `VEL`
- 周期输出 `PWM`
- 收口统一格式与频率
