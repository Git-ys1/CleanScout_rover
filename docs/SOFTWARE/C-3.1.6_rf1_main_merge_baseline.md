# C-3.1.6：RF1 稳定基线补充回灌到 main

## 1. 本轮目标

本轮不继续推进 RF1 新功能，只做主线回灌与统一归档。

当前回灌目标有四类：

- `README.md`
- `docs/`
- `jixiebi/`
- RF1 运行主线代码：
  - `_local/openrf1_keil_work_3/`
  - `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/BH-STM32.uvprojx`
  - `_local/openrf1_keil_work_2/Libraries/`
  - `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/RTE/`
  - `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/DebugConfig/`
  - `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/EventRecorderStub.scvd`

本轮明确不带入：

- `J-jixiebi/`
- 当前工作树中的未跟踪大包资料
- 与 RF1/J 线无关的脏目录

## 2. 回灌源选择

当前公开可核的最新 RF1 稳定分支为：

- `feature/c-3.1.4a-openrf1-timer-good-encoder-retest`

选择该分支作为本轮唯一回灌源，原因如下：

- 它是当前最后一版原生定时器主线冻结点；
- `C-3.1.3` 软编码器分支已经完成问题定位与止损职责，应只保留为证据链；
- `C-3.0.7 / C-3.0.8 / C-3.1.0 / C-3.1.3` 后续分支主要承担诊断与归因，不适合作为正式主线继续叠补丁。

## 3. 为什么没有直接把 feature/c-3.1.4a 整分支 merge 到 main

本轮不直接整分支 merge，原因很直接：

- 相对 `main`，该分支历史中还夹带了大量阶段性文档演进；
- 当前仓库根目录存在无关脏文件；
- 本轮要求的是“补充合并”，不是覆盖主线。

因此实际做法是：

1. 从干净的 `main` 工作副本新开集成分支 `feature/c-3.1.6-openrf1-main-merge`
2. 从 `feature/c-3.1.4a-openrf1-timer-good-encoder-retest` 只恢复当前确认要进入主线的稳定内容
3. 在 `feature/c-3.1.6-openrf1-main-merge` 上提交
4. 再用 `--no-ff` 合并回 `main`

## 4. 本轮实际回灌内容

### 4.1 RF1 主线代码

当前回灌后的 RF1 统一代码基线是：

- `_local/openrf1_keil_work_3/` 自写运行层
- `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/BH-STM32.uvprojx` 对应的 Keil 工程

同时补齐了能够在 `main` 上独立编译该工程所必须的支撑目录：

- `_local/openrf1_keil_work_2/Libraries/`
- `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/RTE/`
- `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/DebugConfig/`
- `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/EventRecorderStub.scvd`

这是本轮新增的关键修正点。若只恢复 `_work_3` 和 `.uvprojx`，`main` 上的 Keil 工程无法独立编译。

### 4.2 文档入口

本轮将稳定分支中已跟踪的 `docs/` 统一带入 `main`，原因是：

- 当前分支较多，分散在自测分支中的文档不利于检索与交接；
- `docs/` 已经形成阶段化归档体系，适合作为统一证据链进入主线；
- 这次带入的是稳定分支里**已跟踪**的文档，不是把本地未整理的大包资料直接塞进主线。

### 4.3 J 线实验副本

本轮将小写目录：

- `jixiebi/`

补入 `main`，作为 J 线受控实验副本与可二开资料。

本轮明确不带入：

- `J-jixiebi/`

原因是后者当前仍属于原始资料/脏目录，不满足主线交接要求。

### 4.4 README

本轮同步回灌 `README.md`，使主线仓库在根入口就能指向：

- 系统总纲
- `docs/` 归档入口
- `jixiebi/` 受控实验副本
- RF1 当前主线背景

## 5. 软编码器分支的处理原则

分支：

- `feature/c-3.1.3-openrf1-tracking-port-soft-encoder`

本轮保留，但只作为证据链使用。

它的作用已经固定为：

- 说明当时为何需要止损绕行；
- 保存坏编码器排障路径；
- 作为后续复盘时的历史证据。

它不是当前 RF1 正式主线。

## 6. 当前 main 形成的统一基线

本轮合并完成后，`main` 中 RF1 统一基线应理解为：

- 代码主线：`C-3.1.4A` 原生定时器复测收敛结果
- 证据分支：`C-3.1.3` 软编码器止损与坏编码器证伪链
- 文档主入口：根 `README.md` + `docs/`
- J 线受控实验副本：`jixiebi/`

也就是说，当前 `main` 不是“软编码器正式主线”，而是“原生定时器主线 + 完整证据链归档”的统一交接版本。

## 7. 本轮结果

本轮完成的事情只有一类：

- 把 RF1 最新稳定基线、主线文档和 J 线受控资料，干净补充合并到 `main`

本轮没有做：

- 新一轮 RF1 功能开发
- 新一轮协议扩展
- 树莓派接入
- ROS 对接

当前 `main` 的价值是：

- 可追溯
- 可交接
- 可继续在统一基线之上推进下一轮 RF1 工作
