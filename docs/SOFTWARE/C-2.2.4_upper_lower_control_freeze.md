# C-2.2.4 上位-下位控制真相冻结

## A. 当前已验证事实
- 树莓派侧已能通过 `USB Serial@115200` 打开 `UNO` 串口链路。
- 当前 `UNO` 草图上电后会输出 `CSR_UNO_READY`。
- 当前联调已验证：
  - 发送 `5` 可收到 `ACK:5`
  - 发送 `8` 可收到 `ACK:8`
  - 发送非法字符如 `X` 可收到 `ERR:X`
- 当前链路已经打通到“命令与回显”，问题不再是“树莓派发不出去”，而是“当前控制语义和长期控制语义尚未冻结一致”。
- `UNO` 打开串口即可能因 `DTR` 触发自动复位，因此 `READY` 是可信握手信号，固定秒级等待不是根治方案。

## B. 当前联调模式
- 当前联调脚本工作方式是：一次打开串口、发送一个字符、然后退出。
- 在这种单发测试方式下：
  - `400ms` 只适合“上位机持续刷命令”的系统
  - 不适合人工单发观察
- 当前把命令保持时间拉长到 `2s~5s`，本质上是联调友好策略，用于肉眼观察，不是导航正式参数。
- 当前本地 `UNO` 草图存在未提交实测差异：
  - `COMMAND_TIMEOUT_MS = 4000UL`
  - 该差异只作为联调证据成立，不属于本轮代码冻结内容
- 当前联调阶段的真相口径应冻结为：
  - 允许单次命令保持较长时间
  - 允许人工单发命令看现象
  - 固定等待 `2 秒` 不是长期方案
  - 后续握手必须改为等待 `CSR_UNO_READY`

## C. 正式控制模式冻结
- 当前 `UNO + AFMotor` 只承担底层执行，不是成熟底盘控制器。
- 长期正式控制模式固定为：
  - 树莓派持续发送速度设定值
  - 下位机始终只执行最近一帧有效命令
  - 若超过短超时未收到新命令，下位机立即急停
  - 下一轮接入霍尔编码电机后，下位机升级为轮速闭环，并向上位机回传编码器/里程计基础量
- 长期正式接口语义固定为速度，不再冻结为 `1~9` 数字动作码。
- 对当前麦轮底盘，长期速度语义应保留：
  - `linear.x`
  - `linear.y`
  - `angular.z`
- 多来源控制的长期入口固定为上位速度总线，后续统一收口到 `Twist` 语义；多来源优先级管理按 `twist_mux` 思路冻结。
- 当前动作码协议只定义为联调过渡协议：
  - 当前过渡协议：`1~9` + `CSR_UNO_READY / ACK / ERR`
  - 长期正式协议：速度设定帧，至少包含 `vx / vy / wz`

## D. 下一轮前置条件
- 若要做成熟 `PID / 轮速控制 / odom / 巡航 / SLAM`，必须接入霍尔编码电机。
- 当前无编码器阶段只能定义为开环或半开环联调，不得定义为正式导航底盘。
- 下一轮应冻结为：
  - 上位机持续发送速度命令，推荐 `10Hz` 起步，可提升到 `20Hz`
  - 下位机短超时急停，推荐 `0.3s~0.5s`
  - 下位机最少回传：`ACK / 最近命令 / 当前模式 / 错误码`
  - 接入编码器后再扩展：编码计数、实际轮速、里程计基础量
