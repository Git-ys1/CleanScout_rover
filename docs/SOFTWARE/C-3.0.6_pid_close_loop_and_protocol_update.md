# C-3.0.6：OpenRF1 闭环与持续协议更新说明

## 1. 本轮目标

本轮在 `_local/openrf1_keil_work_3/User` 的自写运行层上，直接推进三件事：

- 禁用 JTAG，保留 SWD
- 在当前开环真值基线上叠加四轮 PI 闭环
- 将协议改为目标值锁存 + 固定频率控制 + 超时停车

本轮不回退到重测开环真值表，不回到 `_work_2` 的 vendor 运行语义，也不引入 ROS、树莓派或运动学扩展。

## 2. JTAG 禁用与 SWD 保留

本轮将 SWJ 配置前移到 `main()` 最早期执行：

```c
RCC_APB2PeriphClockCmd(RCC_APB2Periph_AFIO, ENABLE);
GPIO_PinRemapConfig(GPIO_Remap_SWJ_JTAGDisable, ENABLE);
```

放置顺序固定为：

1. `csr_init_debug_ports()`
2. `csr_systick_init()`
3. `csr_motor_init()`
4. `csr_encoder_init()`
5. `csr_proto_init()`

这样做的目的不是关闭全部调试口，而是：

- 释放 `PA15 / PB3 / PB4`
- 保留 `PA13 / PA14` 作为 SWD 下载调试口

同时，`csr_encoder_drv.c` 中 `CN3/TIM2` 初始化里原先“顺手禁用 JTAG”的隐式行为已移除，只保留 `TIM2` remap。

## 3. 当前控制周期与协议

### 控制与遥测节拍

- 控制循环：`20ms / 50Hz`
- 遥测循环：`100ms / 10Hz`

### 主协议

主控制命令固定为：

```text
W,a,b,c,d
```

含义：

- `a~d` 为四个轮子的目标速度
- 单位固定为 `m/s`

收到 `W` 后，运行层会：

- 更新四轮 `target_vel`
- 切换到 `CLOSED_LOOP` 模式
- 持续运行控制循环
- 不会在单次执行后立即停车

### 调试协议

以下命令继续保留，但只作为 bringup 和诊断入口：

- `M,<ch>,<pwm>`
- `E,<ch>`
- `D,<ch>`
- `STOP`

其中：

- `M` 进入 `RAW` 模式，直接锁存原始 PWM
- `E` 只返回缓存的 `count / last_delta`，不再打断闭环采样窗口
- `D` 返回该通道编码器 A/B 输入脚当前电平、定时器计数、累计计数和最近一次 delta，用于 CN1/CN3 原始链路排查
- `STOP` 会清零目标、清零 PI 状态并立即停车

### 板端回传

- `CSR_RF1_READY`
- `ACK:W`
- `ACK:M`
- `ACK:E`
- `ACK:STOP`
- `ERR:<reason>`
- `VEL,rt1,rt2,rt3,rt4,tg1,tg2,tg3,tg4`
- `PWM,p1,p2,p3,p4`
- `ENC,<ch>,<count>,<delta>`
- `DBG,<ch>,<phaseA>,<phaseB>,<timerCount>,<count>,<delta>`

## 4. 当前 PI 闭环口径

### 速度换算

当前第一版闭环沿用现有默认参数：

- `CONTROL_HZ = 50`
- `WHEEL_RESOLUTION = 1560.0`
- `WHEEL_DIAMETER_M = 0.08`
- `wheel_scale = PI * WHEEL_DIAMETER_M * CONTROL_HZ / WHEEL_RESOLUTION`

编码器窗口值 `delta` 在每个 `20ms` 控制周期读取一次，并换算为当前实时速度 `RT (m/s)`。

### 控制器

本轮先固定为 PI：

- `kp[4] = 4500.0`
- `ki[4] = 500.0`
- `kd[4] = 0.0`

保护项固定为：

- 积分限幅：`±0.30`
- 输出限幅：`±1000`
- 速度低通：`filtered = 0.65 * previous + 0.35 * raw`
- PWM 斜率限制：每个 `20ms` 最多变化 `120`
- 低速 bringup 阶段禁止反向刹车：目标为正时输出不允许变负，目标为负时输出不允许变正
- 目标速度接近 `0` 时清积分
- 超时停车时清积分

### 方向修正

当前约束为：

- `csr_motor_drv.c` 继续保持四路已验证的开环 H 桥真值，不在底层再动方向模型
- `g_csr_motor_dir_sign[4]` 只在闭环控制层用于输出符号修正
- `g_csr_encoder_dir_sign[4]` 只在编码器层用于速度符号修正

当前默认闭环方向表冻结为：

- `motor_dir_sign = { +1, +1, +1, +1 }`
- `encoder_dir_sign = { +1, -1, +1, +1 }`

依据：

- `M` 原始调试命令已经在 `csr_motor_drv.c` 内统一为 `+pwm` 正向，`W` 层不再对 CN3/CN4 做二次翻转。
- 串口实测显示 CN2 在 `+pwm` 时编码器速度为负、`-pwm` 时编码器速度为正，因此闭环层先对 CN2 编码器符号反相。
- CN4 在初次判断中被误列为反相；C-3.0.6 复测 `M,4,+700 / M,4,-700` 后确认当前不需要编码器反相，因此 CN4 回到 `+1`。
- CN1/CN3 当前仍为无编码器计数状态；这不是符号表能修复的问题，本轮不再用方向表假装修好。

这组值服务于 `W` 模式的统一语义，不影响 `M` 原始调试命令。

## 5. 当前超时机制

本轮固定两档看门狗：

- `W` 闭环模式：`250ms`
- `M` 调试模式：`2000ms`

行为定义：

- 收到有效命令即刷新对应模式的超时计时
- 超时后立即执行 `STOP`
- `STOP` 会同时清目标、清积分、清输出

## 6. 当前已知仍待调优项

本轮完成的是“闭环主链和持续协议打通”，不是最终调参完成。

后续仍需继续收口的项：

- 每轮 `kp / ki` 独立调优
- CN1/CN3 编码器链路无计数问题：当前软件映射已按 vendor TIM5/TIM2 full remap 口径对齐，下一步应查硬件链路或板级专有差异
- 编码器方向符号二次确认，尤其是 CN1/CN3 在恢复计数后的符号回填
- `WHEEL_RESOLUTION / WHEEL_DIAMETER / wheel_scale` 实测校准
- `VEL` 与实际轮速的一致性复核
- `W` 持续发送下的四轮同速平顺性验证

## 7. 当前串口实测结论

本轮已在 COM14 / USART2 上验证：

- `CSR_RF1_READY` 正常输出
- `ACK:M` / `ACK:E` / `ACK:STOP` / `ACK:W` 正常
- `W` 模式 `250ms` 超时停车生效
- USART2 接收已改为 RX 中断环形缓冲，修复持续发送 `W` 时偶发 `ERR:bad_prefix / ERR:parse_float` 的问题
- `VEL` / `PWM` 周期输出正常
- CN2 / CN4 编码器有稳定计数
- CN1 / CN3 编码器当前读数仍为 `0`
- CN2 / CN4 在加入低通、禁止反向刹车和 PWM 斜率限制后，不再出现明显前后反向抽动
- `0.020 m/s` 当前低于可稳定连续运行区间：控制器起步后很快判定超速并把 PWM 拉回 `0`，后续应使用稍高目标或补充更细的起步/保持策略

对照 vendor 例程中的 TIM5 / TIM2 编码器初始化后，当前自写层的 CN1/CN3 软件映射未发现明显缺项：

- CN1：`TIM5 + PA0/PA1`
- CN3：`TIM2 full remap + PA15/PB3`
- JTAG 已在最早期禁用，SWD 保留

因此，当前不能再把问题归因于串口协议或 `W` 持续控制逻辑。下一步应优先排查 CN1/CN3 的编码器硬件链、接插件、线序、供电或 OpenRF1 板级专有差异。

## 8. 本轮建议的串口记录

至少留存以下三类记录：

1. 开环 `M` 模式可控，与闭环 `W` 模式可控的对比
2. 单轮低速 `W` 命令下的持续转动记录
3. 四轮同速 `W` 持续发送与超时停车记录
