# C-2.2.7 Serial Protocol Contract

## Scope

This file freezes the current Raspberry Pi to UNO serial contract for `C-2.2.7`.

## Current protocol owner

- current valid control protocol: `W`
- current bridge integration branch: `feature/c-2.2.7-integration-pi-uno-w-bridge`

## Pi to UNO

The Raspberry Pi sends wheel target ticks per second directly:

```text
W,w1,w2,w3,w4
```

Notes:

- `w1..w4` are signed wheel target values
- Raspberry Pi keeps sending at `20 Hz`
- Raspberry Pi waits for `CSR_UNO_READY` before treating the link as active

## UNO to Pi

UNO sends:

```text
CSR_UNO_READY
ACK:W
ERR:<reason>
ENC,t1,t2,t3,t4,v1,v2,v3,v4
PID,target1,target2,target3,target4,meas1,meas2,meas3,meas4,out1,out2,out3,out4
```

## Explicit non-contract items

The following is not the current active protocol and must not be used as the live bridge contract:

```text
CMD,vx,vy,wz
```

That frame belongs to an older Raspberry Pi assumption and is not the current UNO-facing protocol.

## Staging rule

Current stages are frozen as:

1. Pi sends `W,w1,w2,w3,w4`
2. UNO returns `ACK:W`, `ENC,...`, and `PID,...`
3. Only after `W` bridge is stable does Raspberry Pi add `cmd_vel -> W`
4. Only after that do odom, EKF, SLAM, and higher navigation layers reconnect

## Pass criteria

The only valid pass signal is:

1. Pi sends `W,...`
2. UNO returns `ACK:W`
3. `ENC` and `PID` continue updating consistently

The following do not count as pass on their own:

1. only receiving `CSR_UNO_READY`
2. only seeing Raspberry Pi serial writes succeed
3. only seeing UNO local prints
4. only proving `CMD,...` parsing in isolation
