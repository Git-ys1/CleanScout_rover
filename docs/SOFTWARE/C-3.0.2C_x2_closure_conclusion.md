# C-3.0.2C：X2 止损，回退 X1 作为当前同步转动基线

## Summary
- 本轮结论已经收敛，不再继续把 `C-3.0.2C` 扩成新的功能开发轮。
- 当前 `UNO` 协议未改，仍然是：
  - `W,w1,w2,w3,w4`
  - `CSR_UNO_READY / ACK:W / ERR:<reason> / ENC,... / PID,...`
- 本轮真实结论不是“`X2` 已通过”，而是：
  - `X2` 在当前硬件条件下会明显放大轮间差异
  - `X1` 曾经证明过可以让三轮流畅同步转动
  - 当前主线应回退到 `X1`
  - 下一轮应针对 `X1` 重调 `PI`，而不是继续让 `X2` 阻塞主线

## Current Facts
### 1. 当前已确认事实
- 当前测试主入口仍是：
  - [c003_uno_encoder_pid_bridge.ino](f:/Project/CleanScout_rover/sketches/c003_uno_encoder_pid_bridge/c003_uno_encoder_pid_bridge.ino)
- 当前测试串口仍是：
  - `COM14`
- 当前协议与串口观测链是正常的：
  - `ACK:W`
  - `ENC,...`
  - `PID,...`
- 当前问题不再是协议问题，也不再是“树莓派是否持续发包”的问题。

### 2. X2 的阶段性结论
- `X2` 这轮已经重新上板、编译通过、串口可观测。
- 但在 `W,300,300,300,300` 下，`X2` 仍无法把当前四轮闭环收敛到同步转动态。
- 因此当前对 `X2` 的结论必须写成：
  - `X2` 是一条未通过的优化实验线
  - 不是当前可冻结主线

### 3. X1 的真实地位
- 用户已明确反馈：在更早的 `X1` 阶段，三轮是可以一起流畅同步转动的。
- 因此当前应承认：
  - `X1` 才是目前已经被实测证明过的同步转动基线
  - 当前把 `X2` 调参直接落回 `X1` 后看到的异常，不应被误写成“`X1` 本身不行”
- 更准确的说法应是：
  - 当前回退后的 `X1` 仍挂着偏向 `X2` 的 `PI / 补偿` 口径
  - 所以它还不是“重新按 `X1` 正确整定后的代表状态”

## Engineering Judgement
# 结论：当前不应继续拿 X2 阻塞主线，应回退 X1，并在下一轮专门重调 X1 的 PI

这个结论分成四点：

1. 当前 `X2` 的主要问题不是协议，不是编译，不是串口通信。
2. 当前 `X2` 的主要问题是：在现有硬件差异下，它把轮间差异放大了。
3. 当前 `X1` 已有历史实测依据，证明它能实现更流畅的同步转动。
4. 因此当前主线应回到 `X1`，然后重新按 `X1` 的测速颗粒度与轮间特性重调 `PI`。

## Scope Boundary
本结论文档明确不再主张：
- 不把 `X2` 写成当前通过基线
- 不把下一轮继续定义成 `X2` 轮间差异优化轮
- 不把当前回退后带着 `X2` 参数的 `X1` 表现，误写成“`X1` 本身失败”

## Next Round Entry
下一轮 `C-3.0.2D` 的唯一目标应冻结为：
- 保持 `W` 协议不变
- 保持当前新版底盘常量不变
- 保持主线回到 `X1`
- 专门针对 `X1` 重新整定 `PI`
- 验证目标是“恢复三轮流畅同步转动”，而不是继续追 `X2`
