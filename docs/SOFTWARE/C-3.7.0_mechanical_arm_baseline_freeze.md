# C-3.7.0 机械臂官方基线冻结与独立开发起点

## 目标

从这一轮开始，`CleanScout_rover` 下位机进入机械臂版本 `C-3.7.0`。

本轮不把机械臂和 RF1 底盘强行融合，而是先明确三件事：

1. 当前最新机械臂官方例程是谁。
2. 这份官方例程在仓库里的正式位置是什么。
3. 后续机械臂调试从哪个空目录开始，避免污染底盘主线和官方基线。

## 本轮不是孤立改动

这次切到机械臂线，不是突然从资料包里随手拎一份代码出来开工，而是建立在下位机前一阶段已经完成的收口基础上：

1. `C-3.6.0` 已把 RF1 底盘正式固件收口到 `firmware/openrf1_motion_controller/`。
2. `C-3.6.1` 已清理 RF1 正式工程中的重复参数和无效控制代码。
3. 在确认底盘正式主线已有唯一入口之后，才允许把机械臂官方例程单独拉成 `C-3.7.0`。

也就是说，当前机械臂线的起点不是“和底盘并行乱改”，而是：

- 先把底盘正式主线稳住
- 再冻结机械臂官方基线
- 再从独立目录启动后续自研控制

## 目录冻结结果

### 1. 官方机械臂基线

- 正式路径：`firmware/mechanical_arm_official_baseline/`
- 定位：当前最新机械臂 STM32 官方例程冻结基线
- 工程文件：`firmware/mechanical_arm_official_baseline/Project/RVMDK（uv5）/BH-STM32.uvprojx`
- 目标芯片：`STM32F103RC`
- 可直接烧录产物：`firmware/mechanical_arm_official_baseline/Output/template.hex`

这份基线已经与 RF1 底盘正式固件目录分开，不再放在普通 `docs/` 资料区里混用。

### 2. 机械臂独立开发占位

- 正式路径：`firmware/mechanical_arm_controller/`
- 当前状态：仅保留占位文件 `.gitkeep`
- 定位：后续机械臂自研/调试入口

## 开发边界

当前阶段的边界明确如下：

1. `openrf1_motion_controller/` 继续代表底盘下位机正式主线。
2. `mechanical_arm_official_baseline/` 只保留官方冻结基线，不直接承接日常修改。
3. `mechanical_arm_controller/` 作为新开发目录，先独立跑通基础机械臂控制。
4. 机械臂与底盘的融合属于后续阶段，不在本轮基线冻结中完成。

## 回溯依据

本轮目录冻结前，已经结合当前仓库实际历史做过回溯核对，重点依据包括：

### 4. 知识库成熟项目审计

当前下位机子项目已经按 super-yusu 成熟项目流程做过完整回溯与审计，但知识库仓库由管理员统一提交；本主仓库只保留必要结论，不直接带入知识库内部管理文件。

## 当前判断

从源码入口和烧录核验看，这份官方例程的核心能力不是 RF1 四轮闭环，而是：

- 多路舵机控制
- 动作组解析与执行
- PS2 控制
- 串口指令解析
- W25Q64 参数/动作存储
- 简单机械臂运动学

因此它适合作为“机械臂独立控制板”的当前官方起点。

## 入库纪律

为避免后续再次混乱，机械臂线遵守以下规则：

1. 任何“官方对照”问题，先回到 `mechanical_arm_official_baseline/`。
2. 任何“我们自己的机械臂新增逻辑”，先落到 `mechanical_arm_controller/`。
3. 未经明确验证，不把机械臂代码直接并进 RF1 底盘正式固件。
4. 旧的 `jixiebi/` 仍保留为历史实验线，但不再代表当前 STM32 机械臂官方基线。
