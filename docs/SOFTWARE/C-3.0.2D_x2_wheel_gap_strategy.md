# C-3.0.2D：当前不再推进 X2，而是回到 X1 做 PI 重调

## Core Judgement
当前 `C-3.0.2D` 的唯一正确口径是：
- `X2` 暂停
- `X1` 回到主线
- 下一轮只做 `X1` 的 `PI` 重调

这不是技术后退，而是基线纪律。

## Why This Is Correct
### 1. X2 当前没有形成可用主线
- `X2` 已证明能编译、能上板、能回传 `ENC/PID`
- 但 `X2` 尚未证明能在当前硬件条件下实现稳定同步转动
- 因此它只能保留为未通过的优化实验线

### 2. X1 有历史实测依据
- 用户已明确确认：
  - 之前的 `X1` 阶段，三轮是可以流畅同步转动的
- 这意味着当前工程上最可靠的选择不是继续赌 `X2`
- 而是回到已经被证明过的 `X1` 基线

### 3. 当前回退后的异常不能直接归咎于 X1
- 当前从 `X2` 直接落回 `X1` 时，控制参数仍带有明显 `X2` 痕迹
- 所以当前若出现不同步，不能直接写成：
  - “`X1` 也不行”
- 更准确的判断是：
  - 当前 `X1` 还没有经过独立的 `PI` 重调

## Implementation Direction
本轮只围绕下面几项做：
- 分轮 `Kp`
- 分轮 `Ki`
- 分轮 `I limit`
- 分轮输出上限
- 分轮起步补偿

本轮明确不做：
- 不再切回 `X2`
- 不再把 `X2` 轮间差异作为当前主线目标
- 不继续围绕 `D9/D10 / Port B` 做 `X2` 解释链

## Diagnostic Rule
### 当前允许写进文档的判断
- `X2` 暂停
- `X1` 是当前应恢复的可用基线
- 下一轮的关键不是改协议，而是重调 `X1` 的 `PI`

### 当前不允许写进文档的判断
- “`X1` 已经被证明不行”
- “必须先把 `X2` 调通，项目才能继续”

## Next Step
下一轮按下面顺序推进：
1. 固定 `WheelEncoder` 为 `X1`
2. 保持当前新版底盘常量
3. 重新调 `X1` 的 `PI`
4. 以 `W,300,300,300,300` 为第一验收档
5. 目标是恢复三轮流畅同步转动
