# C-2.2.5A UNO 编码器接入架构真相冻结

## 1. 当前阶段定位
- 本轮性质：文档冻结轮。
- 当前下层执行基线仍为 `Arduino UNO + AFMotor v1`。
- 当前上层控制中心仍为树莓派。
- 当前 `1~9` 字符动作码仍只属于联调过渡协议，不是长期正式接口。
- 本轮不改 `UNO` 代码，不改 `Tyler_1`，不引入编码器采样实现，不迁移风机。

## 2. 当前引脚预算重算
### 2.1 已被 AFMotor 固定占用
- `D3 / D5 / D6 / D11`：4 路 PWM
- `D4 / D7 / D8 / D12`：AFMotor 锁存控制

### 2.2 当前可用于编码器输入的口
- `A0`
- `A1`
- `A2`
- `A3`
- `A4`
- `A5`
- `D2`
- `D9`
- `D10`

### 2.3 必须写死的口径
- `SERVO1 / SERVO2` 本质分别是 `D9 / D10` 数字口；不用舵机时，它们就是普通数字输入。
- `A0~A5` 可以作为数字输入参与编码器采样。
- `D0 / D1` 继续保留给 `USB Serial`，不纳入当前编码器方案。
- `D13` 继续不作为当前编码器优先输入口。

## 3. 硬件映射冻结结论
### 3.1 优先推荐映射
前提：先拔掉 `A2` 上的风机继电器控制线。

| 电机通道 | A 相 | B 相 |
| --- | --- | --- |
| `M1` | `A0` | `A1` |
| `M2` | `A2` | `A3` |
| `M3` | `A4` | `A5` |
| `M4` | `D9` | `D10` |

冻结理由：
- `A0~A5` 全部集中在 `Port C`
- `D9/D10` 集中在 `Port B`
- 8 路输入分组整洁，最适合当前 `UNO` 走 `PCINT`
- 不需要碰 `D0/D1`
- 不需要把某个编码器对拆到杂散口

### 3.2 备选映射
前提：`A2` 暂不释放，风机继电器仍占用。

| 电机通道 | A 相 | B 相 |
| --- | --- | --- |
| `M1` | `A0` | `A1` |
| `M2` | `A3` | `A4` |
| `M3` | `A5` | `D2` |
| `M4` | `D9` | `D10` |

冻结结论：
- 该方案物理上可行，但不作为首选。
- 问题不在“脚不够”，而在“编码器对被拆到更不整洁的口”。
- `D2` 只作为备选输入，不是当前优先方向。

### 3.3 风机口径
- 当前推荐先拔掉 `A2` 风机继电器控制线，换取整洁的 8 路编码器输入。
- 这不是永久放弃风机，而是当前编码器接入轮的优先级选择。
- 风机迁移到树莓派或其他控制路径，本轮不做，不与编码器接入轮混做。

## 4. 编码器采样方向冻结
### 4.1 当前推荐实现方向
- `UNO` 端编码器采样优先走 **Pin Change Interrupt (PCINT)**。
- 不走“8 路 A/B 相全部依赖 `attachInterrupt()`”的方案。

### 4.2 当前推荐端口分组
- `A0~A5`：`Port C`，对应 `PCINT8~PCINT13`
- `D9 / D10`：`Port B`，对应 `PCINT1 / PCINT2`
- 备选中的 `D2`：`Port D`，对应 `PCINT18`

### 4.3 必须写死的技术判断
- 这轮真正的技术难点不是“够不够脚”，而是“如何稳定读取 8 路正交编码器”。
- 对 `UNO` 来说，外部中断强口本来就有限，不能把方案建立在“8 路 `attachInterrupt()`”的假设上。
- 优先映射优于备选映射，不是因为“多了脚”，而是因为 `Port C + Port B` 的分组更整洁，`PCINT` 处理更统一。

## 5. 编码器数据的用途
- 四轮原始 `tick` 计数
- 四轮方向判断
- 四轮速度估计
- 下一轮四轮速度 `PI/PID` 闭环
- 轮级反馈到车体级速度估计
- 后续 `odometry` 基础量
- 后续树莓派 / ROS 建图、定位、巡航所需的底层反馈基础

冻结结论：
- `A/B` 相数据不是“回传看看热闹”，而是后续闭环控制和里程计的基础。
- 如果没有这部分反馈，当前 `UNO` 只能继续停留在开环动作机阶段。

## 6. 下一轮代码实施顺序
### 第 1 步：原始采样阶段
- 只做四轮编码器原始采样与计数
- 输出：
  - `tick`
  - `direction`
  - `wheel speed estimate`
- 本阶段不急着做闭环，不急着改 ROS 协议

### 第 2 步：闭环控制阶段
- 再做四轮速度 `PI/PID`
- 输入：目标轮速
- 输出：PWM / 方向
- 初始建议先上 `PI`
- `D` 项后置，避免速度噪声过早放大

### 第 3 步：协议对接阶段
- 最后再进入与树莓派 / ROS 的协议对接
- 至少回传：
  - `tick`
  - `wheel speed`
- 有余力时再扩展：
  - `battery`
  - `state`
  - `odom` 基础量

冻结结论：
- 下一轮必须按“先采样、后闭环、再对接”的顺序推进，不允许跳步。

## 7. 建议频率
- 编码器边沿采样：事件驱动 `PCINT`
- 轮速估计周期：`20ms (50Hz)` 起步
- 轮速控制周期：`20ms (50Hz)` 起步
- 树莓派正式速度命令频率：`10Hz` 起步，推荐向 `20Hz` 收敛

当前口径：
- 本轮不写 PID 代码
- 本轮冻结的是实施秩序和接口真相，不是直接实现控制器

## 8. 与树莓派 / ROS 的长期接口关系
### 8.1 当前联调阶段
- 当前仍允许保留 `1~9` 字符动作码做底层联调。
- 该协议只属于过渡协议，不作为长期正式接口冻结。

### 8.2 长期正式语义
- 长期正式接口不是动作码，而是速度语义。
- 对当前麦克纳姆底盘，长期冻结为：
  - `vx`
  - `vy`
  - `wz`
- 对应 ROS 语义：
  - `Twist.linear.x`
  - `Twist.linear.y`
  - `Twist.angular.z`

### 8.3 上下位职责
- 树莓派负责：
  - 上层感知
  - 规划
  - 导航
  - OpenClaw 等上位能力
- `UNO` 负责：
  - 电机执行
  - 编码器采样
  - 下一轮轮速闭环
  - 里程计基础量回传

### 8.4 当前系统结论
- 当前无编码器阶段只能定义为开环或半开环联调阶段。
- 不得把当前底盘写成“已具备正式导航基础”。
- 下一轮若要进入成熟 `PID / odom / 巡航 / SLAM`，霍尔编码电机是前置条件。

## 9. 本轮未冻结的内容
- `M1~M4` 与物理车轮 `前左/前右/后左/后右` 的最终对应关系
- 编码器采样的具体寄存器实现
- 速度估计公式与滤波细节
- 四轮 `PI/PID` 参数
- 树莓派到 `UNO` 的正式速度帧协议
- 风机迁移方案

冻结要求：
- 下一轮实施前，必须先完成 `M1~M4` 对物理车轮位置的一次性核对。

## 10. 当前真相结论
- 当前 `UNO` 编码器方案在硬件上是可行的，不需要去挖 AFMotor 的隐藏引脚。
- 优先路线是先释放 `A2`，用 `A0~A5 + D9 + D10` 形成整洁的 8 路编码器输入。
- 当前推荐采样方向是 `PCINT`，不是 8 路外部中断。
- 当前推荐工程顺序是：先采样，再闭环，最后再对接树莓派 / ROS。
- 当前长期正式接口语义必须冻结为 `vx / vy / wz`，不是 `1~9` 动作码。
