# C-1.1.3B Event vs Mode

## 1. 目标

本文件用于说明为什么 `C-1.1.3B` 必须把蓝牙输入拆成模式命令、事件命令和运动命令三类。

## 2. 三类命令

| 类别 | 字符 | 作用 |
| --- | --- | --- |
| 模式命令 | `A / M / T` | 只修改 `systemMode` |
| 事件命令 | `G` | 只触发一次风机翻转 |
| 运动命令 | `1..9` | 只在人工控制域驱动运动 |

## 3. 为什么 G 必须是事件命令

`G` 的问题根因不是风机模块，而是旧代码中把单个字符同时用于：

- 模式切换
- 风机开关

如果 `G` 继续留在 `cmdChar` 这种持续态命令变量里，会带来两个直接问题：

- 一次 `G` 可能把当前模式顶掉
- 一次 `G` 可能在多个 loop 周期里反复执行

因此本轮固定采用：

- 接收到 `G` 后只置位 `fanTogglePending`
- 本轮循环中消费一次
- 消费后立即清零

## 4. 为什么 G 不能改 systemMode

`G` 的唯一语义已经冻结为风机翻转事件。

这意味着：

- `G` 不允许再进入体感模式
- `G` 不允许再触发停车作为模式副作用
- `G` 不允许再把 `MAN` 或 `AUTO` 顶成 `GEST`

无论当前模式是：

- `MAN`
- `GEST`
- `AUTO`

处理 `G` 时都必须保持 `systemMode` 不变。

## 5. AUTO 下的处理

为了避免破坏自动风机策略，本轮冻结为：

- `AUTO` 下接收 `G`
- 事件立即消费
- 风机逻辑不被人工翻转打断
- 模式保持 `AUTO`

## 6. 模块边界

本轮不允许在 sketch 中重新散写风机 GPIO 业务逻辑。

风机仍由 `CleanScoutFan` 统一管理，sketch 只负责：

- 输入分类
- 模式切换
- 一次性事件转交
