# C-2.2.4 串口链路说明

## 1. 当前链路口径
- 当前链路：`Raspberry Pi -> UNO 板载 USB 虚拟串口`
- 当前波特率：`115200`
- 当前联调协议：单字符动作命令 + 串口回显
- 当前握手信号：`CSR_UNO_READY`

## 2. UNO 自动复位事实
- 当前 `UNO` 属于 AVR `USB Serial` 打开即可能自动复位的板型。
- 根因是主机打开串口时，`DTR` 会触发复位。
- 这就是当前每次重新打开串口后，都需要重新等待 `READY` 的原因。

## 3. READY 握手机制
- 当前联调脚本不应继续依赖固定等待 `2 秒` 作为长期方案。
- 正确方向是：
  - 打开串口
  - 等待 `CSR_UNO_READY`
  - 收到 `READY` 后再下发控制命令
- 这样做比固定 sleep 更稳，因为它直接对齐 `UNO` 自动复位后的真实可用时刻。

## 4. 当前设备名与固定别名策略
- 当前 `UNO` 的 USB 设备名不应冻结为 `/dev/ttyACM0` 这一类会漂移的名字。
- 联调阶段允许临时使用系统分配名，例如：
  - `/dev/ttyACM0`
  - `/dev/ttyACM1`
- 但长期固定策略必须收敛为以下二选一之一：
  - 优先使用 `/dev/serial/by-id/...` 这种稳定设备路径
  - 若项目需要更短路径名，则额外定义自有 `udev` 别名，例如 `clean_scout_uno`
- 当前真相文档冻结的是“固定别名策略必须建立”，不是“当前数值尾缀设备名已经最终冻结”。

## 5. 当前联调协议性质
- 当前协议只用于串口 bring-up 和动作链联调。
- 当前 `1~9` 动作码协议不是长期正式底盘接口。
- 当前 `ACK / ERR / READY` 也只是最小联调回显，不等于后续正式状态回传协议。

## 6. 当前 4000ms 实测的解释
- 当前本地草图中存在 `COMMAND_TIMEOUT_MS = 4000UL` 的未提交实测差异。
- 这个现象说明：
  - 当前单发脚本要看清动作，需要更长观察窗
  - 不说明长期正式控制应该采用 `4000ms` 超时
- 长期正式控制仍应回到：
  - 上位机持续发送速度命令
  - 下位机短超时急停

## 7. 下一轮改进方向
- 树莓派脚本去掉固定秒级等待，改为 `READY` 握手。
- 当前单发字符联调完成后，尽快切换到流式速度命令模式。
- 接入霍尔编码电机后，再冻结正式的速度闭环与 odom 回传口径。
