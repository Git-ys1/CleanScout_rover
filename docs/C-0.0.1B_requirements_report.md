# C-0.0.1B 需求报告（给项目负责人）

## 1. 本轮目标与结果

- 目标：确认当前 C 线代码在 UNO + AFMotor 场景下是否可编译，并给出后续实施所需环境与需求清单。
- 结果：
- 已确认 `F:\AcademicHub\Arduino` 工具链可用于命令行编译。
- `Tyler_1_Library/examples/Tyler_1/Tyler_1.ino` 在 `arduino:avr:uno` 上编译通过。
- 已将 `Adafruit-Motor-Shield-library-master` 作为当前基线依赖纳入仓库。

## 2. 当前已验证信息

- 主控硬件：Arduino UNO。
- 电机扩展：AFMotor（Adafruit Motor Shield v1 系列库）。
- C 线基线：`Tyler_1_Library`（已接入仓库，后续在该基线上二次开发）。

## 3. 需要项目负责人补充/确认的需求项

1. 移植边界确认
- 是否保持 `Tyler_1` 对外 API 不变（兼容原示例）作为第一阶段约束。
- 是否允许在 C-0.0.x 阶段调整目录命名与库结构（如统一到 `libraries/`）。

2. 功能优先级确认（C-0.x 之后）
- 优先做“稳定可控遥控”还是“自动避障”。
- 是否需要把体感模式纳入近期交付（当前与手动共用控制字符解析）。

3. 硬件参数确认
- 电机方向参数 `dir1~dir4` 的目标标定值是否按现有车体固定。
- 超声与舵机默认引脚是否沿用当前示例（A0/A1/10）。
- 蓝牙模块型号与串口接线是否固定（HC-06 TX 到 D9）。

4. 验证口径确认
- “可编译通过”是否作为 C-0.0.x 完成标准之一。
- 实机最小验收动作清单（前进/后退/左右转/避障）是否需要统一模板记录。

5. 许可证与归属策略
- `Tyler_1_Library` 与 AFMotor 均为第三方开源来源，建议补一份 `THIRD_PARTY_NOTICES.md` 记录来源与许可证。

## 4. 风险与建议

- 风险：当前构建依赖本机 Arduino IDE 1.x 工具链路径，跨机器复现依赖环境一致性。
- 建议：
1. 由项目负责人决定是否统一切到 `arduino-cli` 作为团队构建标准。
2. 在下轮任务书中给出“C-0.0.2 的单一目标”，避免把驱动改造与功能开发混在同一轮。
