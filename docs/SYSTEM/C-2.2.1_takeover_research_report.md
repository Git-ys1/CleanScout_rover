# C-2.x 接管研究报告

## 1. 当前阶段定位

当前项目已经从 `C-1.x` 的“UNO 本体主控 + 外设实验性堆叠”阶段，进入 `C-2.x` 的“树莓派上位接管 + 底层执行收口”阶段。

当前阶段口径：

- 下层执行仍是 `Arduino UNO + AFMotor`
- 树莓派是唯一上位控制中心
- `ROS Noetic + OpenClaw` 作为中上层能力来源
- 后续通过 `SSH` 接入树莓派板子进行维护和控制

## 2. 历史基线与当前实施边界

### 2.1 历史基线

`C-1.1.1` 代表的是历史硬件冻结事实：

- 主控：`Arduino UNO`
- 电机驱动：`AFMotor`
- 蓝牙：`HC-06`
- 舵机：`SG90`
- 超声测距：`HC-SR04`
- 麦克纳姆机械安装事实

这些事实仍保留为历史证据链，但不再等于当前实施主线。

### 2.2 当前实施边界

`C-2.x` 当前主线已经切换为：

- 树莓派上位控制中心
- `RPLIDAR A3`
- `MPU6050`
- 霍尔编码电机方向
- `ROS/Noetic/OpenClaw` 中上层

因此，不得再拿 `C-1.1.1` 的旧冻结直接覆盖当前实施决策。

## 3. 树莓派 Noetic 镜像技术结论

根据当前审计日志，树莓派镜像已确认：

- 系统：`Ubuntu 20.04.3 LTS`
- 架构：`arm64`
- ROS：`Noetic 1.15.11`
- 主工作空间：`/home/clbrobot/catkin_ws`
- 次级工作空间：`/home/clbrobot/goolge_ws`
- `SSH` 已开启且处于 `active`
- 当前网络环境通过 `wlan0` 工作
- `ROS_MASTER_URI / ROS_IP / ROS_HOSTNAME` 当前动态指向 `10.11.191.84`

当前镜像内已识别出完整的 `clbrobot`、`rplidar_ros`、`imu_filter_madgwick`、`imu_calib`、`robot_localization`、`move_base` 相关栈结构，说明：

- 树莓派 Noetic 不是空环境
- 它已经具备中上层接管的组织基础

## 4. 当前底层现实与限制

当前现实车底层仍是：

- `Tyler_1` 控制结构
- `UNO + AFMotor`
- 无现成编码器闭环
- 无现成 `rosserial` 对接口径

而当前 Noetic 镜像默认假设的底层能力包括：

- `rosserial_python` 串口底盘节点
- `raw_odom -> odom` 里程计链
- IMU relay、滤波与 EKF 融合
- 适配导航 / SLAM 的底层话题结构

这意味着：

- 当前镜像并不等于现实车可直接运行
- Noetic 上层默认假设的底层能力，明显强于现有 Tyler/AFMotor 现实车

## 5. 当前不做事项

本阶段明确不做：

- 不切 `STM32`
- 不直接改 `Tyler_1` API
- 不先改 `SLAM / navigation / EKF` 实现
- 不拍脑袋先做新硬件施工
- 不先推进树莓派侧大规模系统重构

## 6. 后续协作方式

后续涉及以下外部资料：

- 树莓派官方文档
- ROS / Noetic 兼容关系
- `RPLIDAR A3` 官方口径
- 树莓派 `GPIO / I2C / UART`
- 第三方模块官方规格

由项目负责人优先做网页检索与资料确认，再把结论回传给代码工程师。  
代码工程师当前重点是：

- 仓库接管
- 文档归档
- 镜像入口审查
- 边界切换说明

而不是在无确证情况下全网盲搜并直接改实现。
