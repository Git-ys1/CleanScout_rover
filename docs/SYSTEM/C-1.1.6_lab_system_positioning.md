# C-1.1.6 实验室清扫巡检系统统一定位

## 1. 文档目的

本文件用于固化当前项目的统一系统口径，避免后续继续把 `C`、`J`、`FSD` 三条线当成彼此割裂的单品项目推进。

当前统一口径为：

- 项目仓名 / 工程代号：`CleanScout_rover`
- 当前统一对内系统名称：`实验室清扫巡检系统`

## 2. 当前核心认识

当前项目不是单纯做：

- 一个会吸尘的小车
- 一个能动的机械臂
- 一个会飞的飞行器

而是在尝试构建一个面向实验室场景的综合运维系统。

这个系统要解决的不是消费级设备已经覆盖好的普通地面清扫，而是实验室环境中的以下问题：

- 近地面移动与基础清扫
- 日常巡检与异常发现
- 异常后的近场接近
- 局部试探、拨动、抓取或清理
- 结果记录、留痕与后续协同

## 3. 三条线的统一角色

### 3.1 C 线

`C` 线小车是当前核心地面终端。

当前系统角色：

- 近地面移动平台
- 基础清扫承载体
- 近距离巡检终端
- 后续多子系统协同的主承载体

当前阶段判断：

- 已具备主线原型地位
- 当前仓库的受控代码、构建链和硬件冻结都以 `C` 线为核心

### 3.2 J 线

`J` 线机械臂不是独立展示件，而是 `C` 线未来的近场操作与局部处置能力扩展。

理论角色包括：

- 小件拾取
- 遗落物清理
- 拨动 / 触发 / 试探性操作
- 局部危险小件的辅助处置

当前阶段判断：

- 仍处于资料归纳和并线可行性判断阶段
- 当前重点不是功能开发，而是确认它是否适合成为主线中的“操作子系统”

### 3.3 FSD 线

`FSD` 线飞行器不再按“单独做一个飞行器成品”的思路理解，而应被视为高位感知与协同扩展候选子系统。

可能角色包括：

- 高位巡检
- 异常复核
- 地空协同补视角
- 对地面终端难以覆盖区域的补充观察

当前阶段判断：

- 本仓当前未纳入 FSD 本地资料
- 本轮不做事实分析，只保留系统级角色接口

## 4. 当前最大的项目瓶颈

当前瓶颈已经不只是“功能有没有跑起来”，而是系统定位、创新叙事和价值闭环是否清楚。

必须回答的问题包括：

- 为什么要做这个系统，而不是只做一个小车
- 为什么不是直接买现成清扫机器人
- 为什么机械臂和飞行器不是额外堆料
- 为什么三条线必须汇到同一主线，而不是各做各的成品

## 5. 从现在起的统一输出要求

无论后续是 `J` 线还是 `FSD` 线，输出都不能只写：

- 我有什么硬件
- 我跑通了什么例程
- 我实现了什么单项功能

而必须回答：

- 我在实验室清扫巡检系统中的角色是什么
- 我解决的是哪一类主线问题
- 我与 `C` 线如何协同
- 我当前成熟到什么程度
- 我离真正并入主线还缺什么

## 6. C-1.1.6 阶段结论

- `C` 线继续作为当前唯一受控实现主线
- `J` 线本轮只做资料评估与系统适配判断
- `FSD` 线等待资料并入后再做事实分析

因此，`C-1.1.6` 的正确动作不是扩写新功能，而是先把系统口径和 `J` 线判断边界冻结下来。
