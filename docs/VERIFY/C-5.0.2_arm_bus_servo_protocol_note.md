# C-5.0.2 机械臂总线舵机协议核验记录

## 结论

这轮核验后，机械臂资料和当前仓库代码可以归成同一层语义：

- 资料包里的 `#000P1500T1000!` 是总线舵机 ASCII 文本协议。
- `P1500`、`T1000` 是协议字段，不是裸 PWM 波形本体。
- 当前仓库的机械臂官方 STM32 基线，既会解析这类文本命令，也会把结果落到本地 PWM 舵机输出和串口转发。
- 因此，`yh_pwm_text` 这个名字更准确的理解是“文本命令封装”，而不是“直接输出 PWM 波形”。

## 证据

| 来源 | 发现 | 含义 |
| --- | --- | --- |
| `docs/001-总线舵机资料/1.使用手册/附件1《总线舵机指令表》.docx` | 指令表以 `#000P1500T1000!`、`#000PID!`、`#000PMOD!`、`#000PRAD!` 等 ASCII 帧为主 | 这是总线舵机上层协议，不是二进制帧 |
| `docs/001-总线舵机资料/1.使用手册/附件2《总线舵机和PWM舵机区别》.docx` | 明确写出总线舵机通过数据/串口通信、带 ID、闭环反馈 | 证实“总线舵机”本身就是协议驱动设备 |
| `docs/001-总线舵机资料/4.源码例程/串口控制程序(32)/USER/main.c` | 说明里写了“统一总线口：TX3 RX3” | 这个示例工程确实把串口 3 作为总线入口 |
| `docs/001-总线舵机资料/4.源码例程/串口控制程序(32)/USER/app_uart.c` | `uart3_init(115200); /* 连接总线设备串口 */` | 代码注释直接把 UART3 当总线设备口 |
| `docs/001-总线舵机资料/4.源码例程/串口控制程序(32)/SRC/global/y_global.c` | `parse_action()` 先 `zx_uart_send_str()`，再解析 `#...P...T...!` | 文本命令既被转发，也被本地逻辑消费 |
| `firmware/mechanical_arm_official_baseline/User/Components/y_servo/y_servo.c` | `pwmServo_angle_set()`、TIM7、GPIO PWM 输出仍然存在 | 当前冻结基线不是“只发总线帧”的单一驱动 |
| `OrangePi/rk3588_ai/arm_tracking_demo/arm_driver.py` | 默认协议仍是 `yh_pwm_text`，并打包 `#000PxxxxTxxxx!` | 上位机侧也在使用同一层文本命令 |

## 解释

- 如果你把机械臂理解成“总线舵机”，那现在这套协议是对的。
- 如果你把它理解成“裸 PWM 舵机”，那会误读 `P1500` 的含义。
- 当前仓库更准确的说法是：**命令层使用总线舵机文本协议，执行层由 STM32 适配为本地 PWM 或串口转发**。
