# C-5.2.0 原厂 D435 纯 Python 感知抓取基线验证

## 参考代码学习结论

参考：`docs/00最新参考/01.树莓派源码/src/car_vision/car_vision/`。

采用的结构：

- `dynamic_space_grab.py` 的“目标稳定后锁深度，再进行手眼变换和抓取时序”。
- `transform.py` 的像素深度反投影与齐次矩阵链路。
- `arm_ik_sdk.py` / `Kinematics.py` 的 5DoF IK 与抓取动作语义。

没有迁移：

- ROS2 Node、topic、message_filters、CvBridge 和 JointState。
- 参考工程内写死的颜色阈值、距离阈值和实机偏移量。
- 参考工程正运动学会累计修改全局 DH 参数的实现。

## 新增实现

- `rknn_yolo_detector.py`：RKNN YOLO11 推理与目标选择适配器。
- `tools/d435_yolo_grasp.py`：D435 到抓取计划的纯 Python 运行入口。
- `tools/test_grasp_safety.py`：中心、深度、工作空间安全回归。
- `tools/test_official_kinematics.py`：官方 000--003 IK 与 `$KMS` 协议回归。
- `tools/test_geometry_frames.py`：`T_B_T * T_T_C * p_C` 坐标链与刚体矩阵校验。
- `grasp_state_machine.py`：增加中心门、深度稳定门、工作空间检查、全序列 IK 预检和真实动作等待。

## 2026-07-13 C-5.1.3 实机预抓结果

已验证：

- 固定重试姿态为 `[1380,1909,1900,620,1500,1500]`。
- D435 在该姿态稳定识别竖直 `bottle`，一次锁定置信度约 `0.79`、对齐深度约 `0.160 m`。
- 当时的目标坐标与 Python 参考 IK 能生成运动计划，但该计划错误地补造了 004；
  复核官方代码后确认 `$KMS` 实际只控制 000--003。
- 005 单轴执行 `P0600` 时确实向全开方向运动，读回约 `1112` 后发送了单轴停止。

未通过：

- 机械臂移动到计算的 PRE_GRASP 后，D435 相机直接顶到 bottle；该动作不能视为安全预抓点。
- 未执行完整 APPROACH/CLOSE/LIFT，也没有视觉证据证明抓起 bottle。
- 005 的 `P2400` 全闭方向和夹持力尚未完成有电实测。
- 电池耗尽后位置查询六轴均无回包，最后一次回退命令不能作为到位证据。

结论：当前像素/深度锁定有效，但失败版本使用了参考工程外参种子，并把错误的
末端参考点当成夹爪 TCP。该种子现已删除，未标定时使用全零占位并禁止生成抓取
计划。真实抓取离验收仍有明显距离。

D435 自带的 RGB/红外内部外参继续由 librealsense 用于 depth-to-color 对齐；
仍需实测的是厂家未知的机械安装外参 `T_tool_camera_color_optical`。硬件关系已冻结为
D435 位于 004 定子外壳正上方，TCP 位于 004 转子/005 侧，因此第一版固定
`004=1500`。
