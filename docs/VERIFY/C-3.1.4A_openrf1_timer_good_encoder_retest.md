# C-3.1.4A：OpenRF1 定时器主线回退与好编码器复测结论

## 结论

本轮已将 OpenRF1 运行层从 `feature/c-3.1.3-openrf1-tracking-port-soft-encoder` 回退到 `feature/c-3.0.6-openrf1-closed-loop-freeze` 的原生定时器主线，并完成好编码器电机复测。

最终结论明确如下：

- `C-3.0.6` 的原生 TIM 编码器主线软件成立。
- `CN1 -> TIM5 + PA0/PA1` 与 `CN3 -> TIM2 full remap + PA15/PB3` 在接入好编码器电机后，均恢复了非零计数。
- 此前 `CN1/CN3` 长期无反馈的首因，已从“定时器软件可疑”收敛为“原先测试用编码器电机/编码器链存在硬件问题”。
- `feature/c-3.1.3-openrf1-tracking-port-soft-encoder` 保留为坏编码器证伪与止损绕行证据，不再继续作为正式主线。

## 本轮操作

### 1. 回退锚点

- 当前工作分支：`feature/c-3.1.4a-openrf1-timer-good-encoder-retest`
- 回退源分支：`feature/c-3.0.6-openrf1-closed-loop-freeze`
- 保留证据分支：`feature/c-3.1.3-openrf1-tracking-port-soft-encoder`

### 2. 实际恢复范围

本轮实际恢复了两处，而不是只恢复 `_work_3`：

- `_local/openrf1_keil_work_3`
- `_local/openrf1_keil_work_2/Project/RVMDK（uv5）/BH-STM32.uvprojx`

这样做的原因是：当前 Keil 工程文件仍引用 `_work_3` 的源码；若只恢复 `_work_3` 而不恢复 `.uvprojx`，`csr_soft_encoder.c` 仍会被编译进镜像，无法真正回到 `C-3.0.6` 的纯定时器主线。

### 3. 编译与烧录

- Keil 编译结果：`0 Error(s), 0 Warning(s)`
- 输出镜像：`_local/openrf1_keil_work_2/Output/template.hex`
- 烧录方式：`STM32_Programmer_CLI`
- 成功连接参数：`port=SWD mode=UR reset=HWrst`

说明：

- `HotPlug + software reset` 下擦除失败。
- 切换为 `Under Reset + Hardware reset` 后烧录成功。

## 测试前提

### 1. 主线边界

本轮保留 `C-3.0.6` 已冻结的主线结构：

- `W,a,b,c,d`
- `M,<ch>,<pwm>`
- `E,<ch>`
- `D,<ch>`
- `STOP`
- 50Hz 控制循环
- 约 250ms `W` 看门狗
- 原生 TIM2/3/4/5 编码器读取

本轮明确不再使用：

- `C-3.1.3` 的循迹口软编码器
- `csr_soft_encoder.c/.h`
- `PC4/PC5/PB0/PC14` 作为 `CN1/CN3` 主线反馈来源

### 2. 硬件接线

- `CN2/CN4` 作为控制组，继续保留原生板载定时器通道
- `CN1/CN3` 本轮接入已验证为好的测试编码器电机

- `TIM2 full remap + PA15/PB3` 原生输入链在好编码器电机条件下成立。

## 归因收口

本轮归因固定为：

### 1. 已被排除的主因

- `C-3.0.6` 原生定时器主线软件整体错误
- `TIM5` 方案天然不可用
- `TIM2 full remap + PA15/PB3` 方案天然不可用
- 协议链、主循环、PI 框架导致 `CN1/CN3` 永久无反馈

### 2. 当前最强结论

此前 `CN1/CN3` 无反馈的阻塞首因，是原先测试链路中的编码器硬件问题，而不是定时器主线本身。

更直接地说：

- 在同一套 `C-3.0.6` 定时器软件下，控制组 `CN2/CN4` 一直有数。
- 本轮把 `CN1/CN3` 换成好编码器电机后，它们也恢复了计数。
- 因此，定时器主线软件已被复测通过。

## 对后续主线的影响

### 1. 主线选择

当前正式主线应回到 `C-3.0.6` 的原生定时器架构，而不是继续把 `C-3.1.3` 的软编码器绕行方案长成主线。

### 2. 证据链保留

以下分支与文档仍需保留：

- `feature/c-3.0.7-openrf1-cn1-cn3-encoder-debug`
- `feature/c-3.0.8-openrf1-cn1-cn3-encoder-recovery`
- `feature/c-3.1.0-openrf1-cn1-cn3-board-input-audit`
- `feature/c-3.1.3-openrf1-tracking-port-soft-encoder`

这些内容不再作为主线实现依据，但继续作为“坏编码器证伪与止损绕行”的复盘证据。

## 本轮最终结论

`C-3.0.6` 的 OpenRF1 原生定时器主线已通过好编码器复测。

从本轮起，OpenRF1 的下一阶段工作应建立在：

- 原生 TIM2/3/4/5 编码器链
- 已验证的 `W/M/E/D/STOP` 协议
- 已冻结的 50Hz 控制循环

之上继续推进，而不是继续把软编码器绕行方案作为正式底盘主线。
