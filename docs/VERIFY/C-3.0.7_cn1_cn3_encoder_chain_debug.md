# C-3.0.7：CN1/CN3 编码器链路专项排查记录

## 1. 本轮目标

本轮只排查 `CN1 / CN3` 编码器链路，不改 `W / M / E / STOP` 既有语义，不调 PI，不接树莓派，不重测开环方向真值。

当前基础判断：

- `CN2 / CN4` 已经有编码器计数，可用于闭环调试。
- `CN1 / CN3` 电机能输出，但 `ENC` 仍为 `0`，闭环没有反馈。
- 当前 Keil 工程只编译 `_work_3/User` 自写层和 CMSIS/FWlib，未编译旧 `tracking / ps2 / soft_i2c / y_encoder / y_motor / app_motor` 运行语义。

## 2. 本轮代码诊断改动

### 2.1 新增寄存器诊断命令

新增：

```text
R,<ch>
```

用法：

- `R,0`：输出全局 remap 状态，重点看 `AFIO->MAPR`。
- `R,1`：输出 `CN1 / TIM5` 寄存器状态。
- `R,3`：输出 `CN3 / TIM2` 寄存器状态。

返回：

```text
ACK:R
REG,<ch>,<MAPR_RAW>,<MAPR_EFFECTIVE>,<SMCR>,<CCMR1>,<CCER>,<CNT>
```

字段：

- `MAPR_RAW`：直接读回的 `AFIO->MAPR`
- `MAPR_EFFECTIVE`：自写层影子配置值，用于避免 STM32F1 `MAPR.SWJ_CFG` 读回不可靠导致误判
- `SMCR`：对应 `TIMx->SMCR`
- `CCMR1`：对应 `TIMx->CCMR1`
- `CCER`：对应 `TIMx->CCER`
- `CNT`：对应 `TIMx->CNT`

`R,0` 只用于全局 remap，因此 `SMCR/CCMR1/CCER/CNT` 为 `0`。

说明：

- 本轮发现 `AFIO->MAPR` 直接读回的 `SWJ_CFG` 会显示 `04000000`，但自写层实际写入的影子配置为 `02000000`。
- 因此判断 JTAG/SWD 口径时，以 `MAPR_EFFECTIVE` 为软件写入真值；`MAPR_RAW` 仅作为硬件读回参考。

### 2.2 编码器滤波改为通道宏

当前诊断默认：

```c
#define CSR_ENCODER_FILTER_CN1 0
#define CSR_ENCODER_FILTER_CN2 10
#define CSR_ENCODER_FILTER_CN3 0
#define CSR_ENCODER_FILTER_CN4 10
```

目的：

- `CN1 / CN3` 暂时降滤波，排除低速手拨或霍尔波形被滤波吞掉的可能。
- `CN2 / CN4` 保持既有可用口径，不作为本轮主改对象。

## 3. 当前源码对照结论

### 3.1 当前自写层映射

| 通道 | 编码器资源 | 驱动资源 |
| --- | --- | --- |
| `CN1` | `TIM5 + PA0/PA1` | `PA8 + TIM8_CH1(PC6)` |
| `CN2` | `TIM3 + PA6/PA7` | `PA11 + TIM8_CH2(PC7)` |
| `CN3` | `TIM2 full remap + PA15/PB3` | `PA12 + TIM8_CH3(PC8)` |
| `CN4` | `TIM4 + PB6/PB7` | `PC10 + TIM8_CH4(PC9)` |

### 3.2 与 vendor 例程对照

对照 `docs/STM32F103RCT6/03.源代码程序/9.控制霍尔编码器电机/User/Components/y_encoder/y_encoder.c`：

- vendor `ENCODER_C_GetCounter()` 读取 `TIM5->CNT`，对应当前 `CN1`。
- vendor `ENCODER_D_GetCounter()` 读取 `TIM2->CNT`，对应当前 `CN3`。
- vendor `Encoder_Init_TIM5()` 也是 `PA0/PA1 + TIM5`。
- vendor `Encoder_Init_TIM2()` 也是 `GPIO_FullRemap_TIM2 + PA15/PB3 + TIM2`。

当前可见代码映射仍未发现明显缺项。

## 4. 必测流程

### 4.1 全局 remap

```text
STOP
R,0
```

验收：

- `MAPR` 的 `SWJ_CFG` 应为 `010`：禁 JTAG，保 SWD。
- `MAPR` 的 `TIM2_REMAP` 应为 full remap。

如果这里不成立，优先修初始化顺序或确认烧录镜像。

### 4.2 CN1

手拨或短时驱动时执行：

```text
STOP
D,1
R,1
M,1,700
D,1
R,1
E,1
STOP
```

判定：

- `phaseA/phaseB` 不变：优先查 CN1 编码器硬件链、供电、地线、线序、接插件。
- `phaseA/phaseB` 变但 `CNT` 不变：优先查 `TIM5` 初始化、`PA0/PA1` 输入模式。
- `CNT` 变但 `ENC` 不变：优先查 `csr_encoder_read_and_reset(CN1)` 读取链。

### 4.3 CN3

手拨或短时驱动时执行：

```text
STOP
D,3
R,3
M,3,700
D,3
R,3
E,3
STOP
```

判定：

- `phaseA/phaseB` 不变：优先查 CN3 编码器硬件链，或 PA15/PB3 未释放。
- `phaseA/phaseB` 变但 `CNT` 不变：优先查 `SWJ_CFG / TIM2 full remap / TIM2 encoder mode / ICFilter`。
- `CNT` 变但 `ENC` 不变：优先查 `csr_encoder_read_and_reset(CN3)` 读取链。

## 7. 本机执行状态

本轮代码侧已完成并实机验证：

- Keil 编译通过：`0 Error(s), 0 Warning(s)`。
- STLink 烧录通过，`template.hex` 已写入并校验成功。
- `COM14 @ 115200` 串口验证通过，`STOP / R / D / E` 均有 `ACK` 和返回帧。

结论：

- 当前固件已在板上更新，协议诊断链有效。
- `CN2 / CN4` 在同一固件下可以计数，说明串口协议、通用编码器读取链、`csr_encoder_read_and_reset()` 并非全局失效。
- `CN1 / CN3` 的直接 GPIO 相位没有变化，且对应 TIM 计数为 `0`，当前证据优先指向这两路编码器 A/B 信号没有进入 MCU 输入脚，或对应连接/供电/线序/板级专有路径问题。
- 若客服仍判断为软件问题，下一刀应围绕 `CN1 PA0/PA1`、`CN3 PA15/PB3` 的实际引脚电平来源继续查，而不是再改 `W` 协议或 PI 参数。

## 8. 本轮归因

当前归因：`A 类：硬件链路或板级输入路径问题`。

依据：

- CN1/CN3 的 `phaseA/phaseB` 在短时驱动后仍固定为 `1/1`。
- CN1/CN3 的 `TIMx->CNT` 与 `ENC` 均为 `0`。
- CN2/CN4 的 `phase/timer/ENC` 在同一测试方法下均有效。
- CN1/CN3 的寄存器配置显示编码器模式已启用：`SMCR=0003`、`CCMR1=0101`、`CCER=0001`。

下一步建议：

- 先用万用表或示波器确认 CN1 编码器 A/B 是否真实到达 `PA0/PA1`。
- 再确认 CN3 编码器 A/B 是否真实到达 `PA15/PB3`，并确认板级线路没有把这两脚转接到其他位置。
- 如硬件确认有波形，再回到软件侧查 GPIO 输入模式或 TIM 输入复用；当前证据不足以继续通过 PI 或协议修改解决。
