# C-3.0.4B：OpenRF1 底层方向真值已打通，后续从协议桥接继续推进

## 1. 本轮结论

本轮通过的不是“整车闭环已完成”，而是更关键的一步：

- OpenRF1 底层四路电机驱动已经在最小测试入口下打通
- 板端 `UART2 -> CH340 -> COM14` 调试链已打通
- 先前方向判断混乱的首因，已经定位为 vendor 代码语义错位，而不是板子底层资源失效

一句话冻结：

> OpenRF1 当前已经具备继续做协议桥接和运行时收口的底层基础；后续不再回到“底层完全不可控”的判断。

## 2. 已确认的技术事实

### 2.1 当前最可信的本地工作副本

- `_local/openrf1_keil_work_2/`

说明：

- 这份工作副本来自 vendor 例程 9 的新拷贝
- 本轮只在这份副本上做最小测试入口，不再继续使用先前已经被多轮试验污染的旧工作区

### 2.2 当前最可信的测试入口

- `_local/openrf1_keil_work_2/User/main.c`

当前该入口承担：

- 最小电机正反测试
- `UART2` 调试输出
- `CN1~CN4` 物理接口顺序验证
- 编码器计数辅助观测

### 2.3 当前底层通道真值

已冻结的 vendor 底层映射：

- `MOTOR_A / ENCODER_A -> CN2`
- `MOTOR_B / ENCODER_B -> CN4`
- `MOTOR_C / ENCODER_C -> CN1`
- `MOTOR_D / ENCODER_D -> CN3`

这意味着：

- vendor 的 `A/B/C/D` 代码通道名，不等于板上 `CN1/CN2/CN3/CN4` 顺序
- 后续所有底层方向与协议调试，都必须优先以 `CN1~CN4` 记录

### 2.4 当前串口调试结论

已打通：

- `UART2`
- `CH340`
- `COM14`

当前最小调试输出已验证可用：

- `CSR_RF1_READY`
- `PHASE,...`
- `CNT,...`

## 3. 首因定位

### 3.1 不是板子底层完全坏了

本轮用户已确认能够看到：

- 四轮轮流前后转测试

因此可以排除：

- “OpenRF1 四路底层资源失效”
- “板子完全无法驱动四轮”
- “这套底层完全没有继续推进价值”

### 3.2 真正首因是 vendor 语义混层

当前已定位的主要混乱来源包括：

- `app_motor.c` 中目标与输出存在额外符号翻转
- `y_timer.c` 每 `20ms` 强行调 `app_motor_run()`
- `y_motor.c` 的 `SPEED_PidCtlX()` 为带静态残留的增量式 PD
- vendor `A/B/C/D` 通道顺序与板上 `CN1~CN4` 物理顺序不一致

所以先前出现的：

- 一会儿像 `B/C` 反
- 一会儿像 `A/C` 反
- `+/-` 看起来不干净

其根因不是单一接线问题，而是：

> vendor 闭环入口并不适合作为“底层方向真值测试入口”。

## 4. 本轮冻结口径

从这一轮开始，冻结以下口径：

### 4.1 冻结为真值

- OpenRF1 仍是当前底盘下位机新主线
- `_local/openrf1_keil_work_2` 是当前最可信本地 bring-up 副本
- `main.c` 最小测试入口是当前底层真值入口
- `COM14` 串口观测已可用

### 4.2 明确不再采用

- 不再用 vendor `app_motor_run()` 判定底层方向真值
- 不再把 vendor `A/B/C/D` 当成板子 `CN1/CN2/CN3/CN4`
- 不再回到“板子底层完全不可控”的口径

## 5. 后续顺序

当前最合理的后续顺序固定为：

1. 保持 `_local/openrf1_keil_work_2` 作为底层真值入口
2. 在这套已打通的底层之上继续做极简协议桥接
3. 再进入运行时收口与四轮协调控制
4. 最后再谈树莓派适配

本轮明确不做：

- 直接恢复 vendor 杂项应用层
- 直接接 ROS
- 直接接机械臂 / OpenMV
