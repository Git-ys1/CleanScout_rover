# C-3.0.6：OpenRF1 底层链路复盘与 CN1/CN3 编码器问题冻结

## 1. 本轮冻结结论

`C-3.0.6` 已经完成以下阶段性成果：

- OpenRF1 自写运行层可编译、烧录、串口通信。
- `USART2 / COM14` 可见 `CSR_RF1_READY`、`ACK:M/E/D/STOP/W`。
- `W,a,b,c,d` 已从“单次动作”改为“目标锁存 + 50Hz 控制循环 + 250ms 超时停车”。
- `CN2 / CN4` 编码器链路有效，闭环可观测；用户实测已能向前转，但低速仍有顿挫。
- `CN1 / CN3` 电机输出可用，但编码器回传仍为 `0`，因此闭环无法调节，只会按开环输出冲出。

本轮不能宣称“四轮闭环完成”。当前可冻结为：

- 两轮闭环阶段性成立：`CN2 / CN4`
- 四路开环驱动已具备可控基础
- `CN1 / CN3` 阻塞点收敛为编码器链路无反馈

## 2. 当前运行层边界

当前施工基线是：

```text
_local/openrf1_keil_work_3/User
```

运行层原则：

- 不回退到 UNO。
- 不恢复 vendor `app_motor_run()` 为主线。
- 不再重测 `C-3.0.5` 已打通的开环方向真值。
- `csr_motor_drv.c` 保持四路独立 H 桥输出。
- `main.c` 负责目标锁存、PI 计算、超时停车、遥测调度。
- `csr_encoder_drv.c` 负责 TIM2/3/4/5 编码器计数与诊断快照。

## 3. OpenRF1 物理通道与 vendor 命名差异

vendor 例程中的 `MOTOR_A/B/C/D` 不是物理 `CN1/CN2/CN3/CN4` 顺序。当前复盘结论如下：

| 物理接口 | 当前自写通道 | vendor 电机函数 | vendor 编码器函数 | 编码器资源 | 驱动资源 |
| --- | --- | --- | --- | --- | --- |
| `CN1` | `CN1` | `MOTOR_C_SetSpeed` | `ENCODER_C_GetCounter` | `TIM5 + PA0/PA1` | `PA8 + TIM8_CH1(PC6)` |
| `CN2` | `CN2` | `MOTOR_A_SetSpeed` | `ENCODER_A_GetCounter` | `TIM3 + PA6/PA7` | `PA11 + TIM8_CH2(PC7)` |
| `CN3` | `CN3` | `MOTOR_D_SetSpeed` | `ENCODER_D_GetCounter` | `TIM2 full remap + PA15/PB3` | `PA12 + TIM8_CH3(PC8)` |
| `CN4` | `CN4` | `MOTOR_B_SetSpeed` | `ENCODER_B_GetCounter` | `TIM4 + PB6/PB7` | `PC10 + TIM8_CH4(PC9)` |

这解释了前期混乱的根因：

- vendor 的 `A/B/C/D` 是应用层命名，不等于物理 CN 顺序。
- 当前自写层统一只认 `CN1~CN4`，不再把 vendor `A/B/C/D` 当控制语义。
- 后续树莓派或上层协议必须对接 `CN1~CN4` 真值表，不能直接照搬 vendor A/B/C/D。

## 4. vendor 例程中的方向叠加

vendor `app_motor.c` 中存在多层符号修正：

- `Wheel_A`、`Wheel_C` 的目标值在 PID 输入前被取负。
- `MOTOR_A/B/C/D_SetSpeed()` 输出前又被统一取负。
- vendor `MOTOR_A/B/C/D` 本身又映射到不同物理 CN。

因此，vendor 例程能跑不代表其应用层方向语义可以直接继承。当前项目已经选择：

- 底层 H 桥输出由 `csr_motor_drv.c` 自写并按物理 CN 固化。
- 闭环层只通过 `g_csr_motor_dir_sign[]` 和 `g_csr_encoder_dir_sign[]` 修正控制符号。
- 不再把 vendor 多层取负逻辑搬入主线。

## 5. 当前初始化链路复盘

当前自写层已经做了以下关键初始化：

- `main()` 最早期禁用 JTAG、保留 SWD：

```c
RCC_APB2PeriphClockCmd(RCC_APB2Periph_AFIO, ENABLE);
GPIO_PinRemapConfig(GPIO_Remap_SWJ_JTAGDisable, ENABLE);
```

- `CN3 / TIM2` 使用 full remap：

```c
GPIO_PinRemapConfig(GPIO_FullRemap_TIM2, ENABLE);
```

- `CN1 / TIM5` 使用 `PA0/PA1`。
- `CN2 / TIM3` 使用 `PA6/PA7`，当前已实测有计数。
- `CN4 / TIM4` 使用 `PB6/PB7`，当前已实测有计数。

对照 vendor `y_encoder.c` 后，当前可见的软件映射没有发现明显缺项。仍需保留一种可能：vendor 工程里存在分散在其他初始化文件中的 GPIO/AFIO/时钟副作用，当前自写层尚未完全复刻。

## 6. CN1/CN3 当前问题定位

当前现象：

| 通道 | 电机输出 | 编码器回传 | 闭环状态 |
| --- | --- | --- | --- |
| `CN1` | 可转 | `0` | 无反馈，无法闭环 |
| `CN2` | 可转 | 有计数 | 可进入闭环调试 |
| `CN3` | 可转 | `0` | 无反馈，无法闭环 |
| `CN4` | 可转 | 有计数 | 可进入闭环调试 |

这说明问题不是统一串口协议问题，也不是所有编码器初始化都失败。问题已经收敛为 `CN1 / CN3` 的局部编码器链路。

可能原因按优先级排列：

1. `CN1 / CN3` 编码器 A/B 相线、供电、地线、端子或焊点未真正到 MCU。
2. `CN1 / CN3` 对应电机能转，但编码器线序或插头脚位与当前假设不一致。
3. `TIM5 / TIM2 full remap` 仍缺少某个 vendor 工程中隐式执行的初始化副作用。
4. `CN3` 使用 `PA15/PB3`，虽然 JTAG 已禁用，但仍需用 `D,3` 观察 GPIO 原始相位是否变化，不能只看 `ENC`。
5. `CN1` 使用 `PA0/PA1`，理论上不受 JTAG 影响；若 `D,1` 的 `phaseA/phaseB` 不变，更偏向硬件链路或引脚假设错误。

## 7. 必做诊断命令

当前固件已经提供：

```text
D,<ch>
```

返回：

```text
DBG,<ch>,<phaseA>,<phaseB>,<timerCount>,<count>,<delta>
```

下一轮 CN1/CN3 排查必须按这个顺序做：

1. `STOP`
2. 手拨 `CN1` 对应轮，同时重复发送 `D,1`
3. 执行短时 `M,1,700`，同时重复发送 `D,1`
4. 对 `CN3` 重复 `D,3`

判断规则：

- `phaseA/phaseB` 始终不变：优先查编码器硬件链路、线序、供电、接插件。
- `phaseA/phaseB` 会变但 `timerCount` 不变：优先查 TIM remap、输入模式、滤波、时钟或 GPIO 模式。
- `timerCount` 会变但 `count/delta` 不变：查 `csr_encoder_read_and_reset()`、计数读取宽度、符号回填。

## 8. CN2/CN4 闭环现状

用户最新实测结论：

- `CN2 / CN4` 当前能向前转。
- 低速仍然一顿一顿，但已经没有明显回转。

当前控制层已加入：

- 速度低通
- PWM 斜率限制
- 低速 bringup 阶段禁止反向刹车
- 目标为 0 时清积分
- 超时停车清积分

这说明 `CN2 / CN4` 的闭环链路已经具备继续调参价值。后续应在这两路上继续优化：

- 起步补偿
- 低速最小保持 PWM
- 每轮独立 `kp/ki`
- 速度尺度校准

但这些优化不应掩盖 `CN1 / CN3` 无反馈问题。

## 9. 发布结论

`C-3.0.6` 建议发布为：

- OpenRF1 自写运行层闭环协议基线
- `W` 持续控制协议已打通
- `CN2/CN4` 两轮闭环阶段性成立
- `CN1/CN3` 编码器链路问题冻结，提交项目负责人继续分析硬件链路和板级初始化差异

下一轮不应继续扩大到 ROS、树莓派、运动学或四轮导航。建议单独开轮：

```text
C-3.0.7：OpenRF1 CN1/CN3 编码器链路专项排查
```

验收标准应只看：

- `D,1 / D,3` 的相位是否跳变
- `TIM5 / TIM2` 是否计数
- `ENC,1 / ENC,3` 是否恢复有效 delta
