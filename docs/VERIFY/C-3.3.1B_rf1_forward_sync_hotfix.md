# C-3.3.1B RF1 前进同步热修记录

## 背景

现场反馈：闭环仍不能稳定正常前进，右前轮 `CN4/RF` 体感慢半拍。

本轮只处理 RF1 固件闭环与执行层，不改树莓派、ROS、建图、导航，不改 `W/M/E/D/STOP` 协议。

## 检查结论

1. 四轮不同 PI 不是同步问题的优先解法。
2. 当前执行层有强 deadband/起步补偿，如果再给四轮不同 PI，容易把机械差异、死区补偿和滤波延迟混在一起，导致不同步更难定位。
3. `CN4/RF` 慢半拍不是单纯 PID 不够，而是前进高档时右侧执行补偿不足；应把差异收敛到电机驱动层，不应继续靠四轮不同 PI 硬调。
4. `CSR_VEL_FILTER_ALPHA=0.65` 反馈偏钝，会放大“慢半拍”的观感。
5. `W` 目标变化时若不清理对应通道积分/输出，会让上一条命令的闭环状态残留到下一条命令。

## 代码改动

- `main.c`
  - PI 回到四轮统一：`Kp=1200 / Ki=80 / Kd=0`。
  - 新增目标变化检测：目标符号变化、从 0 到非 0、或目标幅值变化超过 `0.08m/s` 时，清该通道积分、上次误差和输出 PWM。
  - 保留每轮独立 PI 状态，但不再使用四轮不同 PI 作为同步手段。
- `csr_board_map.h`
  - `CSR_VEL_FILTER_ALPHA` 从 `0.65` 降到 `0.50`，减少测速反馈延迟。
- `csr_motor_drv.c`
  - 保留方向相关 deadband 补偿。
  - 前进实际使用的右侧 raw 方向增强：
    - `CN3/RR raw negative floor = 1100`
    - `CN4/RF raw positive floor = 1200`
  - 未把右侧所有方向一概增强，避免右转/横移负向过冲进一步恶化。

## 复测记录

测试方式：COM14，连续发送 `W`，每组结束立即 `STOP`。

| 命令 | VEL 最后一帧 | PWM 最后一帧 | 结论 |
| --- | --- | --- | --- |
| `W,0.15,0.15,0.15,0.15` | `0.160,0.165,0.159,0.140` | `0,0,0,11` | 前进中速基本同步，CN4 接近目标但略低 |
| `W,0.25,0.25,0.25,0.25` | `0.261,0.271,0.228,0.237` | `0,0,-27,15` | 前进高档右侧明显改善，CN4 不再严重慢半拍 |
| `W,-0.15,-0.15,0.15,0.15` | `-0.139,-0.139,0.170,0.170` | `0,0,0,13` | 左转可用，右侧略快 |
| `W,0.15,0.15,-0.15,-0.15` | `0.157,0.160,-0.235,-0.199` | `0,0,0,0` | 右转可用，但右侧负向偏快，后续需单独收敛 |
| `W,0.15,-0.15,-0.15,0.15` | `0.167,-0.158,-0.232,0.144` | `-4,-3,0,0` | 左移符号成立，右后负向偏快 |
| `W,-0.15,0.15,0.15,-0.15` | `-0.141,0.166,0.151,-0.210` | `12,0,0,0` | 右移符号成立，右前负向偏快 |

## 当前判断

- 前进问题已从“右前明显慢半拍”改善为“右侧略低但可跟随”。
- 四轮不同 PI 不作为本轮正式方向；当前采用统一 PI + 驱动层方向补偿。
- 仍待后续收敛的是右侧负向方向在转向/横移中偏快，尤其 `CN3/RR`、`CN4/RF` 的 raw positive/negative 补偿需要拆开继续细化。

## 下一步

1. 现场优先落地测试前进 `0.15` 和 `0.25`，确认物理偏航是否改善。
2. 若前进仍偏右/偏左，只继续调前进实际用到的 raw 方向补偿，不改四轮 PI。
3. 下一轮再单独处理右转/横移中右侧负向过快问题。
