# C-3.1.3 OpenRF1 循迹口软件编码器接管验证

## 本轮结论

C-3.1.3 不再继续消耗原板 `CN1=TIM5 PA0/PA1`、`CN3=TIM2 full remap PA15/PB3` 编码器链路，改用板载 4 路循迹口作为 `CN1/CN3` 的软件正交编码器输入。

本轮不改上层协议与控制框架：`W/M/E/D/STOP`、50Hz 控制循环、PI 框架和 `csr_encoder_read_and_reset()` 对外语义保持不变。

## 循迹口真值修正

按新原理图，4 路循迹口冻结为：

| 循迹口 | MCU 引脚 | 本轮用途 |
|---|---|---|
| `X1` | `PC4` | `CN1_A` 软件编码器输入 |
| `X2` | `PC5` | `CN1_B` 软件编码器输入 |
| `X3` | `PB0` | `CN3_A` 软件编码器输入 |
| `X4` | `PC14` | `CN3_B` 软件编码器输入 |

旧速查表中把其中一路写为 `PB10` 是错误记录。本轮已按新原理图修正为 `PB0`。

## 飞线接线冻结

| 原编码器信号 | 飞线到循迹口 | EXTI 线 |
|---|---|---|
| `CN1_A` | `PC4(X1)` | `EXTI4` |
| `CN1_B` | `PC5(X2)` | `EXTI5` |
| `CN3_A` | `PB0(X3)` | `EXTI0` |
| `CN3_B` | `PC14(X4)` | `EXTI14` |

编码器供电与地线仍走原 6Pin 编码器电机口：`CNx.2=+5V`、`CNx.5=GND`。本轮只绕开原板 A/B 相输入，不改电机动力线，不改编码器供电线。

## 软件后端

当前编码器后端冻结为：

| 通道 | 编码器后端 | 输入来源 |
|---|---|---|
| `CN1` | 软件 EXTI 正交解码 | `PC4/PC5` |
| `CN2` | 硬件 TIM 编码器 | `TIM3 PA6/PA7` |
| `CN3` | 软件 EXTI 正交解码 | `PB0/PC14` |
| `CN4` | 硬件 TIM 编码器 | `TIM4 PB6/PB7` |

`csr_encoder_drv` 统一分发：`CN1/CN3` 调用 `csr_soft_encoder`，`CN2/CN4` 继续读取硬件定时器。上层闭环不需要知道底层来源差异。

## 软件编码器实现

`csr_soft_encoder.c/.h` 提供两套独立状态机：

- `CN1`：`PC4/PC5`
- `CN3`：`PB0/PC14`

GPIO 配置：

- 全部使用 `GPIO_Mode_IPU`
- EXTI 全部双边沿触发
- NVIC 启用 `EXTI0_IRQn`、`EXTI4_IRQn`、`EXTI9_5_IRQn`、`EXTI15_10_IRQn`

中断分发：

| ISR | 作用 |
|---|---|
| `EXTI4_IRQHandler` | 更新 `CN1` 状态机 |
| `EXTI9_5_IRQHandler` | `EXTI5` 更新 `CN1` 状态机 |
| `EXTI0_IRQHandler` | 更新 `CN3` 状态机 |
| `EXTI15_10_IRQHandler` | `EXTI14` 更新 `CN3` 状态机 |

状态机采用 16 状态查表法。ISR 内只做读 A/B、查表、更新计数、清 pending bit，不做串口、速度换算、PI 或延时。

## 旧诊断入口处理

上一阶段 `Y/Q` 是原 CN1/CN3 引脚的 EXTI 诊断入口。C-3.1.3 中它们已从运行固件移除，避免与新 `EXTI0/4/5/14` 软件编码器中断冲突。

## 当前验收口径

本轮通过标准：

- `OpenRF1_开发速查.md` 中循迹口真值已修正为 `X1=PC4 / X2=PC5 / X3=PB0 / X4=PC14`。
- 固件中 `CN1/CN3` 已切到软件编码器后端。
- `CN2/CN4` 仍保持硬件编码器后端。
- 飞线完成后，`D,1/E,1` 与 `D,3/E,3` 应能看到相位变化或非零 delta。

若 `PC4/PC5/PB0/PC14` 飞线后仍无相位变化，下一刀不再调软件，直接查循迹口端子、电平、飞线、编码器 A/B 输出和供电地。

## 实测记录

- 编译：Keil 已通过，`csr_soft_encoder.c` 已进入工程编译，结果为 `0 Error(s), 0 Warning(s)`。
- 烧录：首次由命令行 STLink 烧录中途掉线；随后由人工完成烧录。
- 串口：`COM14 @ 115200` 可正常收发，`STOP / D / E / M` 均可收到 ACK 与回传。
- 飞线手拨：12 秒轮询 `D,1/E,1/D,3/E,3` 期间，`CN1/CN3` 均未出现相位变化或非零 delta。

### 静止与手拨轮询

```text
STOP -> ACK:STOP
D,1 -> ACK:D + DBG,1,1,1,0,0,0
E,1 -> ACK:E + ENC,1,0,0
D,3 -> ACK:D + DBG,3,1,1,0,0,0
E,3 -> ACK:E + ENC,3,0,0
```

12 秒重复轮询结果一致：`CN1/CN3` 相位固定为 `1/1`，`ENC` 固定为 `0/0`。

### 短时带动验证

```text
M,1,700  -> ACK:M + PWM,700,0,0,0
STOP     -> ACK:STOP
D,1      -> ACK:D + DBG,1,1,1,0,0,0
E,1      -> ACK:E + ENC,1,0,0

M,1,-700 -> ACK:M + PWM,-700,0,0,0
STOP     -> ACK:STOP
D,1      -> ACK:D + DBG,1,1,1,0,0,0
E,1      -> ACK:E + ENC,1,0,0

M,3,700  -> ACK:M + PWM,0,0,700,0
STOP     -> ACK:STOP
D,3      -> ACK:D + DBG,3,1,1,0,0,0
E,3      -> ACK:E + ENC,3,0,0

M,3,-700 -> ACK:M + PWM,0,0,-700,0
STOP     -> ACK:STOP
D,3      -> ACK:D + DBG,3,1,1,0,0,0
E,3      -> ACK:E + ENC,3,0,0
```

判定：`CN1/CN3` 的电机 PWM 下发有效，但循迹口软件编码器输入仍未看到边沿。若飞线已经按本轮方案接好，下一步应查飞线端子、电平、编码器 A/B 输出与供电地，而不是继续改协议或 PI。

### CN2/CN4 回归

```text
M,2,700 -> ACK:M + PWM,0,700,0,0
STOP
D,2     -> ACK:D + DBG,2,0,1,65487,5248,94
E,2     -> ACK:E + ENC,2,6347,25

M,4,700 -> ACK:M + PWM,0,0,0,700
STOP
D,4     -> ACK:D + DBG,4,1,1,28,4955,71
E,4     -> ACK:E + ENC,4,5576,19
```

判定：`CN2/CN4` 硬件编码器后端仍有非零计数，新 `EXTI0/4/5/14` 软件编码器没有破坏已成立的 `TIM3/TIM4` 两路。

### 复测记录：CN1/CN3 已出现非零累计值

在用户重新处理连接后，重新执行非运动回归：

```text
STOP -> ACK:STOP
D,1 -> ACK:D + DBG,1,1,1,0,-14,0
E,1 -> ACK:E + ENC,1,-14,0
D,3 -> ACK:D + DBG,3,1,1,0,-8,0
E,3 -> ACK:E + ENC,3,-8,0
```

判定：`CN1/CN3` 软件编码器路径不再是彻底静默，说明 `PC4/PC5/PB0/PC14` 这组输入至少已经出现过有效边沿并被状态机累计。

随后做短时带动验证：

```text
M,1,500  -> ACK:M + PWM,500,0,0,0
STOP
D,1      -> ACK:D + DBG,1,1,1,0,-14,0
E,1      -> ACK:E + ENC,1,-14,0

M,1,-500 -> ACK:M + PWM,-500,0,0,0
STOP
D,1      -> ACK:D + DBG,1,1,1,0,-14,0
E,1      -> ACK:E + ENC,1,-14,0

M,3,500  -> ACK:M + PWM,0,0,500,0
STOP
D,3      -> ACK:D + DBG,3,1,1,0,-8,0
E,3      -> ACK:E + ENC,3,-8,0

M,3,-500 -> ACK:M + PWM,0,0,-500,0
STOP
D,3      -> ACK:D + DBG,3,1,1,0,-8,0
E,3      -> ACK:E + ENC,3,-8,0
```

当前新结论：软件编码器入口已能累计边沿，但 `M,1/3 ±500` 短时带动没有新增 delta。下一步优先确认飞线的 `CN1_A/B`、`CN3_A/B` 是否确实来自对应正在带动的电机编码器，而不是继续改 `W/M/E/D/STOP` 或 PI。
