# C-3.1.3 OpenRF1 循迹口软件编码器接管验证

## 本轮结论

C-3.1.3 不再继续消耗原板 `CN1=TIM5 PA0/PA1`、`CN3=TIM2 full remap PA15/PB3` 编码器链路，改用板载 4 路循迹口作为 `CN1/CN3` 的软件正交编码器输入。

本轮不改上层协议与控制框架：`W/M/E/D/STOP`、50Hz 控制循环、PI 框架和 `csr_encoder_read_and_reset()` 对外语义保持不变。

## 循迹口真值修正

按新原理图，4 路循迹口冻结为：

| 循迹口 | MCU 引脚 | 本轮用途 |
|---|---|---|
| `X1` | `PC4` | `CN1_A` 软件编码器输入 |
| `X2` | `PC5` | `CN1_B` 软件编码器输入 |
| `X3` | `PB0` | `CN3_A` 软件编码器输入 |
| `X4` | `PC14` | `CN3_B` 软件编码器输入 |

旧速查表中把其中一路写为 `PB10` 是错误记录。本轮已按新原理图修正为 `PB0`。

## 飞线接线冻结

| 原编码器信号 | 飞线到循迹口 | EXTI 线 |
|---|---|---|
| `CN1_A` | `PC4(X1)` | `EXTI4` |
| `CN1_B` | `PC5(X2)` | `EXTI5` |
| `CN3_A` | `PB0(X3)` | `EXTI0` |
| `CN3_B` | `PC14(X4)` | `EXTI14` |

编码器供电与地线仍走原 6Pin 编码器电机口：`CNx.2=+5V`、`CNx.5=GND`。本轮只绕开原板 A/B 相输入，不改电机动力线，不改编码器供电线。

## 软件后端

当前编码器后端冻结为：

| 通道 | 编码器后端 | 输入来源 |
|---|---|---|
| `CN1` | 软件 EXTI 正交解码 | `PC4/PC5` |
| `CN2` | 硬件 TIM 编码器 | `TIM3 PA6/PA7` |
| `CN3` | 软件 EXTI 正交解码 | `PB0/PC14` |
| `CN4` | 硬件 TIM 编码器 | `TIM4 PB6/PB7` |

`csr_encoder_drv` 统一分发：`CN1/CN3` 调用 `csr_soft_encoder`，`CN2/CN4` 继续读取硬件定时器。上层闭环不需要知道底层来源差异。

## 软件编码器实现

`csr_soft_encoder.c/.h` 提供两套独立状态机：

- `CN1`：`PC4/PC5`
- `CN3`：`PB0/PC14`

GPIO 配置：

- 全部使用 `GPIO_Mode_IPU`
- EXTI 全部双边沿触发
- NVIC 启用 `EXTI0_IRQn`、`EXTI4_IRQn`、`EXTI9_5_IRQn`、`EXTI15_10_IRQn`

中断分发：

| ISR | 作用 |
|---|---|
| `EXTI4_IRQHandler` | 更新 `CN1` 状态机 |
| `EXTI9_5_IRQHandler` | `EXTI5` 更新 `CN1` 状态机 |
| `EXTI0_IRQHandler` | 更新 `CN3` 状态机 |
| `EXTI15_10_IRQHandler` | `EXTI14` 更新 `CN3` 状态机 |

状态机采用 16 状态查表法。ISR 内只做读 A/B、查表、更新计数、清 pending bit，不做串口、速度换算、PI 或延时。

## 旧诊断入口处理

上一阶段 `Y/Q` 是原 CN1/CN3 引脚的 EXTI 诊断入口。C-3.1.3 中它们已从运行固件移除，避免与新 `EXTI0/4/5/14` 软件编码器中断冲突。

## 当前验收口径

本轮通过标准：

- `OpenRF1_开发速查.md` 中循迹口真值已修正为 `X1=PC4 / X2=PC5 / X3=PB0 / X4=PC14`。
- 固件中 `CN1/CN3` 已切到软件编码器后端。
- `CN2/CN4` 仍保持硬件编码器后端。
- 飞线完成后，`D,1/E,1` 与 `D,3/E,3` 应能看到相位变化或非零 delta。

若 `PC4/PC5/PB0/PC14` 飞线后仍无相位变化，下一刀不再调软件，直接查循迹口端子、电平、飞线、编码器 A/B 输出和供电地。

## 实测记录

- 编译：Keil 已通过，`csr_soft_encoder.c` 已进入工程编译，结果为 `0 Error(s), 0 Warning(s)`。
- 烧录：首次由命令行 STLink 烧录中途掉线；随后由人工完成烧录。
- 串口：`COM14 @ 115200` 可正常收发，`STOP / D / E / M` 均可收到 ACK 与回传。
- 飞线手拨：12 秒轮询 `D,1/E,1/D,3/E,3` 期间，`CN1/CN3` 均未出现相位变化或非零 delta。

### 等效替代测试：将 CN3 接到 CN4 板载通路

测试目的：验证问题是否来自 `CN3` 编码器本体，而不是原 `CN3` 板载输入路径。用户本轮说明：已取消 `CN3` 飞线连接，并将 `CN3` 接到 `CN4` 的板载引脚；因此本轮通过测试原 `CN4` 通道，等效观察 `CN3` 在原本“已知可用”的 `CN4` 编码器通路上是否能恢复反馈。

M,4,500  -> ACK:M + PWM,0,0,0,500
STOP     -> ACK:STOP
D,4      -> ACK:D + DBG,4,1,1,0,5643,0
E,4      -> ACK:E + ENC,4,5643,0

M,4,-500 -> ACK:M + PWM,0,0,0,-500
STOP     -> ACK:STOP
D,4      -> ACK:D + DBG,4,1,1,0,5643,0
E,4      -> ACK:E + ENC,4,5643,0

STOP     -> ACK:STOP
D,4      -> ACK:D + DBG,4,1,1,0,5643,0
E,4      -> ACK:E + ENC,4,5643,0

本轮判定：

- `CN4` 通道本身仍能正常收命令并输出 PWM。
- 但在这次“CN3 接到 CN4 通路”的等效测试中，`CN4` 的累计计数保持 `5643` 不变，`lastDelta` 始终为 `0`。
- 也就是说，在本轮测试条件下，换到原本已知可用的 `CN4` 编码器通路后，仍未观察到新增反馈。

结论边界：

- 如果你本轮确实是把 `CN3` 对应的编码器总成接到了 `CN4` 板载编码器通路，且测试时那一路电机实际发生了转动，那么这个结果更偏向：`CN3` 编码器本体、线束或供电侧存在问题，而不是原 `CN3` 的板载 `TIM2/PA15/PB3` 输入路径问题。
- 如果本轮只改了编码器信号接法，但并没有让对应总成在 `M,4` 测试时实际转动，那么这个结果只能说明“当前 `CN4` 路径没有收到新增边沿”，还不能单独证明 `CN3` 编码器物理损坏。

### 等效替代测试：将 CN1 接到 CN4 板载通路

- 这说明至少在本轮测试条件下，把 `CN1` 改接到原本已知可用的 `CN4` 板载编码器通路后，依然没有观察到新的编码器边沿进入系统。

结论边界：

- 如果你本轮确实已经把 `CN1` 对应编码器总成接到了 `CN4` 板载编码器通路，且测试时该总成对应电机确实实际发生了转动，那么该结果更偏向：`CN1` 编码器本体、线束或供电侧存在问题，而不是原 `CN1` 的板载输入路径问题。
- 如果本轮测试时对应总成并未实际转动，那么这次结果只能说明“当前 `CN4` 路径没有收到新增边沿”，还不能单独证明 `CN1` 编码器物理损坏。

### 严格对照复测：板载复位后，实验室现成 6 线编码电机接入 CN4

测试前提：

- 本轮明确执行了板载复位，避免沿用上一轮未复位状态下的历史累计值。
- 使用一台实验室现成、往年作品上的 6 线编码电机，直接接入板载 `CN4` 编码器通路。
- 目标是验证：在“固件不变、板载通路不变、仅更换编码器总成”的条件下，`CN4` 是否能重新获得稳定新增计数。

### 阶段性归档结论

本轮严格对照测试后的收口结论如下：

1. `CN4` 板载编码器输入通路、当前串口协议链、当前 `D/E` 读取链并未整体失效。
2. 在“板载复位 + 更换为实验室现成编码电机”的前提下，`CN4` 在 `+700` 与 `-700` 两个方向下都能稳定产生新增计数。
3. 结合前两轮 `CN3 -> CN4`、`CN1 -> CN4` 等效测试都未观察到新增反馈，当前最合理的解释已经从“OpenRF1 软件整体有问题”收口为：
   - 原 `CN1` / `CN3` 对应编码器总成本体问题；
   - 或其线束 / 编码器供电侧问题；
   - 至少不再优先怀疑当前 `CN4` 板载输入通路。
4. 这也反向说明：此前针对 `TIM2/TIM5` 与外部中断软件编码器的多轮排查，不足以单独证明“定时器一定坏了”；更大的问题很可能是原始编码器硬件链本身。
5. 本轮结题口径限定为：
   - 外部中断软件编码器止损路线，已经完成阶段性任务并找到真相；
   - 原 `CN1/CN3` 无反馈问题当前收口到编码器硬件侧；
   - 原板 `TIM2/TIM5` 定时器编码器路线尚未在更换新电机后重新上板实测，因此本轮不把“定时器坏了”写成最终结论。
