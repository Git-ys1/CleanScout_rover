# C-2.x 硬件边界更新说明

## 1. 文档定位

本文件用于明确：

- `C-1.1.1` 的历史冻结事实
- `C-2.x` 的当前实施边界

本文件不是对 `C-1.1.1` 的口头覆盖，而是新的版本化边界说明。

## 2. 当前下层执行边界

当前下层执行仍为：

- `Arduino UNO + AFMotor`

当前明确不做：

- 不切 `STM32`
- 不直接替换底层控制 API

## 3. 当前上位控制边界

当前树莓派是唯一上位控制中心。

后续控制分层口径：

- 下层：`UNO + AFMotor`
- 上位：树莓派
- 中上层：`ROS Noetic + OpenClaw`

## 4. 当前新增主线硬件

当前主线新增硬件趋势冻结为：

- 雷达：`RPLIDAR A3 -> Raspberry Pi USB`
- IMU：`MPU6050 -> Raspberry Pi GPIO / I2C`
- 轮速：后续走霍尔编码电机方向

这些是当前实施主线，不再由旧 `HC-06 / SG90 / HC-SR04` 代表系统扩展方向。

## 5. 旧外设的当前定位

以下模块仍作为历史底层事实存在，但不再是当前主线实现重点：

- `HC-06`
- `SG90`
- `HC-SR04`

当前建议口径：

- 它们可以从底层主线中撤出
- 若后续需要保留部分能力，由树莓派侧重新统一接管，不再作为当前底层主线优先实现项

## 6. 当前 Noetic 栈与现实车的关系

当前 `Noetic` 上层软件栈可作为接管对象，但不能直接照抄到底层现实车。

原因：

- `Noetic` 侧默认假设存在更完整的里程计、IMU、`rosserial`、导航接口
- 现实车当前仍是 Tyler/AFMotor 路线，能力边界明显更窄

因此当前实施原则是：

- 上层镜像可接管
- 下层能力必须重新映射到现实车边界
- 不得把镜像中的默认底层能力直接当成现实车已经具备

## 7. 历史冻结 vs 当前实施边界

### 7.1 历史冻结事实

`C-1.1.1` 保留的历史事实包括：

- `UNO`
- `AFMotor`
- `SG90`
- `HC-06`
- `HC-SR04`
- 麦克纳姆机械安装

### 7.2 当前实施主线

`C-2.x` 当前实施主线包括：

- 树莓派上位接管
- `RPLIDAR A3`
- `MPU6050`
- 霍尔编码电机
- `ROS/Noetic/OpenClaw`

结论：

- 旧冻结事实保留
- 但旧冻结不再等于当前主线实施边界
