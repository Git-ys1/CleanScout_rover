# 01. 本项目是什么？为了实现什么而存在？项目结构是？智能 agent 同事还有谁？

CleanScout_rover 最初并不是一个“完整系统”，而是以小车为主的原型项目；在长期推进后，它被重新定义为：面向实验室场景的清扫、巡检、近场操作与后续具身智能协同的原型平台。

它真正想实现的，不再只是“一个会动的小车”，而是：

1. 让地面平台承担实验室近地面移动、基础清扫、巡检与任务承载；
2. 让机械臂承担近场抓取、分拣、局部处置；
3. 让无人机承担高位巡检、异常复核、补充视角获取；
4. 让后续上层智能系统（树莓派 / OpenClaw / ROS2 / 小程序）承担自然语言任务理解、远程运维、状态回传、技能编排和长期记忆。

所以项目最终目标，不是“几个分立作品的拼盘”，而是一个实验室具身智能巡检作业平台原型。

### 当前真实项目结构

- **C 线**：车体 / 小车本体，系统核心地面终端；
- **J 线**：机械臂 / 近场执行子系统；
- **FSD 线**：飞行器 / 高位感知与异常复核子系统；
- **新增上层系统层**：Pi / OpenClaw / ROS2 / 小程序接口。

### 智能 agent 同事

- 项目总监（用户）：最高决策者；
- 网页端项目负责人（本岗位）：长期承担 C 线负责人、三线联合负责人、某阶段也承担 J 线管理角色；
- IDE 端代码工程师（Codex）：代码施工、结构实现、验证和落仓；
- 深度研究员：网页深搜、竞赛与模块研究；
- FSD 线项目负责人：独立推进飞行器线，并需与主线口径汇合；
- OpenClaw / 树莓派责任端：后期参与 Pi4B 与 OpenClaw 平台部署、交接。

一句话结论：这个项目应被理解为“以 C 为核心终端、J 为近场执行、FSD 为高位感知、ROS2 与 OpenClaw 为系统层”的实验室具身智能原型平台。

# 02. 本项目从你接手时是什么状态？（仅 C 线、J 线）作为 3 线联合负责人和 C 线项目负责人，你认为目前的项目实况是什么？

## 我接手时的状态（只看 C / J）

最初的总纲只有 C / J 双线：
- C 线可推进；
- J 线先挂起，只保留版本位与接口意识；
- C 线已有 Arduino 小车原型；
- J 线硬件、代码、通信都还不明朗。

所以我真正接手时，现实是：
- **C 线**：已有能跑的小车原型，基于 Arduino 和 Tyler_1_Library，需要做工程化、受控化；
- **J 线**：还是未知卖家平台，不能直接施工，只能先评估和预留。

## C 线后来经历了什么

1. **Tyler 基线受控化**：把 Tyler_1_Library 纳管成受控工程基线，形成 libraries / sketches / tools 结构和标准编译入口；
2. **小功能与硬件冻结**：风机继电器、蓝牙逻辑、单超声波自动模式、引脚冻结等；
3. **C ↔ J 通讯 PoC**：UNO / F411 / OpenMV 串口联调，形成状态机和系统接口意识；
4. **太阳能供电与比赛应急**：太阳能供电链路、Type-C 诱骗、分级供电口径与 PPT 口径；
5. **战略反思**：确认旧 Arduino 穿透小车路线已经到头，必须转向更先进的平台。

## J 线后来经历了什么

1. 起点完全未知；
2. 做了卖家资料纳管、最小硬件形态评估、资源冲突分析；
3. 与 C 联调时经历了大量失败：`cj_link` 缺失、`WAIT_PICK_WINDOW` 死锁、夹爪原语不成立等；
4. 最终靠重新寻找 vendor 可用基线，形成了 actuator baseline、vendor vision grasp baseline，并在 `C-1.2.7` 确认了稳定的主抓取基线：黄色抓取、三色分拣、抬臂、松爪、回位都已实测。

## 当前项目实况（站在三线联合负责人视角）

- **C 线**：仍然是系统核心载体，但旧架构已经过时，只适合作为 rolling prototype，必须迁移到 ROS2 和新的上层智能体系；
- **J 线**：当前是三线里最成熟、最值钱的实物闭环基线，但还只是“彩色目标抓取”，还没有技能化、服务化；
- **FSD 线**：有较强技术栈雏形，但定位鲁棒性和任务级接口没有收住；
- **总体**：项目已不是“做不出来”，而是“旧架构还能跑，真正先进的新架构才刚起步”。

一句话结论：现在最大的任务不是继续加小功能，而是系统重构。

# 03. 当前 3 线的技术栈、存在的问题、以及刚才商量的将来发展方向（最重要）

## A. C 线技术栈

### 当前技术栈
- Arduino UNO
- AFMotor / Adafruit Motor Shield v1（L293D）
- Tyler_1_Library 受控基线
- `.ino / C / C++`
- SG90、HC-SR04、继电器风机、HC-06（部分阶段可舍弃）
- 做过 F411 / F103 通讯过渡板实验
- 做过太阳能模块供电试验
- 现已开始引入 Raspberry Pi 4B / OpenClaw / ROS2 思路

### 当前问题
1. 底层主控落后：UNO + AFMotor v1 扩展性差、资源紧、没有标准机器人系统接口；
2. 运动控制仍是经验式时序控制：重量、轮子、车速一变，转弯时长重新失效；
3. 通讯链路只是 PoC：没有真正演化成稳定的系统消息接口；
4. 仍是“原型设备”，不是 ROS2 移动机器人。

### 未来方向
1. 不再以 Arduino 为未来主平台；
2. 把 C 线纳入 ROS / ROS2：建图、定位、导航、状态回传；
3. 形成三层分工：
   - 上层：Pi4B / OpenClaw / 小程序 / Web 状态页；
   - 中层：ROS2、topics / services / actions、地图与导航；
   - 下层：专门的实时控制板（更可能是 STM32 / ESP32 等）。
4. OpenClaw 定位为上层智能入口，不承担高实时底盘闭环。

## B. J 线技术栈

### 当前技术栈
- OpenMV
- STM32（卖家板侧）
- Python / OpenMV 脚本
- 卖家深度二开平台（不是标准官方 OpenMV 口径）
- actuator baseline 已冻结
- vendor vision grasp baseline 已冻结
- `C-1.2.7` 已确认三色抓取、分拣、抬臂、回位主基线
- OpenMV 摄像头负责识别与控制；ESP32S3 WiFi 摄像头负责额外图传

### 当前问题
1. 稳定的是“彩色大块目标”，不是通用目标；
2. 还不是“技能化动作平台”，更像闭环演示模块；
3. 与 C 线的系统接口没有真正收口。

### 未来方向
1. 保留现有三色抓取主基线，绝不回退；
2. 进一步做黑色电容、金色螺丝 / 螺帽等真实实验室目标；
3. 把动作抽成标准原语：pan / tilt / approach / contact / grasp / hold / release / reset；
4. 逐步形成可被 ROS2 / OpenClaw 调用的 action / tool 单元，例如 `pick_target`、`sort_target`、`inspect_closeup`、`magnetic_pick`。

## C. FSD 线技术栈

### 当前技术栈
- 飞行器整机平台
- PX4 飞控
- ROS / 分布式 ROS
- MAVROS
- VINS-Fusion
- RealSense 双红外 / 深度相机
- 飞控 IMU
- 室内视觉惯性定位链路

### 当前问题
1. 外参与里程计鲁棒性没有收敛；
2. 还没有与 C 线形成任务级接口；
3. 室内飞行安全闭环没完成。

### 未来方向
1. 先收定位，不先堆功能；
2. 未来作为高位巡检与异常复核子系统；
3. 后续纳入统一上层智能与 ROS2 编排。

## D. 三线统一的新方向（最关键）

未来真正的主线，不是继续修旧作品，而是：

**把当前三个原型，重构成一个统一的具身智能系统。**

分层如下：
- 上层智能入口：Raspberry Pi 4B、OpenClaw、skills / tools、memory、远程运维、小程序接口；
- 中间机器人层：ROS / ROS2、建图、导航、topics / services / actions、状态总线；
- 下层实时执行层：底盘控制板、机械臂执行板、飞控 / 实时控制器、电机 / 传感器 / 急停。

未来任务语义应从“左转、开风机、闭爪”升级成：
- 去实验台区域巡检
- 前往异常点复核
- 拾取地面金属小件
- 返回指定位置
- 上报当前系统状态

一句话结论：三线下一阶段最重要的任务，不是继续堆小功能，而是围绕 ROS2 + OpenClaw + 新的底层分工，把三线收成一个真正的系统。

# 05. 开发流程、开发纪律，以及通过此方法取得的进展

## 一、开发纪律的核心思想

本项目一直强调：

**不是先干再补文档，而是先固化代号、版本、基线与纪律，再进行最小范围、可追踪、可回退的工程施工。**

## 二、平时开发时的标准流程

1. **先确认状态**：当前分支、工作树是否干净、当前基线是谁；
2. **先做 checkpoint**：未 checkpoint，不上板；
3. **尽量用新分支推进**：feature / hotfix / exp / verify 分工明确；
4. **一轮只解决一类问题**：防止多个目标混改；
5. **先 build，再上板；先验证，再宣告完成**；
6. **结果必须留痕**：任务号、基线 commit、分支名、checkpoint、结果 commit、构建结果、实机验证、是否可合并。

## 三、最重要的 Git 纪律

- 未 checkpoint，不上板；
- 当前树是脏的，先处理干净再开工；
- hotfix 只做最小修复，不顺手改架构；
- 回退优先 restore / revert，不乱 reset --hard；
- 方向错了，回到已验证基线重开。

## 四、通过这套方法，项目已经取得的关键进展

### 1. C 线从“别人的小车代码”变成了“我们的受控基线”
没有这套纪律，Tyler 代码只会变成一堆副本。正是通过工程化纳管，才形成了 `libraries/`、`sketches/`、`tools/`、Arduino CLI 编译链和 README 体系。

### 2. 硬件冻结与引脚冻结逐渐变成文档资产
风机继电器、A2 控制、HC-SR04、AFMotor 接线、板间串口、太阳能模块等，之所以后面还能回顾和追溯，靠的就是任务书、文档、验证记录与计划入库。

### 3. C ↔ J 通讯 PoC 虽未成为最终主线，但逼出了“系统接口意识”
这条线虽然未变成最终生产路线，但第一次让项目形成了协议、状态机、接口边界这些系统级概念。

### 4. J 线是靠“纪律”而不是“运气”翻盘的
如果没有“坏分支别继续堆补丁，回到可验证基线重开”，J 线不会从夹爪热修失败线里真正走出来，更不会有后来的主抓取基线。

### 5. 太阳能供电与比赛口径能快速收敛，也是靠同样的方法
最后能形成“小车主载体太阳能供电 + 机械臂独立供电 + 分级供电”口径，以及对应 PPT 文案，靠的也是“问题 → 研究 → 冻结 → 文案”这套工作流。

### 6. Pi / OpenClaw 路线能重新接到主线，也是因为已有结构化推进习惯
树莓派、OpenClaw、串口首配、代理、安装、SSH 等这些事虽然中间有失误，但最后能快速归位，是因为项目已经形成“新方向也必须先收敛成结构化方案”的习惯。

## 五、真实教训

### 1. 执行期必须更像一线指挥
当项目负责人没有直接给“决策 + 步骤 + 命令”时，效率会明显下降。树莓派备份和串口首配阶段就是典型反例。

### 2. 旧坏基线上的“再试一试”非常危险
J 线已经证明：继续在坏线里救火，只会原地踏步。

### 3. 文档本身就是工程资产
很多最关键的东西并不在代码里，而在计划文档、验证文档、评估文档、交接书、PPT 口径里。

## 六、给下一任负责人的工作法建议

1. 任何新任务先问四个问题：当前基线是谁？这轮只解决哪类问题？改坏后怎么退？哪个文档要同步更新？  
2. 对 C 线，优先系统重构，不优先功能缝合。  
3. 对 J 线，优先守住主抓取基线，不回旧热修线。  
4. 对 FSD 线，优先收定位，不先吹协同。  
5. 对 OpenClaw / ROS2 线，优先分层，不要混成新一轮“大杂烩”。

一句话结论：本项目到今天还能继续走下去，不是因为每次都选对了技术，而是因为靠版本、基线、分支、任务书、文档和验证纪律，把“走错的路”也变成了能回头、能重开的资产。
