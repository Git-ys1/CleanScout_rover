## 4. 版本号制度（强制执行）

本项目每轮进程都拥有独立版本号。

### 4.1 版本前缀

目前仅有两个前缀：

- **C**：代表车体 / 小车（che）
- **J**：代表机械臂（jixiebi）

示例：

- `C-0.0.1`
- `C-1.1.3`
- `J-0.3.1`

### 4.2 独立计数原则

- `C` 与 `J` 的后三位数字 **互相独立维护**
- 车体版本推进，不影响机械臂版本编号
- 机械臂版本推进，不影响车体版本编号

即：

- `C-1.2.4` 与 `J-0.1.2` 可以同时存在
- 两条线可以并行推进，但各自版本历史必须独立、可追踪

### 4.3 三层数字含义

版本后三位数字的含义如下：

#### 第 1 位：一级版本（大版本）

- 通常 `0` 代表准备阶段 / 立项前准备 / 环境搭建 / 初始整理
- `1、2、3……` 代表后续大阶段版本
- **只有在出现重大进展后**，才允许升级一级版本
- 是否升级一级版本，由 **项目总监决定**

#### 第 2 位：二级版本（小版本）

- 用于表示某个大版本下的阶段性推进
- 一般在 **某部分问题已结束或某阶段子目标完成后** 才升级
- 是否升级二级版本，由 **项目总监决定**

#### 第 3 位：三级版本（最小版本）

- 用于表示最细颗粒度的轮次推进
- **默认每轮会话结束后递进**
- 用于记录一次有效工作闭环

### 4.4 特殊后缀字母规则

若某轮会话存在以下情况，则可在第三级版本后追加字母后缀：

- 本轮无实际性进展
- 发生意外事项
- 代码工程师施工中断
- 需要项目总监与项目负责人先对话裁决
- 出现阻塞、待确认、待补资源等状态
- 其他不适合直接进入正常下一最小版本的情况

后缀字母按顺序追加，例如：

- `C-0.0.1B`
- `C-0.0.1C`
- `J-0.3.2D`

说明：

- 是否使用字母后缀，由 **项目总监决定**
- 字母后缀用于标记“同一轮级别下的异常 / 中断 / 非实质推进分支”
- 字母后缀不替代正常版本推进，仅作为状态标记

### 4.5 版本使用建议

- 所有任务书、提交记录、分支名、测试记录、验收记录应尽量绑定版本号
- 所有重要现象应能回溯到某个明确版本
- 任何成员都必须能回答：
  - 当前属于哪个子系统版本？
  - 当前从哪个基线版本开始？
  - 本轮目标是什么？
  - 本轮结束后是否进入下一个版本？

## 8. Git 纪律（强制执行）

以下纪律为本项目必须遵守的 Git 施工规则，已基于既有 `git_workflow` 经验整理并项目化适配。

### 8.1 核心目标

所有 Git 操作必须服务于以下四件事：

1. 能明确当前基线是什么
2. 能明确这轮改了什么
3. 改坏后能快速回退
4. 实机现象能和某个提交一一对应

### 8.2 施工前必须 checkpoint

**硬规则：**

- 未 checkpoint，不上板
- 未 checkpoint，不做高风险修改
- 未 checkpoint，不允许“先试一试再说”

**标准操作：**

```powershell
git status -sb
git add .
git commit --allow-empty -m "checkpoint: before C-x.x.x"
git push
```
**说明：**

- 即使没有代码改动，也允许使用 --allow-empty
- checkpoint 的目的不是保存内容，而是保存“从哪里开始施工”的锚点
- J 线同理，将 C-x.x.x 替换为 J-x.x.x

### 8.3 实机前后都必须有提交点

**硬规则：**

- 实机前至少有 checkpoint
- 实机通过后必须有结果提交
- 未提交，不得宣称“本轮完成”

**标准结果提交：**

```powershell
git status -sb
git add .
git commit -m "C-x.x.x: <本轮内容摘要>"
git push
```

**若是 J 线，则改为：**

```powershell
git commit -m "J-x.x.x: <本轮内容摘要>"
```

### 8.4 不在脏工作树上继续叠改

若当前工作树已经是脏的，必须先处理干净，再进入下一轮。

**原因：**

- 无法判断是哪部分引入回归
- 热修与实验会混在一起
- 回退会变脏、变慢、变不准

**处理方式只有两种：**

- 有价值：先提交 checkpoint
- 无价值：直接恢复

**示例：**

- `git restore .`
- 或单文件：
- `git restore App\app.c`

### 8.5 Hotfix 只做最小修复

**硬规则：**

- hotfix 只允许最小范围改动
- hotfix 不允许顺手改架构
- hotfix 不允许顺手清理无关代码
- hotfix 不允许把多个问题混修

**标准思路：**

- 回到 last-known-good
- 新开 hotfix 分支
- 只改真正与当前问题相关的模块
- 先恢复基线，再修最小问题

### 8.6 回退优先 restore / revert，不滥用 reset --hard

**场景 A：未提交改坏**

```powershell
git restore .
```

**场景 B：已提交且已 push 到共享分支**

```powershell
git revert HEAD
git push
```

**场景 C：本地实验分支，确认无人依赖，直接硬退**

```powershell
git reset --hard HEAD~1
```

**说明：**

- 共享分支优先 revert
- reset --hard 只适用于本地未共享实验
- 不得在共享主线或协作分支乱改历史

## 9. 推荐分支模型（项目化命名）

### 9.1 main

**用途：**

- 只放稳定可交付版本
- 只接受经过验证的合并

**禁忌：**

- 不在 main 上直接做高风险实验
- 不在 main 上直接做 bring-up 试错

### 9.2 feature/...

**用途：**

- 新功能开发
- 新模块接入
- 新路径实现

**建议命名：**

- `feature/c-0.0.1-init`
- `feature/c-1.1-line-follow`
- `feature/j-0.0.1-openmv-bringup`

### 9.3 hotfix/...

**用途：**

- 从稳定点出发做最小修复
- 恢复已被改坏功能

**建议命名：**

- `hotfix/c-1.1-motor-dir`
- `hotfix/c-1.2-sensor-debounce`
- `hotfix/j-0.3-uart-fix`

### 9.4 exp/...

**用途：**

- 高风险实验
- 先验证思路，不直接合主线

**说明：**

- 失败实验本身也是证据链
- 实验分支建议保留，不轻易删除

### 9.5 verify/...

**用途：**

- 纯回溯验证
- 纯烧录验证
- 验证历史点是否真为 last-known-good

## 10. 每轮施工标准流程

### 10.1 开工前

**查看状态：**

```powershell
git status -sb
```

**若当前在主线，先同步：**

```powershell
git checkout main
git pull
```

**做 checkpoint：**

```powershell
git add .
git commit --allow-empty -m "checkpoint: before C-x.x.x"
git push
```

**新建分支：**

```powershell
git checkout -b feature/c-x.x.x
git push -u origin feature/c-x.x.x
```

**若是热修，从稳定点起分支：**

```powershell
git fetch origin
git checkout <good_commit_hash>
git checkout -b hotfix/<name>
git push -u origin hotfix/<name>
```

### 10.2 施工中

**原则：**

- 一轮只解决一类问题
- 小步修改，小步构建
- 先 build，再上板
- 回退后必须重新 build

**禁止事项：**

- 不把多个目标混到同一轮
- 不边改边猜、不留锚点
- 不在热修中做结构性重写
- 不因“顺手”连带改动无关模块

### 10.3 施工后

- 先构建
- 再实机验证
- 验证通过后提交结果
- 必要时打 tag

**结果提交：**

```powershell
git add .
git commit -m "C-x.x.x: <summary>"
git push
```

**打 tag 示例：**

```powershell
git tag -a milestone/c-1.1-stable -m "C-1.1.x stable"
git push origin milestone/c-1.1-stable
```

- J 线同理替换前缀。

## 12. 回溯与重开策略

一旦方向错了，不要在脏状态上继续补丁，优先回到已验证基线重新开工。

**标准流程：**

- 找到 last-known-good commit
- 从那个点新开分支
- 不在当前坏分支上继续叠补丁

**示例：**

```powershell
git checkout <bad_branch>
git pull
git checkout <good_commit_hash>
git checkout -b hotfix/<name>
git push -u origin hotfix/<name>
```

**收益：**

- 可快速证明回归来源
- 可避免坏状态污染判断
- 可使 hotfix 保持最小改动面

## 13. 合并纪律

修完后不得直接把未经验证代码塞入 main。

**推荐顺序：**

- 在功能分支 / 热修分支上完成并验证
- 合回当前主开发分支或实验主线
- 打里程碑 tag
- 再合入 main

**示例：**

```powershell
git checkout exp/c-mainline
git merge --no-ff hotfix/c-1.1-motor-dir
git push

git tag -a milestone/c-1.1-stable -m "C-1.1.x stable"
git push origin milestone/c-1.1-stable

git checkout main
git pull
git merge --no-ff exp/c-mainline
git push
```

**说明：**

- `--no-ff` 用于保留合并节点，便于后期回溯
- 实验分支与失败分支原则上保留，不随意删除
