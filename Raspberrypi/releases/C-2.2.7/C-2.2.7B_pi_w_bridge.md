# C-2.2.7B Pi W Bridge

## 文档定位

- 责任方：树莓派端代码工程师
- 目标：把树莓派端从旧的 `CMD,vx,vy,wz` 逻辑改为当前有效的 `W,w1,w2,w3,w4` 协议桥

## 当前判断

当前失败根因不是 `UNO` 没升级，而是树莓派端仍停在旧的速度层协议：

```text
CMD,vx,vy,wz
```

而 `UNO` 当前有效协议已经冻结为：

```text
W,w1,w2,w3,w4
```

所以本轮树莓派侧必须先对齐 `W` 协议，而不是继续要求 `UNO` 接 `CMD`。

## 当前改动

### 1. active bridge 改为 `wheel_bridge.py`

当前桥接节点负责：

1. 持久连接 `/dev/csr_uno`
2. 等待 `CSR_UNO_READY`
3. 以 `20Hz` 持续发送：

```text
W,w1,w2,w3,w4
```

4. 读取：

```text
ACK:W
ENC,...
PID,...
ERR:...
```

### 2. 第二阶段骨架 `cmdvel_to_wheels.py`

该脚本存在，但默认不是当前主入口。

它只负责第二阶段能力预留：

1. 订阅 `/cmd_vel`
2. 结合 `D_EFF / WB / TW / CPR_X1_EST` 做 `cmd_vel -> W` 转换
3. 默认不开启自动控制

## 当前发布口径

### 当前不再发送

```text
CMD,vx,vy,wz
```

### 当前改为发送

```text
W,w1,w2,w3,w4
```

### 当前通过标准

以以下链路为准：

1. 收到 `CSR_UNO_READY`
2. 成功发送 `W,...`
3. 收到 `ACK:W`
4. 持续收到 `ENC,...`
5. 持续收到 `PID,...`

没有 `ACK:W`，不能称为树莓派侧协议对接成功。

## 当前边界

1. 当前不恢复 `CMD,vx,vy,wz`
2. 当前不把 `cmd_vel_to_wheels.py` 包装成默认入口
3. 当前不接 odom / EKF / SLAM / 导航
4. 当前只把树莓派端职责收敛为：把 `W` 协议桥稳定打通

## 发布意义

这轮不是把树莓派侧升级成完整速度前端，而是先把双方统一到同一层级的协议上：

```text
Pi 先对齐 W
然后再做 cmd_vel -> W
最后才谈 odom / EKF / 导航
```
