# C-3.2.1 RF1 ROS 启动链修复 + rosbridge 联调基线实测记录

## 文档定位

- 任务：`C-3.2.1`
- 平台：树莓派端
- 目标：建立 RF1 专用稳定 ROS 启动链，并满足 V 线本地 web 联调合同

## 本轮目标回顾

本轮只做两件事：

1. 建立 RF1 专用的稳定 ROS 启动链
2. 补上 `rosbridge_websocket`，满足本地联调合同

本轮不做：

1. 不恢复旧 Arduino `rosserial`
2. 不做公网穿透
3. 不正式恢复导航参数整定

## 本轮现场关键事实

### 1. OpenRF1 与雷达的 USB 口映射

现场确认：

```text
雷达 -> /dev/ttyUSB0
OpenRF1 -> /dev/ttyUSB1
```

对应 `by-id`：

```text
/dev/serial/by-id/usb-Silicon_Labs_CP2102_... -> ../../ttyUSB0
/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0 -> ../../ttyUSB1
```

### 2. 雷达旧链并不在当前工作区

本轮排查确认：

```text

/home/clbrobot/catkin_ws/devel/lib/rplidar_ros/rplidarNode
```

说明以前雷达长期依赖旧 `~/catkin_ws`，而不是当前 `CleanScout_rover/Raspberrypi/catkin_ws`。

### 3. 当前工作区已迁入雷达源码

本轮已将 `rplidar_ros` 源码迁入当前工作区：

```text
Raspberrypi/catkin_ws/src/rplidar_ros
```

### 4. 当前运行策略

当前策略为：

1. 当前工作区源码优先
2. 旧 `~/catkin_ws/devel` 只作为已编译 `rplidarNode` 运行时来源
3. 不再继续使用假的 `/scan` stub 作为当前有效链路

## 本轮关键排障记录

### 1. `use_cleanscout_pi.sh` 原先硬依赖缺失的 `devel/setup.bash`

本轮已改为：

1. 有 `devel/setup.bash` 时正常 source
2. 缺失时回退到 `ROS_PACKAGE_PATH` 模式

### 2. 新增脚本初始缺少执行权限

本轮已对以下脚本补执行权限：

1. `rf1_serial_bridge.py`
2. `cmdvel_to_rf1.py`
3. `rf1_vel_to_odom.py`
4. `mpu6050_node.py`
5. `openrf1_min_bridge_smoketest.py`

### 3. 接回旧雷达运行时后，曾一度导致 topic 变少

根因不是工作区被删，而是：

1. source 旧 `~/catkin_ws/devel/setup.bash` 后
2. `ROS_PACKAGE_PATH` 被旧工作区覆盖
3. 当前 `csrpi_base_bridge` 与 `mpu6050_i2c_bridge` 一度不在前面
4. 导致 RF1 / IMU 节点无法被解析

本轮已修正为：

```text
当前 Raspberrypi/catkin_ws/src 优先
旧 ~/catkin_ws/src 仅作为雷达运行时补充
```


本轮目的是满足合同 topic，并非完整 odom/EKF 恢复。

### 3. 当前 IMU 为标准 `/imu/data` 直出链

不是旧滤波全链恢复版。
