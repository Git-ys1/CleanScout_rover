## 当前架构

当前双链路并存：

### 1. 本地/LAN 联调链

```text
前端 -> backend -> rosbridge -> ROS -> OpenRF1
```

### 2. 新后端 edge-relay 链

```text
前端 -> backend <-WSS-> edge-relay(Pi) -> ROS -> OpenRF1
```

关键原则保持不变：

1. 前端永远只找 backend
2. backend 不直接轮询 ROS
3. `edge-relay` 是新增 transport，不替换现有 rosbridge

## 与 V 线联调后，当前确认修正过的问题

### 1. 不是 token 错

本轮中期已一度误判 token/path/profile，后续通过 V 线日志纠正。

真实情况是：

1. backend 能接受合法 hello
2. 主要问题在 Pi 侧连接生命周期管理

### 2. 已修正 hello_ack 前不应启动 telemetry/heartbeat

Pi 侧当前逻辑改为：

1. 先建连
2. 第一帧立即发 hello
3. 必须等 `hello_ack.accepted=true`
4. 之后才开始 heartbeat / telemetry

### 3. 已修正 recv timeout 误判为断线

这是本轮联调中最关键的热修之一。

之前 Pi 侧把 websocket 读超时误判为断线，导致频繁自杀式重连。

本轮已修正为：

1. `recv timeout` 不再等于连接关闭
2. 真正 close/exception 才触发断线重连

## 当前启动链的严重问题

本轮必须如实记录：

```text
当前启动链能工作，但不稳定，不适合作为长期可靠建图/导航启动方案。
```

### 1. 当前 `run_rf1_web_stack.sh` 仍然是一把梭启动

当前行为：

1. clean 全部进程
2. 单次 `roslaunch bringup_rf1_web.launch`
3. 等固定 8 秒
4. 直接打印 topic

它没有做到：

1. 分阶段启动
2. 每阶段等待 2 秒
3. 每阶段检查关键 topic/节点
4. 某阶段失败后中止或重试

### 2. 统一主栈下，雷达节点最脆弱

现场已经观察到：

```text
rplidarNode exit code 255
```

表现为：

1. edge-relay、RF1、IMU、rosbridge 都还在
2. 但雷达节点有时会在统一拉起时退出
3. 导致后端看不到 `/scan`

### 3. 当前主栈查询视角容易误导

本轮现场曾出现：

1. 进程实际都在
2. 但 `rostopic list` 只看到少量 topic

原因是：

1. 查询时机不稳定
2. master 环境视角不一致
3. 一把梭启动脚本没有按阶段确认，导致表象混乱

## 项目负责人必须彻查的问题

建议项目负责人后续重点彻查并给出正式修改方案：

### 1. 启动链必须改为分阶段启动

建议顺序：

1. ROS master
2. RF1 基础链
3. IMU
4. RPLIDAR
5. rosbridge
6. edge-relay

每阶段：

1. 启动
2. 等 2 秒
3. 检查关键 topic / 节点
4. 失败则报错，不继续推进

### 2. 必须增加 `echo` 和状态提示

当前脚本提示太弱。

建议后续脚本在每阶段明确打印：

1. 当前要启动什么
2. 等待多久
3. 检查什么 topic
4. 成功/失败结果

### 3. 雷达节点必须独立重点处理

当前最脆弱环节就是雷达。

建议后续：

1. 将雷达作为独立阶段启动
2. 启动后专门检查 `/scan`
3. 必要时支持单独 retry

### 4. edge-relay 不应再单独起第二套 ROS

本轮已经验证：

```text
edge-relay 必须并入现有 ROS 主栈，不能长期走独立 master 方案。
```
