## 当前唯一入口冻结

当前唯一建图入口冻结为：

```text
clbrobot/launch/slam/lidar_slam_pi.launch
```

当前推荐启动包装脚本：

```text
Raspberrypi/catkin_ws/start_slam_mapping.sh

```

1. `start_slam_mapping.sh` 为唯一推荐的人手启动入口
2. `run_slam_mapping.sh` 为兼容脚本，但同样只能通过**单个 roslaunch**启动整套会话
3. 禁止再先手工起 `roscore`，再叠加第二套 `roslaunch`
4. 禁止再在脚本里手工补起第二个 master

### 2. 唯一地图保存入口

```text

```

### 3. 唯一定位/导航入口

## 当前禁止直接混用的旧入口

1. `clbrobot/launch/bringup.launch`

3. 任何未清理旧会话后直接再次起的 `roscore/roslaunch`

## 当前必须执行的清理动作

本轮新增会话清理脚本：

```text
Raspberrypi/catkin_ws/clean_ros_sessions.sh
```

它负责：

1. 清旧 `roscore`
2. 清旧 `rosmaster`
3. 清旧 `roslaunch`
4. 清 `slam_gmapping`
5. 清 `move_base`
6. 清传感器与底盘桥残留节点

这是当前树莓派端必须执行的启动前动作，用来避免：

1. `11311` 端口冲突
2. 同名节点重复注册
3. 旧会话污染新建图会话

## 后续长期开发硬约束

从本轮开始，树莓派长期开发必须遵守以下硬约束：

### 1. 禁止在 `~/.bashrc` 中自动 source 项目工作区

只允许保留系统级：

```text
source /opt/ros/noetic/setup.bash
```

禁止再把以下路径设为默认自动 source：

1. `~/goolge_ws/install_isolated/setup.bash`
2. `~/catkin_ws/devel/setup.bash`
3. 任意 `CleanScout` 项目工作区 setup.bash

### 2. 禁止在 `~/.bashrc` 中自动导出项目级 ROS 变量

禁止把以下变量设为全局 shell 默认值：

1. `ROS_MASTER_URI`
2. `ROS_IP`
3. `ROS_HOSTNAME`
4. `CLBBASE`
5. `CLBLIDAR`

这些变量只能通过项目入口脚本设置。

### 3. 必须通过新工作区入口进入项目环境

统一入口：

```text
Raspberrypi/catkin_ws/use_cleanscout_pi.sh
```

任何需要进入项目环境的 shell，都应显式执行：

```bash
source /home/clbrobot/Work/CleanScout_rover/Raspberrypi/catkin_ws/use_cleanscout_pi.sh
```

### 4. 禁止后台遗留 master 成为默认操作方式

禁止长期依赖：

1. `nohup ... roscore &`
2. `nohup ... rosmaster --core &`
3. 任意未登记、未清理的后台 ROS 会话
4. 在同一个 master 上重复启动第二套冲突的 `roslaunch`

若需要后台启动，必须先通过：

```text
check_ros_master.sh
clean_ros_sessions.sh
```

确认不会污染旧会话。

### 5. 启动前先查、再清、后起

建议固定顺序：

1. `check_ros_master.sh`
2. `clean_ros_sessions.sh`
3. `use_cleanscout_pi.sh`
4. `start_slam_mapping.sh` 或其他唯一入口脚本

### 6. 当前正常 ROS 启动模型

当前冻结的正常模式为：

```text
一个入口脚本 -> 一次 roslaunch -> 由 roslaunch 自己管理 master
```

明确禁止的旧模式为：

```text
先手工起 roscore
再起第二套 roslaunch
再在脚本里后台补 master
```

本轮已实际验证：

1. `start_slam_mapping.sh` 单入口模式
2. 直接拉起 `/scan /imu/data /odom /map /tf`

这说明这条启动模式比旧的“多入口拼接 + 后台 master”稳定得多。

### 7. 旧入口保留但不得直接混用

以下只保留为历史参考：

1. `bringup.launch`
2. 旧 `goolge_ws`
3. 旧 `~/catkin_ws`

不得与当前新工作区唯一入口混用。

## 旧会话根因查明

本轮已确认：

1. 当前反复出现的“旧 rosmaster 仍在”并不是旧仓库自动启动
2. 真实根因是此前调试阶段用后台 `nohup ... roscore/rosmaster ... &` 启动过 master
3. 这些进程在父 shell 退出后变成了 `PPID=1` 的孤儿进程，继续占用 `11311`
4. 新终端由于旧 `.bashrc` 自动设置 `ROS_MASTER_URI=http://当前IP:11311`，会立刻连到这些残留 master

因此：

```text
/rosout 和 /rosout_agg 不是问题本身，
它们只是“旧 master 还活着”的表象。
```

   - 检查是否有残留 `rosmaster / roscore / roslaunch`
   - 检查 `11311` 是否被占用

3. `start_slam_mapping.sh`
   - 先进入 `CleanScout` 环境
   - 再清理旧会话
   - 最后前台启动唯一建图入口

## 一句话总结

`C-2.3.5A` 当前冻结的是“唯一入口和唯一清理动作”，目的是把树莓派端从反复撞旧会话和旧入口的混乱状态里拉出来，为正式建图、保存地图、定位导航提供唯一权威主线。

当前进一步冻结后的长期约束是：

```text
系统级 shell 只保留 ROS 本体，项目级环境一律从 CleanScout 新工作区入口显式进入；
任何会话在启动前必须先查残留 master、先清旧会话，再起唯一入口。
```
