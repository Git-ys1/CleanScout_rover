## 当前冻结硬件接线

MPU6050 接树莓派 4B，使用 I2C：

1. `VCC -> 3V3`
2. `GND -> GND`
3. `SDA -> GPIO2 / Pin 3`
4. `SCL -> GPIO3 / Pin 5`
5. `AD0 -> GND`
6. `XDA / XCL / INT -> 不接`

## 当前冻结地址

当前地址冻结为：

```text
0x68
```

原因：`AD0 -> GND`

## 当前冻结接入路线

本轮不重写现有后处理，只补前端 raw driver：

1. `mpu6050_i2c_bridge` 读取 Raspberry Pi I2C
2. 发布 `raw_imu` (`clb_msgs/Imu`)
3. `imu_calib/apply_calib` 消费 `raw_imu`
4. 输出 `imu/data_raw`
5. `imu_filter_madgwick` 输出 `imu/data`
6. `robot_localization` 继续消费 `imu/data`

## ROS 启动纪律

这台树莓派本身长期带有既有 ROS 环境、旧工作区 overlay、以及多轮临时 bringup 进程。

因此本机后续开发必须默认警惕以下问题：

1. 旧的 `roscore` / `rosmaster` 未退出
2. 旧的 `roslaunch` 残留造成 `run_id` 冲突
3. shell 里遗留的 `ROS_MASTER_URI / ROS_IP / ROS_HOSTNAME`
4. 旧工作区 overlay 导致节点、消息或 Python 包解析错位

这不是本轮单点偶发现象，而是当前树莓派长期开发环境的固定风险。

### 当前冻结建议

后续每轮 bringup 前，先做最小清理：

```bash
pkill -9 -f roscore
pkill -9 -f rosmaster
pkill -9 -f roslaunch
```

然后显式设置本轮 ROS 环境，再启动：

```bash
export ROS_MASTER_URI=http://127.0.0.1:11311
export ROS_IP=127.0.0.1
export ROS_HOSTNAME=127.0.0.1
source /opt/ros/noetic/setup.bash
source /home/clbrobot/Work/CleanScout_rover/Raspberrypi/catkin_ws/devel/setup.bash
```

### 当前开发建议

1. 本轮临时 bringup 优先用单独 `rosmaster --core` + `rosrun` 方式验证
2. 只有在链路稳定后，再回写成正式 `roslaunch`
3. 原有 launch 不删除，继续保留为参考入口
4. 后续最终整合阶段，再统一清理和收口旧入口

### 结论

当前这台机器确实容易因为“原本就有 ROS 环境”而出现会话错乱。

所以后续树莓派端文档必须把：

1. 启动前清理
2. 环境变量冻结
3. 工作区 source 顺序
4. 临时验证入口

都明确写进 release 文档，避免后面重复踩坑。
