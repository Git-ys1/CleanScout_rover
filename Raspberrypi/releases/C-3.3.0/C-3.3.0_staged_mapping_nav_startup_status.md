## 本轮为什么必须改成分阶段启动链

因为当前旧 `run_rf1_web_stack.sh` 的模式是：

1. 一次 clean
2. 一个总 launch 并发拉全栈
3. 固定等待几秒

这在树莓派 4B 上已实测出现：

1. 节点并发竞争
2. 雷达偶发未就绪
3. 查询视角混乱
4. 启动看起来“有东西”，但并不等于真正稳定

所以 `C-3.3.0` 必须转为：

1. 先 master
2. 再 RF1 core
3. 再 IMU
4. 再雷达
5. 再 SLAM 或 NAV

## 当前真实阻塞项

### 阻塞 A：前序判断里包名核查方式需要纠正

现场核查结果：

```text
旧链实际配置：pkg="gmapping" type="slam_gmapping"
```

但：

- `gmapping` 当前在 `/opt/ros/noetic/lib/gmapping/slam_gmapping` 可见
- `map_server` 存在
- `amcl` 存在
- `move_base` 存在

这说明：

1. 旧地图链确实使用过 gmapping
2. 我前序直接以包名 `slam_gmapping` 查询，这个判断方式不准确
3. 正确的包名应按旧 launch 配置核查为 `gmapping`

所以当前问题不是“你记错了以前没用 gmapping”，而是：

```text
历史上确实用过 gmapping，本机系统运行时也存在 gmapping；前序把包名查错了，需要纠正。
```

### 阻塞 B：雷达运行时仍依赖旧工作区编译产物

当前雷达源码虽然已迁入当前工作区：

- `Raspberrypi/catkin_ws/src/rplidar_ros/`

但运行时仍然实际调用：

```text
/home/clbrobot/catkin_ws/devel/lib/rplidar_ros/rplidarNode
```

这意味着：

1. 当前雷达仍有旧工作区运行时依赖
2. 这是当前建图/导航启动链脆弱性的核心来源之一
3. 如果它继续挡住后续 406 建图/导航，就必须考虑后续单独处理雷达底层与重新编译问题

1. source 环境

这样做的目的就是：

```text
不假装 406 建图链已经完整可跑。
```

### 3. `run_navigation_406.sh`

当前会：

1. source 环境
2. 检查 `map_server / amcl / move_base`
3. 若依赖齐全则继续起 RF1 core / 雷达 / nav_406
