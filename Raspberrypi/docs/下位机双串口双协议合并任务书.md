# 下位机双串口双协议合并任务书

## 0. 给执行工程师的指令

请先完整审计当前云端 `main`，再修改代码。不要依据旧实验目录直接拼接工程，不要覆盖
已经验证过的底盘和机械臂基线，也不要在未完成台架验证前烧录后直接进行整车运动测试。

本任务由 STM32 / 下位机维护方执行。Raspberrypi 维护方只冻结树莓派 RF1 兼容契约和
后续 ROS Noetic 分布式验收门槛，不接管固件实现。

## 1. 任务目标

把当前同一块 STM32F103RCT6 上的两套独立固件良性合并为一套正式固件：

1. 保留 OpenRF1 四轮底盘 50 Hz 闭环、编码器反馈、遥测和失联停车。
2. 保留机械臂当前已验证的单舵机、多舵机、停止和查询能力。
3. 树莓派继续通过原有 USART2 控制底盘，不修改当前运动协议。
4. 香橙派通过新分配的 USART3 控制机械臂，使用独立机械臂协议。
5. 两个上位机可以同时在线、同时收发，但不能污染、抢占或误解析对方的数据。
6. MCU 内部只有一套启动入口、时基、中断表、资源表和安全监督器。
7. 交付达到本任务定义的 ROS-ready 门槛后，再由 Raspberrypi 维护方启动整车分布式 ROS
   全链路验收。

最终架构口径固定为：

```text
单一 STM32 固件
├── USART2 / 树莓派 / 底盘运动协议
├── USART3 / 香橙派 / 机械臂协议
├── UART5  / MCU 内部 / 总线舵机半双工链路
└── 共享安全监督器，但不共享协议接收状态
```

## 4. 串口资源冻结

| 端口 | 引脚 / 接口 | 连接方 | 唯一职责 | 波特率 |
| --- | --- | --- | --- | --- |
| USART2 | PA2_TX2 / PA3_RX2，用户口 H5 | 树莓派 | 底盘命令和底盘遥测 | 115200 8N1 |
| USART3 | PB10_TX3 / PB11_RX3，蓝牙口 H6 | 香橙派 | 机械臂命令、查询和状态 | 115200 8N1 |
| UART5 | PC12 半双工链路 | MCU 到总线舵机 | 内部执行总线 | 保持已验证配置 |
| USART1 | 调试口 | 开发调试 | 日志或台架诊断 | 不得参与正式主机协议解析 |

硬约束：

1. USART2 不接受 `#`、`{`、`$` 开头的机械臂业务帧。
2. USART3 不接受 `W/M/E/D/STOP` 底盘业务帧。
3. 两个主机端口不得互相转发原始命令、ACK、遥测或舵机回包。
4. UART5 不是香橙派入口，不得把它与 USART3 混为同一个 RX 状态机。
5. 每个端口必须有独立 RX ring、帧缓冲、解析状态、错误计数和 TX 队列。
6. 串口 RX 中断只允许读取字节并写入对应 ring，不得在中断中执行字符串解析、舵机动作、
   `printf`、Flash 操作或阻塞发送。

当前机械臂工程的 USART1/2/3/5 共用 `uart_receive_buf`、`buf_index`、`uart_mode` 的实现
不得带入合并工程。

## 5. USART2 底盘协议契约

### 5.1 上位机到 MCU

以下语义、字段顺序、单位和换行边界必须保持兼容：

```text
W,a,b,c,d\n
M,ch,pwm\n
E,ch\n
D,ch\n
STOP\n
```

其中：

1. `W` 的四轮顺序固定为 `CN1/LR、CN2/LF、CN3/RR、CN4/RF`。
2. `W` 单位保持 m/s。
3. `M` 只用于受控诊断，不得改变现有范围与看门狗语义。
4. `STOP` 只停止底盘运动域，不等同于机械臂停止或全局急停。

允许向后兼容地新增只读 `INFO\n` 查询，但不得替换、重命名或改变上述现有命令。

### 5.2 MCU 到树莓派

必须保留：

```text
CSR_RF1_READY
ACK:W
ACK:M
ACK:E
ACK:D
ACK:STOP
ERR:<reason>
VEL,...
PWM,...
NAVDBG,...
ENC,...
DBG,...
```

不得改变 `VEL` 前四个实测轮速的顺序，不得向 USART2 输出机械臂文本帧或机械臂 ACK。
若实现 `INFO\n`，响应必须是独立换行帧并包含 combined 固件版本、协议版本和能力位；现有
树莓派桥忽略未知帧时仍须保持正常运行。

### 5.3 实时性与看门狗

1. 运动控制周期保持 20 ms / 50 Hz。
2. 正常遥测保持 100 ms / 10 Hz，除非用测量证明新值更安全。
3. `W` 命令超时停车保持不大于 250 ms。
4. `M` 诊断模式超时保持不大于 2000 ms。
5. 机械臂长帧、Flash、舵机动作和 USART3 流量不得让底盘控制周期持续丢帧。
6. 原 `Raspberrypi` 的 `rf1_serial_bridge.py` 在兼容模式下不得要求修改即可工作。

## 6. USART3 机械臂协议契约

### 6.1 Legacy 兼容模式

合并后的第一版必须继续接受当前香橙派已经使用的帧：

```text
#000P1500T0200!
{#000P1500T0200!#001P1600T0200!}
#000PDST!
#000PRAD!
```

要求：

1. 单舵机和多舵机帧只有在完整收到终止符后才能执行。
2. `{...}` 必须先完整校验，再原子提交目标，不能收到一半就移动部分舵机。
3. 未知、超长、超时、越界和嵌套错误帧必须丢弃，绝不能执行残帧。
4. `#255PDST!` 在越界路径未修复并验证前必须禁用。
5. 003 号舵机反向只允许在 MCU 或 OrangePi 一端实现一次。
6. 必须审计 `$KMS` 等历史命令的实际调用者；不支持时返回明确错误，不得静默执行错误
   语义。

Legacy 模式以兼容现有 OrangePi 驱动为目标，不得突然增加会长期堆积且无人读取的大量
无请求回包。

### 6.2 ARM_V2 增强模式

新增可协商的增强模式，建议由以下握手启用：

```text
OrangePi -> @HELLO:ARM_V2!
MCU      -> @READY:ARM_V2!
OrangePi -> @INFO!
MCU      -> @INFO:COMBINED:<fw-version>:ARM_V2!
```

进入增强模式后：

1. 每次只允许一个未确认机械臂命令，采用 stop-and-wait，避免没有序号时 ACK 错配。
2. 成功返回 `@ACK:OK!`。
3. 失败返回 `@ERR:<reason>!`，至少区分 `BAD_FRAME`、`LIMIT`、`BUSY`、`TIMEOUT`、
   `UNSUPPORTED` 和 `WATCHDOG`。
4. OrangePi 发送 `@PING!` 维持 live session。
5. live session 超过 300 至 500 ms 未收到命令或心跳时，机械臂进入经过验证的 hold 或
   逐轴 stop，不影响底盘继续安全运行。
6. Legacy 与 ARM_V2 必须有明确状态，不得对同一帧重复执行。

ARM_V2 需要 OrangePi 后续增加 ACK reader 和 heartbeat，但 Legacy 必须保证现有工具仍可
用于回归与回滚。

## 7. 共享安全监督器

合并固件需要新增独立于两套协议解析器的 `safety_supervisor`。最低要求：

1. 上电时四轮目标为零，机械臂不执行缓存中的未确认动作。
2. 底盘 `STOP` 只清底盘状态；机械臂 `PDST` 只停止指定机械臂轴。
3. 两个主机端口均支持全局锁存急停：

```text
USART2: ESTOP\n
USART3: @ESTOP!
```

4. 全局急停必须同时停止四轮和机械臂，并保持锁存。
5. 清除急停只能在四轮目标为零、机械臂静止、无活动故障时执行；具体 CLEAR 命令必须
   写入协议文档并经过台架验证。
6. USART2 断开只触发底盘看门狗，不得破坏机械臂串口状态。
7. USART3 断开只触发机械臂 hold/stop，不得复位底盘控制器。
8. 一个端口 RX overflow、坏帧或发送阻塞不得拖死另一个端口。

底盘与机械臂的姿态联锁策略在没有实车数据前先采用 `monitor_only`：记录“机械臂展开且
底盘运动”等风险状态，但不得在本轮未经验证就静默篡改导航速度。正式限速策略另行标定。

## 8. 调度与资源要求

建议模块结构：

```text
User/
├── main.c
├── csr_motion_controller.c/.h
├── csr_motion_link.c/.h
├── arm_host_link.c/.h
├── arm_protocol.c/.h
├── arm_servo_executor.c/.h
├── arm_servo_bus.c/.h
├── safety_supervisor.c/.h
└── board_resource_map.c/.h
```

主循环优先级固定为：

```text
安全故障处理
-> 到期的 20 ms 底盘控制 tick
-> USART2 命令解析
-> USART3 命令解析
-> 机械臂非阻塞执行 tick
-> 低优先级遥测与日志
```

必须完成资源表审计：

1. 运动侧 TIM2/3/4/5 编码器、TIM8 电机 PWM。
2. 机械臂 TIM7 本地 PWM。
3. USART2、USART3、UART5、USART1 的 GPIO 和 NVIC。
4. SysTick 只能有一个实现。
5. Flash、RAM、栈和所有 RX/TX buffer 的 map 文件占用。
6. 中断优先级必须保证底盘控制与 USART2 接收不会被机械臂长时间压制。

机械臂执行后端必须显式配置为 `local_pwm` 或 `servo_bus`，不得默认把同一条命令同时镜像
到两套执行后端。

## 9. 禁止事项

1. 禁止让两个 UART 共用一个全局接收缓冲或帧状态机。
2. 禁止在 USART 中断内执行完整文本解析或动作。
3. 禁止把机械臂命令转发到树莓派端口。
4. 禁止把底盘遥测输出到香橙派端口。
5. 禁止改变 `W` 的单位、轮序、50 Hz 控制和 250 ms 看门狗。
6. 禁止为了合并方便而让树莓派改用机械臂协议。
7. 禁止让香橙派获得普通底盘运动命令权限。
8. 禁止引入假的编码器、假的舵机 ACK 或伪造遥测。
9. 禁止删除两份独立可回滚基线。
10. 禁止未架空车轮、未卸载机械臂负载就运行并发运动测试。

## 10. 推荐实施顺序

1. 冻结两个基线的 commit、HEX、编译日志和当前协议。
2. 从运动工程建立新的 combined 工程，并证明未移植机械臂时底盘行为不变。
3. 建立统一时基、资源表和主循环调度器。
4. 实现独立 USART2 motion link，回归全部旧底盘协议。
5. 实现独立 USART3 arm link，只做收帧、校验和 dry-run ACK，不接执行器。
6. 移植机械臂最小执行层、限位、逐轴停止和查询。
7. 接入共享安全监督器和全局急停。
8. 完成双路并发压力测试。
9. 先架空轮子、卸载机械臂负载完成板级验证，再做受控单域动作。
10. 完成烧录、回读校验和交接，通知 Raspberrypi 维护方进行 ROS 全链路测试。

## 11. 自动检查与构建

至少执行并保存：

1. Keil clean build，`0 Error(s)`；新增代码不得留下未解释 warning。
2. 输出 HEX、AXF、map 和 build log。
3. 检查最终工程只有一个 `main`、一个 SysTick 和每个 IRQ 唯一定义。
4. 搜索并证明不存在跨 UART 共用的 `uart_receive_buf`、`buf_index`、`uart_mode`。
5. 搜索并证明 USART2 输出函数不发送机械臂帧，USART3 输出函数不发送底盘遥测。
6. 对两个协议解析器执行 host/native 单元测试或等价的串口回放测试。
7. 对最大长度帧、空帧、残帧、连续帧、溢出帧和随机字节执行 fuzz/回放检查。
8. 检查 map 文件，记录 Flash、RAM、栈和 buffer 占用，禁止只凭芯片容量猜测“肯定够”。

## 12. 台架验收矩阵

### 12.1 不接执行器

| 场景 | 预期结果 |
| --- | --- |
| USART2 连续 50 Hz 零 `W` | 正常 ACK，控制 tick 稳定，不输出非零 PWM |
| USART3 连续合法机械臂帧 | 解析成功但 dry-run 不动作 |
| 两路同时满负载 | 无串帧、无 overflow、无控制周期持续超限 |
| USART2 注入机械臂帧 | 明确拒绝，机械臂不动作 |
| USART3 注入 `W` 帧 | 明确拒绝，底盘目标不变 |
| 任一路发送残帧/超长帧 | 丢弃整帧，不执行部分命令 |
| 拔掉树莓派链路 | 250 ms 级底盘停车，机械臂会话保持 |
| 拔掉香橙派链路 | 机械臂 hold/stop，底盘控制保持 |
| 任一路发送 ESTOP | 两个执行域同时停止并锁存 |

### 12.2 受控单域动作

1. 先架空四轮，只接树莓派，回归前进、后退、左右转和 STOP。
2. 再保持底盘零目标，只接香橙派，逐轴回归机械臂限位与 PDST。
3. 禁止在单域回归未通过前进行并发动作。

### 12.3 双域并发

1. 树莓派持续发送 50 Hz `W`，香橙派同时以实际视觉控制频率发送机械臂帧。
2. 记录每个底盘控制 tick 间隔、两路 RX overflow、协议错误和看门狗触发次数。
3. 机械臂长帧、查询和动作期间不得造成底盘误停车或明显控制抖动。
4. 停止任一上位机后，另一执行域必须继续按各自契约运行。

## 13. ROS-ready 交付门槛

只有以下全部满足，才通知 Raspberrypi 维护方开始分布式 ROS 验收：

1. 合并固件已经实际烧录、校验并复位成功。
2. 树莓派原串口仍连接 USART2，香橙派新串口连接 USART3。
3. 树莓派无需修改现有 RF1 协议即可看到 `CSR_RF1_READY`、`ACK:W`、`VEL` 和 `PWM`。
4. `/rf1/vel` 的前四项顺序、符号和单位与 C-4.1.x 基线一致。
5. 50 Hz 零目标持续运行时没有串口重连、ERR 洪泛或非零输出。
6. 香橙派能在 USART3 独立控制机械臂，且其字节不会出现在 USART2。
7. 两路同时在线至少稳定运行 30 分钟，无 RX overflow、控制周期持续超限或看门狗误触发。
8. 树莓派断联停车和香橙派断联 hold/stop 均已独立验证。
9. 全局急停已从两个端口分别验证。
10. 已提供回滚 HEX，现场可在合并版本异常时恢复纯运动或纯机械臂版本。

达到门槛后，Raspberrypi 维护方将按以下顺序继续：

```text
树莓派硬件链
-> /rf1/vel 与 /scan
-> PC 编码器 /odom
-> TF
-> gmapping
-> AMCL + move_base/TEB
-> edge-relay 与多功能硬件
-> 分布式 ROS 长稳和受控实车验收
```

8. 明确写出 `ROS_READY=YES` 或 `ROS_READY=NO`，不得用模糊措辞代替。

## 16. 完成定义

本任务的完成标准不是“两个工程能一起编译”，而是：

1. 一个 MCU 固件同时稳定服务两台上位机。
2. 两个 UART 各自只有一个协议和一个控制域。
3. 底盘实时性和现有树莓派协议没有回归。
4. 机械臂帧完整、安全、可停止、可诊断。
5. 任一链路故障不会拖死另一链路。
6. 全局急停能覆盖两个执行域。
7. 交付证据足以让 Raspberrypi 维护方直接开始 ROS Noetic 分布式全链路测试。
