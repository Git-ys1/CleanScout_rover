# 树莓派 ROS 与合并下位机验收记录

## 1. 验收结论

| 项目 | 结论 |
| --- | --- |
| 日期 | 2026-07-19 |
| Raspberry Pi 运动链 | `RPI_ROS_MOTION_READY=YES` |
| 完整分布式系统 | `FULL_DISTRIBUTED_ROS_READY=NOT_EVALUATED` |
| 测试方式 | 四轮架空，ROS Noetic 实机测试 |
| 本轮边界 | 仅 Raspberry Pi、USART2、底盘和编码器里程计 |

合并固件保持了 Raspberry Pi 现有运动协议，ROS 无需修改协议即可完成四轮目标下发、
`ACK/VEL/PWM` 回传、里程计计算和 `odom -> base_link` TF 发布。本轮没有测试 Orange Pi、
机械臂、雷达、建图、定位或 TEB 导航，不能用本记录替代这些子系统的验收。

## 3. 静态与零指令检查

启动后确认：

1. `/rf1/ready=True`，`/rf1/status` 为 `RF1 serial connected`。
2. 持续收到 `ACK:W`，静止时 `VEL` 前四项和 `PWM` 四项均为零。
3. `/rf1/wheel_target_ms` 为 `50.00 Hz`。
4. `/rf1/vel` 和 `/rf1/pwm` 均约为 `10.04 Hz`。
5. `/odom` 约为 `29.8 Hz`，`/tf` 约为 `30.0 Hz`。
6. 静止时 `/odom` 的速度为零，没有自行启动或持续漂移。
7. `roswtf` 清理测试工具的失效注册后，仅报告静止时 `/cmd_vel` 没有发布者，属于预期。

`VEL` 的八个字段符合现有协议：前四项是实测轮速，后四项是目标轮速；里程计节点只消费
前四项。

## 4. 架空方向回归

轮序固定为 `[LR, LF, RR, RF]`。每组动作后均显式发送零速度，并确认最后的 `VEL` 和
`PWM` 都回到四零。

| ROS 指令 | 期望轮目标 m/s | 实测结果 |
| --- | --- | --- |
| 前进 `vx=+0.10` | `[+0.10,+0.10,+0.10,+0.10]` | 四路同为正，最大约 `0.090/0.110/0.096/0.086` |
| 后退 `vx=-0.10` | `[-0.10,-0.10,-0.10,-0.10]` | 四路同为负，最大绝对值约 `0.092/0.089/0.088/0.094` |
| 左移 `vy=+0.08` | `[+0.08,-0.08,-0.08,+0.08]` | 四路符号正确，`odom.linear.y` 为正 |
| 右移 `vy=-0.08` | `[-0.08,+0.08,+0.08,-0.08]` | 四路符号正确，`odom.linear.y` 为负 |
| 左旋 `wz=+0.35` | `[-0.0648,-0.0648,+0.0648,+0.0648]` | 四路符号正确，`odom.angular.z` 为正 |
| 右旋 `wz=-0.35` | `[+0.0648,+0.0648,-0.0648,-0.0648]` | 四路符号正确，`odom.angular.z` 为负 |

左旋和右旋期间，编码器里程计的平均角速度分别约为 `+0.204 rad/s` 和 `-0.199 rad/s`；
本数据用于验证方向和数据链，不是在架空条件下重新标定真实角速度。

## 5. 停车与稳定性

1. 单次 `/cmd_vel` 后，`cmdvel_to_rf1` 在约 `400.2 ms` 结束非零目标；修复后复测为
   `401.0 ms`，符合配置的 `0.4 s` 超时。
2. 目标归零后的下一个 `50 Hz` 发布周期内，四轮目标变为零，随后 `VEL/PWM` 均归零。
3. 多组动作、两次标准入口重启和三个错误监听窗口内，`/rf1/error` 没有消息。
4. 下位机维护方已记录约 `254 ms` 的固件 `W` 看门狗停车；本轮没有用突然拔线重复该板级
   测试，避免把其结果冒充为 ROS 测量。

## 6. 树莓派侧修复

### 6.1 串口读写锁竞争

初次运行时，参数虽然配置为 `50 Hz`，`ACK:W` 实测只有约 `20.3 Hz`。对照实验把串口
读取超时从 `200 ms` 临时缩短到 `20 ms` 后，固件可返回约 `49 Hz` ACK，证明瓶颈位于
树莓派桥，而不是新固件。

`rf1_serial_bridge.py` 已将以下职责拆开：

1. 连接对象状态锁。
2. 单读者锁，避免重连 drain 与后台读线程同时读取。
3. 写锁，保证帧写入和关闭互斥，但不再被阻塞式 `readline()` 占用。
4. 旧连接发生迟到错误时，只能关闭对应旧对象，不能误关已经重连的新对象。

修复后保留原 `200 ms` 读取超时，`ACK:W` 实测 `50.07 Hz`，间隔约 `7-32 ms`；标准入口
再次重启后复测 `49.97 Hz`。因此没有用过短读取窗口换取表面频率，也没有改变串口协议。

### 6.2 RF1 标准入口

`run_rf1_core_stack.sh` 同时修复：

1. 通过 `bash` 调用历史上没有执行位的清理脚本。
2. 用 `/run_id` 主动等待 roscore 就绪，消除固定 sleep 导致的 run-id 竞态。
3. 将 `/rf1/vel` 就绪等待扩展为有明确报错的 10 秒门槛。
4. 补齐入口脚本自身的执行位。

修复后的标准入口连续执行两次均成功到达 `RF1 core 已就绪`。

## 8. 后续边界

树莓派运动链可以进入下一阶段，但仍需分别完成：

1. 车辆落地后的直行、转向和里程计尺度复核。
2. 雷达、静态 TF、PC 建图、AMCL 和 TEB 导航联调。
3. 树莓派、香橙派和下位机双路同时在线的长时间分布式运行。
4. 机械臂运动、限位、负载和 USART3 测试，由 Orange Pi 与下位机维护方负责。

以上边界不阻塞 `RPI_ROS_MOTION_READY=YES`，但在对应测试完成前不得扩写为整车全部通过。
