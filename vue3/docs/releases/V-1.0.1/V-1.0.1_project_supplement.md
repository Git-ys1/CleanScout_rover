# V-1.0.1 项目补充冻结说明

## V线项目重新定位

从 `V-1.0.1` 起，V 线不再只定义为“小程序前端”，而正式定义为本项目的软件交互系统线。

它由三部分组成：

1. 前端端：基于 uni-app + Vue3 + CLI，承担用户交互界面、对话控制界面、设备状态展示、管理员直接控制入口，以及后续 App/H5/小程序多端适配。
2. 后端端：承担多账户登录、账号与权限、设备绑定关系、openclaw 转发数据缓存、消息中转、状态持久化，以及对前端暴露统一 API / WS 接口。
3. 设备网关端：由树莓派 + openclaw 转发链继续承担设备实时状态来源、指令转发和设备执行结果回传。

结论：

- V 线不再是纯页面项目。
- V 线最终形态是前端 + 后端 + 设备接入协议三层软件系统项目。
- 当前本地仓仍只承载前端工程与冻结文档。

## 三层系统架构

当前冻结的系统架构如下：

```text
用户 / 管理员
        |
        v
前端端（uni-app + Vue3 + CLI）
        |
        v
后端端（账户 / 缓存 / API / 权限 / 消息协调）
        |
        v
设备网关端（树莓派 + openclaw 转发链）
        |
        v
设备 / 巡检车
```

当前阶段的边界约束：

- 前端仓继续使用当前 `vue3` 工程。
- 后端纳入系统边界，但本轮不创建 `backend/`。
- 设备网关继续由树莓派 + openclaw 转发链承担。

## 云端追踪与发布纪律

这是 `V-1.0.1` 最重要的新增纪律。

### 原则 1：实时追踪当前主项目最新施工分支

V 线每轮发布前，必须先在主项目 GitHub 分支页面确认当前最新施工分支。

当前主项目云端仓库地址：

- `https://github.com/Git-ys1/CleanScout_rover`

当前实例分支为：

`feature/c-3.0.8-openrf1-cn1-cn3-encoder-recovery`

但该分支只作为当轮实例记录，不能被制度化为固定长期目标。制度始终写成：

“以发布前确认的当前最新施工分支为准，不以历史旧分支为准。”

### 原则 2：云端镜像永远发布到最新分支根目录的 `vue3/`

发布目标固定为：

```text
<当前最新主分支根目录>/vue3/
```

### 原则 3：每轮 release 文档必须记录发布元数据

从 `V-1.0.1` 起，每轮 release 文档必须显式写：

- `based_on_branch: feature/...`
- `branch_source: GitHub branches page`
- `published_to_root: vue3/`
- `published_at_commit: ...`

### 原则 4：V线不得在过时分支上长期漂移

V 线可以本地独立开发，但每轮冻结结果必须投递到当前主项目最新施工分支，否则会与 C 线、树莓派线和设备链路脱节。

## Git纪律补充冻结

从 `V-1.0.1` 起，V 线 Git 纪律补充冻结为：

1. 开工前必须先建 checkpoint。
2. checkpoint 必须是可回溯的 Git 留痕，不能只靠口头说明。
3. 开工后必须及时 commit，不允许把本轮有效成果长期滞留在未提交工作树。
4. 本地 commit 完成后，才能执行云端发布。
5. 提交标题统一使用中文，版本号、分支名及必要技术关键词保留英文。

推荐标题示例：

- `V-1.0.1: 冻结系统补充文档与 Git 发布纪律`

## V线官方底层开发手册

从 `V-1.0.1` 起，以下四份文档统一命名为“V线官方底层开发手册”：

### 手册 A：uni-app CLI 快速上手

用于：

- 创建工程
- 升级依赖
- 运行与发布
- Node / HBuilderX 环境约束
- 平台命令规范

官方入口：https://uniapp.dcloud.net.cn/quickstart-cli

### 手册 B：页面与 pages.json

用于：

- 页面目录规范
- 页面注册规则
- 首页设置
- 页面管理规则

官方入口：https://uniapp.dcloud.net.cn/collocation/pages

纪律要求：

- 页面文件与 `pages.json` 注册必须同步施工
- 未注册页面会被编译忽略

### 手册 C：WebSocket / SocketTask

用于：

- 实时连接
- 消息收发
- 连接状态监听
- 错误处理

官方入口：https://uniapp.dcloud.net.cn/api/request/websocket.html

纪律要求：

- 前端实时通信一律按 `SocketTask` 思路设计
- 不再以旧式全局 socket API 为基线

### 手册 D：Pinia 状态管理

用于：

- 应用级状态管理
- 会话状态
- 设备状态
- 管理员状态
- 用户状态

官方入口：https://uniapp.dcloud.net.cn/tutorial/vue3-pinia.html

纪律要求：

- 统一使用 `Pinia`
- 不准继续在页面里堆临时全局变量当状态层

代码工程师遇到以下问题时，一律优先查这四份手册：

- 页面不显示
- 页面路由错
- 通信链路错
- 状态不同步
- CLI 环境错

## 后端正式纳入说明

本轮新增的核心变化是：

后端从 `V-1.0.1` 起正式纳入系统边界，不再是“以后可能有用”的可选项。

### 后端第一阶段职责

1. 账户体系：多账户登录、用户信息、登录态管理。
2. 权限体系：普通用户、管理员、调试/开发权限预留。
3. 设备数据缓存：缓存 openclaw 转发来的状态数据、最近消息和最近任务结果。
4. 对前端统一输出接口：REST API、WebSocket / SSE / 转发接口。
5. 对设备侧留接口：树莓派 / openclaw 消息上送、指令转发回设备。

### 当前阶段为什么还不立刻开发后端

因为 `V-1.0.1` 仍然是立项补充轮。本轮只要求：

- 认知冻结
- 文档冻结
- 边界冻结
- 目录与仓库纪律冻结

因此本轮不要求马上起一个后端仓，也不要求现在创建 `backend/` 实体目录。

## 后续预留

从下一阶段开始，必须预留以下设计空间：

- `backend/`
- `api schema`
- `user model`
- `device model`
- `message model`

这些内容在 `V-1.0.1` 只做边界冻结，不做代码实现。

## 与 V-1.0.0 的关系

- `V-1.0.0` 继续保留为前端控制端立项冻结的历史快照。
- `V-1.0.1` 在不回写 `V-1.0.0` 历史结论的前提下，补充冻结主分支追踪纪律、官方文档手册化，以及后端正式纳入系统边界。
