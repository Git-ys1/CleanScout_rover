# V-1.0.1 立项补充冻结计划

## Summary
- 当前仓库只有 `docs/releases/V-1.0.0/README.md`，根 `README.md` 仍是“前端控制端”口径；V-1.0.1 需要把项目重新冻结为“前端 + 后端 + 设备网关”的软件交互系统线。
- 本地 `vue3` 仓没有配置主项目远端，因此“当前最新施工分支”的制度来源应写死为 `GitHub 分支页人工确认`，而不是本仓自动查询。
- 本轮仍然只做文档和制度冻结，不创建 `backend/`，不开发后端，不改业务功能，不回写 V-1.0.0 历史结论。

## Key Changes
- 新增 `docs/releases/V-1.0.1/README.md` 作为本轮标准 release 索引，同时新增 `docs/releases/V-1.0.1/V-1.0.1_project_supplement.md` 承载补充冻结正文。
- `docs/releases/V-1.0.1/README.md` 顶部固定加入发布元数据块，字段名固定为：
  - `version: V-1.0.1`
  - `based_on_branch: <当轮发布前在 GitHub 分支页确认的最新施工分支>`
  - `branch_source: GitHub branches page`
  - `published_to_root: vue3/`
  - `published_at_commit: pending-cloud-publish`
- `docs/releases/V-1.0.1/V-1.0.1_project_supplement.md` 固定包含以下章节：
  - V 线项目重新定位：明确为“软件交互系统线”
  - 三层系统架构：前端、后端、树莓派/openclaw 设备侧
  - 云端追踪与发布纪律：实时追踪最新施工分支、镜像永远发布到该分支根目录的 `vue3/`、不得长期漂移在过时分支
  - 官方底层开发手册：CLI、pages.json、WebSocket/SocketTask、Pinia 四册
  - 后端正式纳入边界：账户、权限、缓存、统一 API/WS 出口、设备侧接口
  - 当前阶段不开发后端的原因：本轮只做认知、边界、目录与纪律冻结
  - 后续预留：`backend/`、`api schema`、`user model`、`device model`、`message model`
- 根 `README.md` 做口径升级，不再保留“后端暂不立项”的主叙述；调整为：
  - 标题和开篇定位改成“V线软件交互系统线”
  - 新增“当前主分支追踪规则”章节，明确以 GitHub 分支页确认最新施工分支
  - 将“官方参考入口”升级为“V线官方底层开发手册”，只保留四册为手册级入口
  - 新增“后端已纳入系统边界”章节，并写明本轮不开发、后续必须立项
  - 保留当前真实工程结构为 `src/*`，并明确说明页面注册规则在本仓落地为“页面文件位于 `src/pages/`，`pages.json` 路径仍按 uni-app 约定填写 `pages/...`”
- V-1.0.0 release 文档保持历史快照，不回写“后端正式纳入”等 V-1.0.1 结论；如需关联，只在 V-1.0.1 文档中引用 V-1.0.0。
- 文档级接口/规则在 V-1.0.1 明确冻结为：
  - 发布追踪元数据：`based_on_branch`、`branch_source`、`published_to_root`、`published_at_commit`
  - 前端实时通信基线：`SocketTask`
  - 前端状态管理基线：`Pinia`
  - 后续系统建模预留：`user model`、`device model`、`message model`

## Assumptions
- “当前最新施工分支”的制度来源固定为 GitHub 分支页人工确认；因为本地 `vue3` 仓当前没有主项目 remote，不能把本仓查询写成制度前提。
- `feature/c-3.0.8-openrf1-cn1-cn3-encoder-recovery` 可作为 V-1.0.1 文档中的当前实例分支，但正文必须同时声明“制度以发布前确认的最新施工分支为准”。
- V-1.0.1 只冻结制度与边界，不创建 `backend/`，不新增 API schema 文件，不回改 V-1.0.0 的历史结论。
- 手册级官方文档固定为四册：CLI、pages.json、WebSocket/SocketTask、Pinia；`manifest` 若保留，只能降级为补充工程资料，不再列为手册级入口。
