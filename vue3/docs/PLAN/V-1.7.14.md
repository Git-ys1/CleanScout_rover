# V-1.7.14 计划：FunASR 中文语音输入接入（H5 + 微信小程序，先做非流式）

## Summary

本轮语音输入路线正式冻结为：

```text
前端录音 -> backend ASR 适配层 -> 独立 FunASR 中文识别服务 -> 文本回填聊天输入框 -> 用户确认后再发给 OpenClaw
```

已拍板的关键决策：

- ASR 引擎：`FunASR`
- 语言方向：`中文优先`
- 前端范围：`H5 + 微信小程序`
- 第一版录音交互：`点击开始 / 点击结束`，整段音频上传
- 第一版部署形态：`独立 ASR 服务`，不揉进 Node backend 进程
- 第一版识别后行为：`先回填输入框，用户确认后再发送`
- 第一版音频策略：`backend/ASR 服务统一转码`，通过 `ffmpeg` 归一到 `16k 单声道 wav`
- 本轮不做：真流式 ASR、自动发送、语音播报、说话人分离、情感识别

模型选择固定为：

- 主 ASR：`paraformer-zh`
- VAD：`fsmn-vad`
- 标点恢复：`ct-punc`

后续如果进入真流式轮，再单独切到 `paraformer-zh-streaming`。  
官方依据：

- FunASR 仓库：https://github.com/modelscope/FunASR
- FunAudioLLM / SenseVoice 主页：https://funaudiollm.github.io/

## Key Changes

### 1. 新增独立中文 ASR 服务，不直接让 Node backend 说 FunASR

新增一个独立 Python ASR 服务，作为 backend 的上游依赖。V 线 backend 只做网关协调，不直接内嵌 Python 推理。

服务职责固定为：

- 接收上传音频文件
- 用 `ffmpeg` 转成 `16k / mono / wav`
- 用 `FunASR AutoModel(model="paraformer-zh", vad_model="fsmn-vad", punc_model="ct-punc")` 做识别
- 返回最终中文文本和基础元数据

服务接口固定为最小单接口：

- `POST /recognize`
  - 输入：`multipart/form-data`
    - `file`
    - `lang=zh`
  - 输出：
    - `success`
    - `text`
    - `segments`（可先保留数组，第一版允许为空或只返回整句）
    - `durationMs`
    - `audioFormat`
    - `sampleRate`
    - `model=paraformer-zh`

backend 到 ASR 服务的配置固定新增：

- `ASR_ENABLED=true`
- `ASR_PROVIDER=funasr`
- `ASR_BASE_URL=http://127.0.0.1:7001`
- `ASR_LANGUAGE=zh`
- `ASR_REQUEST_TIMEOUT_MS=60000`

### 2. backend 新增 ASR 适配层，但不改现有聊天接口

Node backend 新增独立 `asr` integration 和路由，保持现有 `/api/chat/send` 不变。

新增接口固定为：

- `POST /api/asr/transcribe`
  - 鉴权：登录用户可用
  - 输入：前端上传录音文件
  - 行为：backend 转发给独立 FunASR 服务
  - 输出：
    - `text`
    - `durationMs`
    - `provider`
    - `language`
    - `model`

- `GET /api/integrations/asr/status`
  - 鉴权：登录用户可用
  - 输出：
    - `enabled`
    - `provider=funasr`
    - `language=zh`
    - `status=healthy|disabled|error`
    - `message`

backend 第一版不做的事：

- 不做音频长期存储
- 不把语音原文件入库
- 不把 ASR 文本自动发给 `/api/chat/send`
- 不做流式 chunk 转发

### 3. 聊天页新增语音输入，但默认“识别后回填输入框”

聊天页在现有固定底部输入区新增语音按钮，第一版交互固定为：

1. 用户点击“开始录音”
2. 页面进入录音态，显示录音中状态和时长
3. 用户点击“结束录音”
4. 前端上传整段音频到 `/api/asr/transcribe`
5. 识别结果回填到当前聊天输入框
6. 用户手动点“发送消息”，再走现有 OpenClaw 文本链路

第一版页面状态固定新增：

- `idle`
- `recording`
- `transcribing`
- `error`

第一版聊天页新增控件固定为：

- 录音按钮
- 停止按钮
- 识别中提示
- “将识别结果填入输入框”的确认提示
- 失败提示

第一版不做：

- 按住说话
- 边录边识别
- 自动直接发送
- 中断恢复
- 语音历史列表

### 4. H5 与微信小程序统一按“前端录音，后端转码”处理

由于 H5 和微信小程序录音格式不完全一致，本轮不强制前端直接输出统一格式。

前端策略固定为：

- H5：使用浏览器录音能力产出 `webm/ogg/wav` 中浏览器可稳定提供的格式
- 微信小程序：使用平台录音管理器，拿到本地临时音频文件
- 前端统一走文件上传，不在前端做复杂转码
- backend/ASR 服务统一做 `ffmpeg` 转码

因此第一版交付要求包含：

- ASR 服务运行环境必须安装 `ffmpeg`
- backend 文档必须明确 `ffmpeg` 是 ASR 服务前置依赖
- 前端不新增本地音频格式适配逻辑

### 5. 中文语音输入与 OpenClaw 的边界固定

本轮必须明确边界：

- `FunASR` 只负责：语音 -> 中文文本
- `OpenClaw` 只负责：中文文本 -> 对话 / 意图 / 控制链

也就是说：

- 语音输入不是直接接 OpenClaw
- OpenClaw 本轮无需感知音频文件
- 语音识别成功后，依然复用现有 `/api/chat/send`

## Public Interfaces / Config

新增 backend 接口：

- `POST /api/asr/transcribe`
- `GET /api/integrations/asr/status`

新增 backend 配置：

- `ASR_ENABLED`
- `ASR_PROVIDER=funasr`
- `ASR_BASE_URL`
- `ASR_LANGUAGE=zh`
- `ASR_REQUEST_TIMEOUT_MS`

新增独立 ASR 服务配置：

- `FUNASR_MODEL=paraformer-zh`
- `FUNASR_VAD_MODEL=fsmn-vad`
- `FUNASR_PUNC_MODEL=ct-punc`
- `FFMPEG_PATH`（如需显式配置）
- `ASR_SERVICE_PORT=7001`

新增前端聊天页状态与交互：

- 录音状态：`idle | recording | transcribing | error`
- 识别结果默认只回填输入框，不自动发送
- 当前链路仍为：`语音 -> 文本 -> 聊天发送`

## Test Plan

- 独立 FunASR 服务启动后，`/recognize` 可返回中文文本
- backend 的 `GET /api/integrations/asr/status` 能正确返回 `healthy`
- backend 的 `POST /api/asr/transcribe` 能成功代理到 FunASR 服务
- H5 录音后上传成功，识别结果能回填聊天输入框
- 微信小程序录音后上传成功，识别结果能回填聊天输入框
- 回填后的文本可以继续通过现有 `/api/chat/send` 正常发送
- OpenClaw 未开启时，语音识别仍可工作，只是聊天发送仍走 mock/当前文本链路
- backend 与数据库无新增强耦合，不要求新增表结构
- 第一版不出现自动发送，不出现边录边识别假流式

## Assumptions

- 第一版中文 ASR 采用 `paraformer-zh + fsmn-vad + ct-punc`
- 第一版仅做“整段录音上传识别”，不做真流式
- 第一版独立部署 Python ASR 服务，Node backend 只做转发和鉴权
- 第一版必须依赖 `ffmpeg` 做跨平台录音格式统一
- 第一版聊天默认“识别后回填输入框”，不自动发送到 OpenClaw
- 语音输入只加在当前聊天页，不扩到首页或管理员页
