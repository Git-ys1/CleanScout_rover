# V-1.2.0 计划：移动端导航重构 + 管理员系统增强 + OpenClaw 接入预备改造

## Summary
- 现仓基线已确认：`src/pages.json` 仅有页面注册、无 `tabBar`；`backend/` 已落地 `Express + Prisma + SQLite`；V-1.1.0 的登录、注册、管理员命令和 mock 联调文档已冻结。
- 本轮不推翻 `src/* + backend/*` 现有结构，只做三类增强：原生 `tabBar` 导航、管理员系统从命令页升级为工作台、`backend -> OpenClaw Gateway` 适配层。
- 版本纪律从本轮改为 `V-1.2.1 / V-1.2.2 / ...`；不再使用 `A/B/C`。release 目录仍使用 `docs/releases/V-1.2.0/README.md` 记录整轮冻结。
- 当前工作树存在根目录 `package-lock.json` 变更。实现前先做一次只读 diff，若只是当前前端依赖漂移则纳入 `checkpoint: before V-1.2.0`，若是用户未说明的独立改动则先停下确认，不得直接覆盖。

## Key Changes
- Git 与发布纪律
  - 开工前提交 `checkpoint: before V-1.2.0`，分支固定为 `feature/v-1.2.0-mobile-admin-openclaw-bridge`。
  - 本轮提交标题固定为：
    - `V-1.2.1: refactor mobile navigation with tabBar and profile admin entry`
    - `V-1.2.2: add admin user management and system config backend`
    - `V-1.2.3: add admin management UI and guards`
    - `V-1.2.4: add OpenClaw backend adapter shell and transport switch`
    - `V-1.2.5: freeze docs and publish vue3 mirror to latest branch`
  - release 文档继续写明 `based_on_branch`、`branch_source`、`published_to_root`、`published_at_commit`。

- V-1.2.1 移动端导航重构
  - 在 `src/pages.json` 新增原生 `tabBar`，一级导航固定为 `首页 / 对话 / 我的`，对应 `pages/index/index`、`pages/chat/index`、`pages/profile/index`。
  - `tabBar` 使用本地图标资源入仓，固定为 3 组占位图标：未选中/选中共 6 个文件，放在 `src/static/tabbar/`。
  - 首页删除“对话/个人中心/管理员”一级按钮堆叠，只保留用户信息、设备摘要、最近消息摘要；从“总览入口页”收敛为真正首页。
  - 个人中心新增管理员入口卡片，仅 `authStore.role === 'admin'` 时显示，进入方式继续使用 `navigateTo('/pages/admin/index')`。
  - 所有跳转到 `tabBar` 页的逻辑统一改为 `uni.switchTab`；管理员页仍是普通页面，不进入 `tabBar`。
  - 前端状态层新增 `adminStore`，聊天页扩展 transport 状态展示；`appStore.currentTab` 与 `tabBar` 对齐。

- V-1.2.2 管理员能力升级 backend
  - Prisma 模型调整：
    - `User` 新增 `isEnabled Boolean @default(true)`。
    - 新增单例 `SystemConfig`：`id String @id` 固定值 `system`、`registrationEnabled Boolean @default(true)`、`appEnabled Boolean @default(true)`、`maintenanceMessage String @default("")`、`openclawEnabled Boolean @default(false)`、`updatedAt DateTime @updatedAt`。
    - `seed.js` 保证默认管理员存在，同时确保 `SystemConfig(id='system')` 存在。
  - 管理员用户管理接口新增：
    - `GET /api/admin/users`
    - `POST /api/admin/users`
    - `PATCH /api/admin/users/:id`
    - `DELETE /api/admin/users/:id`
  - 用户管理规则固定：
    - 管理员创建用户时必须显式填写初始密码。
    - `POST /api/admin/users` 请求体固定为 `{ username, password, role, isEnabled }`。
    - `PATCH /api/admin/users/:id` 仅支持 `{ password?, role?, isEnabled? }`，本轮不支持改用户名。
    - 禁止删除自己。
    - 禁止删除最后一个管理员。
    - 禁止把最后一个管理员降级为 `user`。
    - `isEnabled=false` 的用户禁止登录，`/api/auth/login` 与 `authRequired` 都要拦截并返回 `AUTH_USER_DISABLED`。
  - 系统开关接口新增：
    - `GET /api/admin/system-config`
    - `PATCH /api/admin/system-config`
  - 业务开关规则固定：
    - `registrationEnabled=false` 时，`POST /api/auth/register` 返回 `403`，错误码固定 `AUTH_REGISTRATION_DISABLED`。
    - `appEnabled=false` 时，普通用户的业务接口统一返回 `503`，错误码固定 `SYSTEM_MAINTENANCE`，并返回 `maintenanceMessage`。
    - 维护模式豁免接口固定为：`/api/auth/login`、`/api/auth/logout`、`/api/auth/me`、管理员路由、`/api/system/health`、`/api/integrations/openclaw/status`。
    - 普通用户在维护模式下仍可登录，但访问聊天、设备摘要等业务接口时收到维护提示；管理员仍可登录并进入后台。
  - 中间件新增：
    - `authRequired` 增加 `isEnabled` 校验。
    - 新增 `appAvailabilityRequired`，挂在聊天、设备摘要等非管理员业务接口前。
    - 新增 `loadSystemConfig` 工具/服务，统一读取单例配置。

- V-1.2.3 管理员页面重构 frontend
  - 管理员页重构为一个工作台页面，内部采用 3 段切换，不新开一级路由：
    - `用户管理`
    - `系统开关`
    - `设备/网关接入状态`
  - `adminStore` 固定承载：
    - `users`
    - `systemConfig`
    - `openclawStatus`
    - `activeSection`
    - `loadingStates`
  - 用户管理区功能：
    - 列表显示 `username / role / isEnabled / createdAt`
    - 新增用户弹层或内嵌表单，字段固定 `username / password / role / isEnabled`
    - 每行支持启用/停用、角色修改、删除
  - 系统开关区功能：
    - 注册开关
    - 软件总开关
    - 维护提示语
    - OpenClaw 接入软开关
  - 接入状态区功能：
    - 展示当前 transport：`mock / openclaw`
    - 展示 OpenClaw 状态：`disabled / healthy / degraded / error`
    - 展示最近探测结果、当前 `apiMode`、`model`
    - 固定提示“下一轮对接树莓派 + OpenClaw 实机链路”
  - 个人中心新增管理员入口卡片，仅 admin 显示；普通用户不显示。
  - 角色保护继续保留：未登录不能进管理员页，普通用户不能进管理员页。

- V-1.2.4 OpenClaw backend adapter shell
  - 适配方向固定为：`uni-app 前端 -> backend -> OpenClaw Gateway`，禁止前端直连 Gateway。
  - 官方依据写入长期文档与 release 文档：
    - OpenClaw 首页：https://docs.openclaw.ai/zh-CN
    - Gateway 运行手册：https://docs.openclaw.ai/zh-CN/gateway
    - Gateway 协议：https://docs.openclaw.ai/zh-CN/gateway/protocol
    - Web / Control UI / 远程访问：https://docs.openclaw.ai/zh-CN/web/index
    - Control UI / SSH 隧道说明：https://docs.openclaw.ai/zh-CN/web/dashboard
    - OpenResponses 计划：https://docs.openclaw.ai/zh-CN/experiments/plans/openresponses-gateway
  - 适配层目录固定为 `backend/src/integrations/openclaw/`，至少包含：
    - `client.js`
    - `service.js`
    - `types.js` 占位
  - `.env.example` 新增：
    - `OPENCLAW_ENABLED=false`
    - `OPENCLAW_BASE_URL=http://127.0.0.1:18789`
    - `OPENCLAW_API_MODE=chat`
    - `OPENCLAW_MODEL=openclaw/default`
    - `OPENCLAW_BEARER_TOKEN=`
    - `OPENCLAW_REQUEST_TIMEOUT_MS=30000`
  - 运行时启用规则固定为：
    - `env` 里的 `OPENCLAW_ENABLED` 是运维硬开关。
    - `SystemConfig.openclawEnabled` 是管理员软开关。
    - 只有两者都为真时，聊天 transport 才允许走 OpenClaw。
  - 新增状态接口：
    - `GET /api/integrations/openclaw/status`
    - 该接口要求登录，不要求 admin；返回脱敏状态，不返回 token/password。
    - 逻辑：
      - 硬开关或软开关任一关闭 -> `status='disabled'`、`activeTransport='mock'`
      - 开启时探测 `GET /v1/models`
      - 若包含 `openclaw/default` -> `status='healthy'`
      - 请求失败或模型不存在 -> `status='degraded'` 或 `error`
  - 聊天 transport 切换规则固定：
    - `OPENCLAW` 不可用 -> 走当前 mock 路径
    - `OPENCLAW` 可用且状态健康 -> 用 Node 内置 `fetch` 调 `POST /v1/chat/completions`
    - `OPENCLAW_API_MODE=responses` 时，切到 `POST /v1/responses`
    - 任何 OpenClaw 调用失败都回退 mock，并在响应里显式带出回退信息
  - `POST /api/chat/send` 响应扩展为：
    - `userMessage`
    - `replyMessage`
    - `transport`
  - `transport` 形状固定为：
    - `{ mode: 'mock' | 'openclaw', fallback: boolean, status: 'disabled' | 'healthy' | 'degraded' | 'error', message?: string, model?: string, apiMode?: 'chat' | 'responses' }`
  - 聊天页联动：
    - 页面顶部展示当前 transport 和接入状态
    - 当回退 mock 时，明确显示“当前不是实车链路”
  - 下一轮树莓派联调路径固定写入文档：
    - 首选 `SSH 隧道`：`ssh -N -L 18789:127.0.0.1:18789 user@host`
    - backend 继续请求本机 `http://127.0.0.1:18789/v1/...`
    - 热点直连仅列为次优方案，且必须配认证，不作为默认路径

## Test Plan
- 导航与路由
  - `src/pages.json` 含 3 项 `tabBar`，首页/对话/我的均可通过 `switchTab` 切换。
  - 管理员页不在一级 `tabBar` 中。
  - 个人中心只有 admin 看得到“进入管理台”。
- 用户与权限
  - 管理员可查看、新增、停用、启用、删除普通用户。
  - 停用用户登录返回 `AUTH_USER_DISABLED`。
  - 管理员不能删除自己。
  - 最后一个管理员不能被删除，也不能被降级。
- 系统开关
  - 关闭注册后，注册接口返回 `AUTH_REGISTRATION_DISABLED`。
  - 关闭软件可用后，普通用户访问聊天/设备摘要返回 `SYSTEM_MAINTENANCE` 和维护提示；管理员仍可进入后台。
- OpenClaw 适配
  - `OPENCLAW_ENABLED=false` 时，`GET /api/integrations/openclaw/status` 返回 `disabled`。
  - 启用后，若 `/v1/models` 可达且含 `openclaw/default`，状态返回 `healthy`。
  - `POST /api/chat/send` 在 OpenClaw 调用失败时自动回退 mock，且响应 `transport.fallback=true`。
- 文档与发布
  - `docs/releases/V-1.2.0/README.md` 存在且写入新版本纪律与 OpenClaw 对接依据。
  - 根 README 已加入 OpenClaw 官方文档清单。
  - 云端镜像继续发布到发布前确认的最新主施工分支根目录 `vue3/`。

## Assumptions
- 管理员新增用户时由管理员手工填写初始密码；本轮不补“首次改密”流程。
- 原生 `tabBar` 使用入仓占位图标，不等待单独设计资源。
- `OPENCLAW_ENABLED` 为运维硬开关，`SystemConfig.openclawEnabled` 为后台软开关；两者都开才走 OpenClaw。
- `GET /api/integrations/openclaw/status` 对已登录用户开放，但返回值必须脱敏，不能泄露 Gateway token/password。
- 本轮仍不实现真实 Gateway WebSocket 控制协议，仅落地 HTTP `/v1/models`、`/v1/chat/completions` 和 `responses` 后备位。
