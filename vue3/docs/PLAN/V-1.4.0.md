# V-1.4.0 计划：前端打包脚本冻结 + 后端部署硬化 + ROS 联调合同

## Summary
- 当前仓库已具备 `tabBar` 导航、`backend/`、OpenClaw/ROS env 壳子和 ROS adapter 基线；`V-1.4.0` 不改业务主线，只补上线前必须冻结的“构建、部署、联调合同、发布纪律”。
- 本轮新增根级 `scripts/` 目录，并新增 `docs/releases/V-1.4.0/` 文档组；同时补一份 `docs/releases/V-1.4.0/README.md` 作为本轮索引和元数据入口。
- 发布仍按新纪律执行：先在 V 线自测分支完成冻结，再只把 `vue3/` supplement merge 到 `main`，不覆盖 `Raspberrypi/` 或其他 C 线目录。
- 推荐本轮自测分支固定为 `feature/v-1.4.0-packaging-deploy-ros-contract`，提交按 `V-1.4.1` 到 `V-1.4.4` 拆分。

## Key Changes
### V-1.4.1 前端打包脚本冻结
- 新建 `scripts/release-mp-weixin.ps1` 与 `scripts/release-mp-weixin.sh`，本质都只包装现有 `npm run build:mp-weixin`，不发明新打包命令。
- 两个脚本都固定做三件事：检查当前目录为仓库根、执行构建、显式输出产物目录 `dist/build/mp-weixin` 和“下一步去微信开发者工具导入该目录”。
- 新建 `scripts/release-app-plus.ps1` 作为占位脚本，不尝试伪构建；脚本固定输出“当前仓未启用 build:app-plus，本轮只冻结 App 打包位说明”，并返回非零退出码，防止误用。
- 新建 `docs/releases/V-1.4.0/mini-program-build.md`，固定包含：
  - 本地命令
  - 产物目录 `dist/build/mp-weixin`
  - 微信开发者工具导入流程
  - App Plus 占位说明：后续再补 `build:app-plus`，CLI 只产出 App 资源 / `wgt`，真 `APK/IPA` 云打包仍走 Windows 侧 HBuilderX

### V-1.4.2 后端部署硬化
- `backend/src/app.js` 的 CORS 改为环境变量白名单，不再硬编码 localhost-only。
- 新增 `CORS_ALLOWED_ORIGINS`，格式固定为逗号分隔的完整 Origin 列表，例如：
  `https://vline.example.com,https://admin.example.com`
- CORS 行为固定为：
  - `Origin` 为空时继续放行，兼容 `curl`、反向代理健康检查和部分无源请求
  - 有 `Origin` 时只允许命中白名单的域名
  - `.env.example` 给出本地开发默认示例，文档明确要求公网部署必须改成真实域名
- 新增 `deploy/systemd/vline-backend.service`，按 Linux + systemd 单机部署冻结，默认值固定为：
  - `User=vline`
  - `WorkingDirectory=/opt/vline-backend/backend`
  - `EnvironmentFile=/etc/vline-backend.env`
  - `ExecStart=/usr/bin/env node src/app.js`
  - `Restart=always`
- 新增 `scripts/deploy-backend.sh` 与 `scripts/update-backend.sh`：
  - `deploy-backend.sh` 负责首装：准备目录、同步 `backend/`、安装生产依赖、生成 Prisma Client、执行生产迁移、安装/启用 systemd 服务
  - `update-backend.sh` 负责更新：同步代码、安装依赖、执行生产迁移、重启服务
- 后端生产脚本默认采用 `npx prisma migrate deploy`，不沿用当前本地开发用的 `prisma db push`，避免生产环境结构漂移。
- 新建 `docs/releases/V-1.4.0/backend-deploy.md`，必须写清：
  - 当前 backend stack
  - `SQLite` 只适合单机 / 小流量阶段
  - 生产 `DATABASE_URL` 必须指向 repo 工作树外部路径，例如 `/var/lib/vline-backend/dev.db`
  - systemd 服务、环境变量、部署/更新脚本的执行顺序

### V-1.4.3 ROS 联调合同文档
- 新建 `docs/releases/V-1.4.0/ros-integration-contract.md`，把当前 backend 对树莓派侧的期待冻结为合同文档。
- 文档固定写清：
  - backend 当前依赖的 ROS 环境变量：`ROS_ENABLED`、`ROS_TRANSPORT`、`ROSBRIDGE_URL`、`ROS_CMD_VEL_TOPIC`、`ROS_ODOM_TOPIC`、`ROS_IMU_TOPIC`、`ROS_SCAN_TOPIC`、`ROS_CMD_REPEAT_HZ`、`ROS_CMD_DEFAULT_HOLD_MS`、`ROS_RECONNECT_DELAY_MS`
  - 优先要求 `rosbridge`，若不用，则必须由 C 线提供等价 HTTP / WS bridge
  - 默认 rosbridge 端口 `9090`
  - 期待 topic：`/cmd_vel`、`/odom`、`/imu/data`、`/scan`
  - **如果 backend 在 ECS / 公网，而树莓派在手机热点或 NAT 内，必须额外提供隧道、VPN 或反向桥，不能假设公网 backend 可直接访问树莓派 `9090`**
- 文档里同时重申当前职责边界：ROS 负责固定控制和平面状态，OpenClaw 负责自然语言平面。

## Assumptions
- `CORS_ALLOWED_ORIGINS` 采用逗号分隔的完整 Origin 白名单，不支持通配符。
- 生产部署目标按 Linux + systemd 单机 VPS 冻结；默认目录采用 `/opt/vline-backend` 和 `/var/lib/vline-backend`，但部署脚本允许用环境变量覆盖。
- 本轮不新增 `build:app-plus` 到 `package.json`，只冻结脚本位和文档位，避免伪支持。
- 生产数据库仍是 `SQLite`，但数据库文件必须移出 repo 工作树；若后续进入多实例或公网正式流量阶段，再单独立项迁移 PostgreSQL。
